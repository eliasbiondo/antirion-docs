# EPIC-009 "Alerts and notifications" — re-audit (2026-04-20)

Scope: EPIC-009 (`docs/requirements.yaml:213`) and its features FEAT-099..FEAT-106 (lines 1329–1418), plus stories STORY-223..STORY-246 (lines 11939–13106). Cross-referenced against `project.notifications_model` (line 37), `project.non_functional_baseline` (lines 13–38), the data models (AlertRule line 22668, AlertEvent line 22733, AlertTimelineEntry line 22785, Runbook line 22803, AlertSnooze line 23612, PostMortem line 23638, ActionItem line 23667, OncallSchedule line 23691, OncallShift line 23720, StatusPageIncident line 23857, Incident line 23885, Slo line 24400, Notification line 24690, NotificationPreference line 21326), and every inbound reference to an EPIC-009 entity from EPIC-001/003/004/005/007/008/010/011/012/013.

This audit is **read-only**. `docs/requirements.yaml` is unchanged. This is a fresh pass and re-derives every finding from the current file — it does not assume the prior `epic-009-review.md` findings still apply. Items already resolved in `docs/audit/fix-log.md` (e.g. AlertRule.scope_kind, AlertSnooze.fingerprint, Incident model, ActionItem model, baseline cite of Team.lead_user_id, AlertEvent.source/rule_id-nullable) are not reported again.

---

## Summary

The previous audit cycle closed many of the structural blockers — `AlertRule.scope_kind` and a nullable `org_id` exist, `AlertSnooze.fingerprint` exists, a separate `Incident` model exists with `severity/participants/alert_event_id/org_id`, `PostMortem` was restructured into `summary/timeline/impact/root_cause` plus a separate `ActionItem` model, `AlertEvent.source` and a nullable `rule_id` exist, and the baseline now names `Team.lead_user_id` as the canonical owner. But the model changes were applied *without* updating the stories that depend on them. The result is that several FEAT-104 stories now contradict their own data models more strongly than they did at the previous audit.

Three systemic problems still dominate this epic in its current form:

1. **STORY-237 was never redirected to the new `Incident` model.** The Incident model was added expressly to back FEAT-104 incident declaration, but STORY-237 still creates a `StatusPageIncident` directly, still uses fields (`state`, `participants`, `alert_event_id`) that exist on `Incident` but not on `StatusPageIncident`, and still names a severity (`SEV-2`) that matches neither `Incident.severity` nor `StatusPageIncident.impact`. The fix landed half a step.
2. **STORY-236's PostMortem API still posts `content: string`, but the PostMortem model no longer has a `content` field.** The model was restructured to `summary/timeline/impact/root_cause` text fields. The story's API and "linked... with author and content" wording were never updated, so STORY-236 cannot be implemented against the current schema.
3. **The notifications_model contract is still ignored by every notification-bearing EPIC-009 story.** AlertRule still carries a raw `channels: json`, STORY-232 still has only two happy-path acceptance criteria (Slack fire, PagerDuty ack-sync), and no story records "the resolved audience list and the channel attempt" that line 38 demands. STORY-223, STORY-225, STORY-227, STORY-232, STORY-237, STORY-242, STORY-246 all skip audience resolution. This was C1 in the prior audit and is still the single largest gap.

Beyond these three, the platform-scope `Slo` cannot be authored (`Slo.org_id` is non-nullable but STORY-246 explicitly speaks of "platform-scope SLO (gateway fleet, worker fleet, ingest pipeline)"); `STORY-243` records an `AlertTimelineEntry.type = "escalation.secondary"` value not in the enum; `STORY-228` and `STORY-234` still both implement "ack an AlertEvent" with overlapping verbs and divergent API paths (`/acknowledge` vs `/ack`); FEAT-102 still does not list the in-app inbox or webhook delivery as exercised channels; and the baseline-promised "ingestion-health alert" and "release-promotion gate on burned error budget" are still unowned by any story.

Counts: **6 critical, 12 major, 11 minor, 8 cross-epic, plus a Missing-references and a Uncited-but-required section.**

## Critical

### C1. STORY-237 still uses `StatusPageIncident` after the `Incident` split

- `docs/requirements.yaml:12747-12772` (STORY-237 AC-001/002/003 and `data_models`):
  - "I click 'Declare incident', pick severity SEV-2 and add 2 participants" (line 12749)
  - "A StatusPageIncident is created linked to the AlertEvent with state 'investigating'" (line 12751)
  - "Participants are notified" (line 12752)
  - `data_models: [StatusPageIncident, AlertEvent, AuditLog]` (lines 12770-12773)
- `docs/requirements.yaml:23857-23884` (StatusPageIncident model): fields are `id, platform_incident_id, title, impact {minor|major|critical}, status {investigating|identified|monitoring|resolved}, components, updates, started_at, resolved_at`. **No `state`, no `severity`, no `participants`, no `alert_event_id`, no `org_id`.**
- `docs/requirements.yaml:23885-23914` (Incident model — exists, this is the model the prior audit's M1/C5 fix added): fields are `id, org_id, alert_event_id, severity {info|warn|crit}, title, participants, state {investigating|identified|monitoring|resolved}, started_at, resolved_at, published_status_page_incident_id`.

**Problem.** The Incident model was added precisely to back STORY-237 (per `docs/audit/fix-log.md` C5), but STORY-237 was never redirected to it. As written today, the story still creates a `StatusPageIncident` and references three fields that do not exist on `StatusPageIncident` (`state`, `participants`, link to AlertEvent) and one — severity `SEV-2` — that matches neither `StatusPageIncident.impact` (`minor|major|critical`) nor `Incident.severity` (`info|warn|crit`). The story is unimplementable against either model, and the half-applied fix made the contradiction sharper than it was at the previous audit.

**Fix.** Replace `StatusPageIncident` with `Incident` in STORY-237's `data_models`, change "A StatusPageIncident is created" to "An Incident is created", drop `SEV-2` for one of `info|warn|crit` (or expand the enum to include the SEV-N vocabulary if product wants it), and add a separate AC for publishing — "I publish the incident" → `Incident.published_status_page_incident_id` is set and a `StatusPageIncident` is created via FEAT-173 (this also removes the tenant→public-page leak called out in X7).

### C2. STORY-236's POST `/api/alerts/:id/post-mortem` body is `content: string`, but `PostMortem.content` no longer exists

- `docs/requirements.yaml:12694-12697` (STORY-236 AC-002): "I click 'Add post-mortem', write content and save" → "A PostMortem row is linked to the AlertEvent with author and content".
- `docs/requirements.yaml:12719-12728` (STORY-236 API):
  ```
  - method: POST
    path: /api/alerts/:id/post-mortem
    request:
      body:
        content: string
  ```
- `docs/requirements.yaml:23638-23666` (PostMortem model): fields are `id, alert_event_id, title, summary (text), timeline (text), impact (text), root_cause (text), author_user_id, created_at, updated_at`. The `content: string` field was removed when C6 was applied. **No `content` field exists.**

**Problem.** The API and the AC both speak of a `content` payload that the data model no longer accepts. The story cannot be implemented against the current schema. This is a regression of the same class as C1 — the model was restructured by the prior fix but its callers were not.

**Fix.** Replace the API body with `{ title, summary, timeline, impact, root_cause }` and rewrite AC-002 to "fill the four required sections and save" → "A PostMortem row is linked to the AlertEvent with author and the four required sections set". Add `PostMortem` model description hint that all four text fields must be non-empty at save time (the description already says template is enforced — line 23639).

### C3. `Slo.org_id` is non-nullable but STORY-246 declares platform-scope SLOs

- `docs/requirements.yaml:24406-24408` (Slo model): `org_id type: FK:Org` — no `nullable: true`.
- `docs/requirements.yaml:13088-13106` (STORY-246):
  - "For a tenant-scope SLO the event routes to the tenant's AlertRule subscribers per FEAT-102" (line 13104)
  - "For a platform-scope SLO (gateway fleet, worker fleet, ingest pipeline) the event routes to the platform operator audience" (line 13105)
- `docs/requirements.yaml:28` (baseline `availability_slo`): "gateway data plane targets 99.95%" — implies a platform-scope `Slo` row.

**Problem.** Identical to the AlertRule problem the prior audit's C3 fixed: a platform-scope `Slo` cannot exist because `Slo` belongs to a tenant. STORY-246's tenant-vs-platform distinction has no data-model anchor. The baseline's gateway-availability SLO has nowhere to live, even though STORY-244 explicitly defines such an SLO at line 13057 ("matches project.availability_slo gateway data-plane target") — that example would be authored under the platform's own identity, not a tenant's.

**Fix.** Mirror the AlertRule fix from the prior audit. Make `Slo.org_id` nullable, add `Slo.scope_kind {tenant|platform}` (default `tenant`), gate read/write to platform-scope SLOs by mode + role. Update STORY-244 to explicitly name the scope of the example SLO.

### C4. STORY-243 records `AlertTimelineEntry.type = "escalation.secondary"` — value not in enum

- `docs/requirements.yaml:22800-22802` (AlertTimelineEntry.type): enum `fire|ack|resolve|page|note|snooze|escalate`.
- `docs/requirements.yaml:13017-13026` (STORY-243 AC-001): "An AlertEvent paged the primary 10 minutes ago" → "The secondary is paged" → "An AlertTimelineEntry records 'escalation.secondary'".

**Problem.** `escalation.secondary` is ad-hoc and not a declared enum value. STORY-229 AC-002 (line 12372) already uses `page` for the same physical action; STORY-243 needs a way to distinguish "first page" from "escalated page" and the enum doesn't carry that distinction.

**Fix.** Either (a) write STORY-243 as `type: "escalate"` with a `message: "secondary paged after no-ack window"` carried in `AlertTimelineEntry.message` (line 22799), or (b) add `escalation_level int` to AlertTimelineEntry and let the enum stay as `escalate`. (a) is the smallest change and uses the existing schema.

### C5. STORY-227 implies an `AlertRuleSubscription` model that does not exist; the story has no API and no audit

- `docs/requirements.yaml:12255-12278` (STORY-227):
  - "A subscription row is created linking me to the rule" (line 12271)
  - "Future fires notify me via my preferred channel" (line 12272)
  - `data_models: [AlertRule, NotificationPreference]` (lines 12273-12275)
  - No `api:` block at all.

**Problem.** No `AlertRuleSubscription` (or analogous) model exists in the file. `NotificationPreference` (line 21326) is keyed by `(user_id|org_id, event_type)` and has no per-AlertRule binding. The "preferred channel" concept is also unresolved — `NotificationPreference.channels` is `json` with no per-rule selector. The story declares a subscribe action but provides no endpoint, no AuditLog entry and no `unsubscribe` flow. P1 story, no implementation surface.

**Fix.** Either (a) introduce `AlertRuleSubscription { id, alert_rule_id, user_id, created_at }` and add `POST /api/alert-rules/:id/subscribe` and `DELETE /api/alert-rules/:id/subscribe`, with an AuditLog `alert_rule.subscribe`, or (b) repurpose `NotificationPreference.event_type = "alert_rule:<id>"` and document it explicitly on the model. Either path needs an explicit acceptance for the channel resolution.

### C6. `AlertEvent.source` exists but no story sets it; rule-less AlertEvents from EPIC-003/004/005/007/008/010/011 are indistinguishable

- `docs/requirements.yaml:22743-22745` (AlertEvent.source): `description: rule|budget|anomaly|slo|system`.
- `docs/requirements.yaml:22739-22742` (AlertEvent.rule_id): nullable. The X1/X4 fix made these fields exist.
- Generators that produce an AlertEvent without an AlertRule:
  - `docs/requirements.yaml:4929` (STORY-061, EPIC-003) — `byok.degraded`
  - `docs/requirements.yaml:7808` (STORY-118, EPIC-005) — `canary.rollback`
  - `docs/requirements.yaml:10364` (STORY-178, EPIC-007) — Approval `sla_breached`
  - `docs/requirements.yaml:10830` (STORY-189, EPIC-008) — `budget.soft`
  - `docs/requirements.yaml:13911` (STORY-263, EPIC-010) — `audit.sink.degraded`
  - `docs/requirements.yaml:16413` (STORY-329, EPIC-011) — `no_provider_credential`
  - `docs/requirements.yaml:16621` (STORY-335, EPIC-011) — `provider.down`
  - `docs/requirements.yaml:16649, 7923` (STORY-336, STORY-117 EPIC-005/011) — `circuit.open`
  - `docs/requirements.yaml:18204` (STORY-386, EPIC-011) — provider rate-limit pressure (this one does have an AlertRule)
- None of the above ACs sets `source` on the emitted AlertEvent.

**Problem.** The whole point of the X1/X4 fix was that AlertEvents can come from non-rule sources, and the `source` field discriminates. Today, every system-generated AlertEvent leaves `source` unset (or implicitly `system`). Downstream consumers — STORY-236 history, STORY-246 SLO routing, FEAT-099 rule UI — cannot tell a budget AlertEvent from a runtime AlertEvent. The schema is right but the contract is unenforced.

**Fix.** Add a sub-acceptance to each generator AC: "AlertEvent.source is set to `<budget|anomaly|system>`". This is a one-line patch per story and is the only way to make the source field load-bearing. The corresponding consumer stories (STORY-236 filters, STORY-246 routing) should also assert source-based behavior.

## Major

### M1. FEAT-100 and FEAT-103 still overlap; `/acknowledge` vs `/ack` API drift

- `docs/requirements.yaml:1343-1352` (FEAT-100): "Acknowledge, resolve, snooze and escalate fired alerts." Stories STORY-228 (ack+resolve) and STORY-229 (snooze event + escalate).
- `docs/requirements.yaml:1373-1384` (FEAT-103): "Snooze an alert rule or acknowledge an active alert with a reason and duration." Stories STORY-233 (snooze rule), STORY-234 (ack alert), STORY-235 (mute fingerprint).
- `docs/requirements.yaml:12320-12321` (STORY-228 AC-001): `POST /api/alerts/:id/acknowledge`.
- `docs/requirements.yaml:12631-12632` (STORY-234 AC-001): `POST /api/alerts/:id/ack`.
- STORY-229 AC-001 (line 12361) snoozes an *event* for 1h. STORY-233 AC-001 (line 12557) snoozes a *rule* for 2h.

**Problem.** STORY-228 AC-001 and STORY-234 AC-001 both implement "acknowledge an AlertEvent", and they hit *different* HTTP paths (`/acknowledge` vs `/ack`). A client and a backend cannot agree on which is canonical. STORY-229 AC-001 also overlaps with STORY-233 + STORY-235 (event-level mute is the same shape as fingerprint-level snooze).

**Fix.** Pick one API path (`/ack` is shorter and used by STORY-234) and remove the other endpoint. Collapse FEAT-100 and FEAT-103 into one feature, or redraw boundaries: FEAT-100 = AlertEvent lifecycle (ack, resolve, escalate), FEAT-103 = AlertRule-level suppression (rule snooze, fingerprint snooze). Drop STORY-229 AC-001's event-level snooze (already covered by STORY-235's fingerprint mute).

### M2. STORY-232 still covers only Slack fire and PagerDuty ack-sync

- `docs/requirements.yaml:1366` (FEAT-102 description): "Deliver notifications via email, Slack, PagerDuty and webhooks."
- `docs/requirements.yaml:12511-12545` (STORY-232 — the single story for FEAT-102): two acceptance criteria, AC-001 Slack fire (line 12522-12530) and AC-002 PagerDuty ack-sync (line 12531-12538). No email, no webhook, no in-app inbox, no per-recipient failure, no retry/backoff exercise, no signature-verification failure path.
- `docs/requirements.yaml:12543-12545` (non_functional): mentions at-least-once delivery to 6h and PagerDuty signature, but no AC exercises either.

**Problem.** Three of the four channels FEAT-102 promises (email, webhook, in-app inbox) have zero coverage. The retry/backoff and signature-verification non-functionals have no AC backing them, so a build cannot fail on them. STORY-232 is the single most-cited story in the epic (every notification-bearing story routes through it) yet it specifies the smallest possible behavior.

**Fix.** Add at least one AC per declared channel (email, webhook, in-app inbox), one AC for retry/backoff that asserts the 6h cap, and one AC for invalid PagerDuty signature → drop event + record `AlertTimelineEntry` with `type=note, message="signature_invalid"`.

### M3. STORY-232 is silent on deployment-mode collapse rules and audience resolution

- `docs/requirements.yaml:37-38` (notifications_model): spells out the (a)(b)(c)(d)(e) audience taxonomy, the saas/self_hosted collapse for (d) and (e), the unset-contact fallback chain, and "every audit record on a notification-bearing event records the resolved audience list and the channel attempt".
- `docs/requirements.yaml:12511-12545` (STORY-232): no mention of `security_contact`, `billing_contact`, resource owner, platform operator, trust_and_safety, mode collapse, or unset-contact fallback. No AC writes the resolved audience list to AuditLog.

**Problem.** FEAT-102 is the only place the collapse rules are supposed to live. STORY-232 never exercises them. The downstream stories that say "per FEAT-102" (STORY-227 line 12272, STORY-242 line 12990, STORY-246 line 13104) are deferring to undefined behavior.

**Fix.** Add ACs for: unset `security_contact` in each mode, platform-operator routing in each mode, cross-tenant signal dropped in self_hosted, and audit-trail write of the resolved audience per line 38. STORY-263 (line 13911-13912) is the gold-standard pattern to copy.

### M4. AlertRule still carries a raw `channels: json`, ignoring the notifications_model

- `docs/requirements.yaml:11960` (STORY-223 AC-001): "I pick channels [email, slack]".

C1 of the prior audit is *deferred* in the fix-log; this re-audit confirms it is still open. **Fix.** Replace `AlertRule.channels: json` with `AlertRule.audience: json` (list of canonical role refs: `{role:"resource_owner"}`, `{role:"org_contact", contact:"security_contact"}`, `{role:"platform_operator"}`, `{role:"user_subscription"}`). Rewrite STORY-223 AC-001, STORY-225 AC-002 and STORY-227 to pick audiences instead of channel strings. STORY-232 then owns audience→user-set→channel resolution.

### M5. STORY-237 "Participants are notified" has no channel resolution, no audience

### M6. PagerDuty is hard-coded in STORY-229/STORY-232 with no integration gating

### M9. STORY-237 has no AC for publishing the Incident as a `StatusPageIncident`
