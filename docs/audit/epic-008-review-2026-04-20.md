# EPIC-008 "Budgets and cost control" — requirements.yaml re-audit (2026-04-20)

Scope audited: `docs/requirements.yaml` EPIC-008 (line 191) with
features FEAT-082..FEAT-098 (lines 1161–1328) and stories
STORY-184..STORY-222 (lines 10564–11938). Cross-referenced against the
project baseline (`project.non_functional_baseline`, lines 13–38), the
canonical `notifications_model` (line 37), the models section (lines
20927+) and the cross-cutting epics EPIC-004 (observability), EPIC-006
(caching), EPIC-009 (alerts), EPIC-010 (workspace admin), EPIC-011
(gateway runtime) and EPIC-013 (platform operator).

This is a **read-only** audit. No edits were made to
`requirements.yaml`. Findings were re-derived from the current file —
prior findings (`docs/audit/epic-008-review.md`, fix-log rows) are not
assumed to still apply; each still-present finding below is rooted in
the line it now cites.

---

## Summary

Most of the previous round's structural defects have landed: the
baseline `shared_limiters` clause now cites FEAT-138, STORY-187 cites
STORY-193/196, `BudgetEvent` carries amount/reason/valid_until/
approval_id/budget_id, `UserBudget.budget_id` exists, Project gained
`deleted_at`, Org gained the three missing contact fields, STORY-189
AC-002 returns 402, STORY-190 uses the canonical model id, STORY-200
no longer mode-switches tenant-scoped roles and no longer fans out to
trust_and_safety, Approval.kind includes `budget-change`, and
STORY-208 AC-003 now cites the config-reload SLA.

What remains splits into four load-bearing problems:

1. **Field-name mismatches that break the stated acceptance criteria**
   — STORY-198/199 write a `BudgetEvent.summary` that does not exist
   (model has `title`+`body`); STORY-215 sets `Billing.trial_ends_at`
   on a Billing model that has no such field; STORY-189 AC-001 writes
   `AlertEvent.type="budget.soft"` to an AlertEvent model that has no
   `type` field; STORY-190 tags a Request with downgrade info "in
   metadata" but Request has no `metadata` column (only a single
   nullable `tag` string).
2. **The budget↔alerts↔notifications seams are still undefined**. A
   budget threshold breach can produce three concurrent paging
   surfaces — a direct `AlertEvent` write (STORY-189), a
   `BudgetEvent.overage` + ad-hoc notification (STORY-200), and a
   user-defined EPIC-009 `AlertRule` on a `budget.*` metric — with no
   delineation and no ack/snooze/on-call lifecycle on any of them.
3. **Core plumbing is asserted but unowned**: no feature writes
   `Request.cost` → `Budget.spent`; no feature reads Budget derived
   fields (`projected`, `daily_burn`, `burn_7d`, `runs_out_on`) with
   a defined `trend` or persists them for the backtest STORY-192
   consumes; no feature runs the cycle-reset job STORY-199 presumes;
   no feature ingests daily `FxRate`; no feature owns the approval
   expiry transition STORY-196 AC-003 depends on; and no Invoice /
   Contest / Credit / PlanChange model exists to anchor FEAT-095 /
   FEAT-096.
4. **`BudgetPolicy.approval_threshold_pct` is still one field used by
   two orthogonal decisions** — waiver escalation (STORY-187) and
   edit escalation (STORY-197) — collapsing two knobs into one.

Sections below enumerate findings by severity and then the new
**Uncited-but-required** section covers features the surrounding
narrative implies but that are nowhere defined.

## Critical

### C1. `BudgetEvent.summary` is asserted twice but does not exist on the model

- `docs/requirements.yaml:11224` (STORY-198 AC-001):
  > A BudgetEvent of type "change" with summary "deleted" is written
- `docs/requirements.yaml:11283` (STORY-199 AC-001):
  > A BudgetEvent of type "change" with summary "cycle reset" is written
- Model `BudgetEvent` (line 22624) declares fields `type`, `amount`,
  `reason`, `valid_until`, `severity`, `scope`, `scope_id`, `title`,
  `body`, `actor_user_id`, `budget_id`, `approval_id` — **no
  `summary`**. STORY-194 AC-001 (line 11023) correctly reads
  `title` and `body`; STORY-198/199 invented a different field name.
- **Problem**: Two critical write paths (delete a budget, roll a
  cycle) claim a column that does not exist. Tests written to the AC
  pass only against a schema that has been unilaterally extended;
  the event timeline in STORY-194 will never show these entries
  correctly.
- **Fix**: rewrite STORY-198 AC-001 and STORY-199 AC-001 to use
  `title` (short label) and `body` (long text) — e.g.
  `title="budget deleted"` / `title="cycle reset"` — matching the
  model fields consumed by STORY-194.

### C2. `Billing.trial_ends_at` is asserted but does not exist on the Billing model

- `docs/requirements.yaml:11770` (STORY-215 AC-001):
  > Billing.plan changes with trial_ends_at set
- Model `Billing` (line 23157) declares: `org_id`, `plan`,
  `contract_start`, `contract_end`, `commitment`, `commitment_used`,
  `next_invoice_at`, `payment_method`, `cost_center`, `po_number`,
  `tax_id`. There is no `trial_ends_at` and no trial-state column of
  any name.
- **Problem**: "Start a trial" is the only place that claims to
  distinguish trial-of-paid from free / paid. Without the column,
  nothing can compute "trial expiring in 3 days" notifications
  (FEAT-169 platform-notice slot) or gate feature access on the
  trial window.
- **Fix**: either add `trial_ends_at: timestamp, nullable` to
  `Billing` (smallest fix, matches the one-row-per-org shape), or
  introduce a `PlanChange` / `PlanState` model (see U8) and move
  trial state there. STORY-215, STORY-216, STORY-217, STORY-218 and
  STORY-219 all presuppose plan-state persistence that is not in
  the model today.

### C3. STORY-189 writes an `AlertEvent` with a `type` field the model does not have

- `docs/requirements.yaml:10830` (STORY-189 AC-001):
  > An AlertEvent of severity "warn" is generated with type "budget.soft"
- Model `AlertEvent` (line 22733) declares: `id`, `rule_id`,
  `source`, `severity`, `status`, `title`, `summary`, `provider_id`,
  `model_id`, `team_id`, `fired_at`, …. **No `type`**. The
  near-analog `source` enum (`rule|budget|anomaly|slo|system`) does
  accept `budget`, but `type` is not a column.
- **Problem**: The story is the sole cited mechanism for a
  soft-threshold signal that is not a `BudgetEvent`. Because the
  field does not exist, the AC cannot be implemented as written; any
  implementation silently picks one of (a) reinterpret `type` as
  `source` and lose the ".soft" qualifier, (b) smuggle
  "budget.soft" into `title`, or (c) add an unsanctioned column.
- **Fix**: either (a) route budget thresholds through a
  system-managed `AlertRule` with `category="budget"` (see U14) so
  the AC becomes "the AlertRule fires with severity warn" and
  inherits EPIC-009 lifecycle, or (b) rewrite AC-001 to write
  `source="budget"`, `severity="warn"`, `title="budget soft
  threshold crossed"` and accept that this is a free-form signal
  outside the EPIC-009 inbox.

### C4. STORY-190 tags a Request "in metadata" but Request has no metadata column

- `docs/requirements.yaml:10886` (STORY-190 AC-001):
  > The request is tagged "downgraded" with the original model in metadata
- Model `Request` (line 21722) has a single nullable `tag: string`
  (line 21800) and no `metadata`, `tags[]`, `attributes` or json
  blob.
- **Problem**: There is no place to put the two values the AC
  requires. If "downgraded" goes into `tag`, the original model id
  has nowhere to live; if the original model id goes into `tag`,
  the downgrade-flag is lost. Downstream analytics (EPIC-004
  breakdowns, FEAT-097 cost-center CSV) cannot filter
  downgrade-served rows.
- **Fix**: either add structured columns to `Request` — e.g.
  `downgraded: bool`, `original_model_id: FK:Model, nullable` — or
  introduce a `Request.metadata: json` column. A typed pair is
  preferable because FEAT-141 (cost attribution) needs the original
  model id to reconcile "requested vs served" cost. This also
  touches X4 / U5 below.

### C5. `BudgetPolicy.approval_threshold_pct` is one field used for two unrelated decisions

- `docs/requirements.yaml:22567` (model field):
  > approval_threshold_pct … Percent increase above which a budget
  > change creates an Approval (default 50).
- `docs/requirements.yaml:10736–10751` (STORY-187 AC-001/002): same
  field gates whether a **waiver** goes through Approval
  (direct-path vs Approval-path).
- `docs/requirements.yaml:11166–11171` (STORY-197 AC-002): same
  field gates whether a **budget amount edit** goes through
  Approval.
- **Problem**: Waiver policy and edit policy are orthogonal business
  decisions ("all waivers need approval, small edits don't" is a
  common stance). One shared threshold collapses them. The field's
  own description only mentions "budget change", so STORY-187's
  reuse of it is invisible to a reader of the model.
- **Fix**: split into `waiver_approval_threshold_pct` and
  `edit_approval_threshold_pct` on `BudgetPolicy` and update
  STORY-187 and STORY-197 to name the specific field each uses.

## Major

### M1. STORY-189 still bypasses the EPIC-009 AlertRule lifecycle

- `docs/requirements.yaml:10830` (STORY-189 AC-001) writes an
  `AlertEvent` directly.
- EPIC-009 owns `AlertRule` (model line 22668) and provides
  ack/resolve/snooze (FEAT-100, FEAT-103), runbooks (FEAT-101),
  notification channels (FEAT-102) and on-call escalation
  (FEAT-105). None are reachable from a bare `AlertEvent` row — an
  `AlertEvent` without a `rule_id` has no `AlertSnooze` (line
  23612), no runbook, no on-call binding.
- **Problem**: An engineer cannot ack/snooze a noisy
  "budget.soft" signal the way they ack every other alert, and
  cannot page the right on-call because `AlertEvent.oncall_user_id`
  is resolved from `AlertRule.oncall_schedule_id` at fire time.
  Ties to C3.
- **Fix**: reroute STORY-189 AC-001 through a system-managed
  `AlertRule(category="budget", metric="budget.utilisation_pct",
  threshold=BudgetPolicy.soft_alert_pct, …)` that EPIC-008
  provisions per budget at budget-create time (see U14). The AC
  becomes "the budget's system AlertRule fires", inheriting the
  EPIC-009 lifecycle and honoring `AlertSnooze`.

### M2. Three parallel paging surfaces for the same budget state

- Threshold breach → `AlertEvent` (STORY-189 AC-001, line 10830).
- Overage → `BudgetEvent.overage` + direct channel delivery
  (STORY-200 AC-001, line 11335).
- User-defined `AlertRule` on a `budget.*` metric — permitted by
  `AlertRule.metric: string` with `category: budget` (no enum
  gate; model line 22695–22706).
- **Problem**: The same condition produces up to three
  notifications. Neither FEAT-084, FEAT-091 nor EPIC-009 declare
  themselves as the single source of truth. Expect duplicate
  pages and inconsistent dedup windows — STORY-200 claims "deduped
  for 1 hour on the same Budget" (line 11337) but neither
  `AlertEvent` nor `BudgetEvent` has a dedup column (and no
  dedup-state store is defined — see U15).
- **Fix**: pick one emitter. The natural choice is to make the
  system-managed AlertRule from M1 the sole paging surface and
  treat `BudgetEvent.overage` / `BudgetEvent.threshold` as a
  persistent *audit* of the Alert fire, not a second paging
  channel. State this explicitly in FEAT-084 and FEAT-091
  descriptions.

### M3. STORY-184/192/202 rely on "an alert fires" with no defined rule or mechanism

- `docs/requirements.yaml:10658` (STORY-185 NFR: "drift between
  live counters and ledger alerts").
- `docs/requirements.yaml:10951–10955` (STORY-192 AC-002):
  > MAPE rises above 20% … An alert fires to notify the finance contact
- `docs/requirements.yaml:11438–11442` (STORY-202 AC-003):
  > The current FxRate is older than 48h … An alert fires to finance contact
- **Problem**: "An alert fires" assumes either (a) a pre-seeded
  `AlertRule` whose metric is MAPE, FX staleness, or counter
  drift, or (b) a direct Notification write. Neither is defined.
  No rule is seeded; no story owns the metric producer. This is
  three distinct zombie alerts.
- **Fix**: either add a feature (see U14) that provisions
  well-known system AlertRules for
  `budget.forecast.mape`, `budget.fx.staleness_hours` and
  `budget.counter.drift`, or rewrite these ACs to cite a concrete
  `Notification.kind="billing"` / `security` delivery path.

### M4. Budget derived fields live on Budget but the computation has no persistence contract

- `docs/requirements.yaml:22515–22530` (Budget fields `projected`,
  `daily_burn`, `burn_7d`, `runs_out_on`).
- `docs/requirements.yaml:10915` (STORY-191 AC-001 formula):
  > Projected is calculated as spent + (7d_burn * days_left * trend)
- `trend` is nowhere defined; `projected` is a scalar on Budget
  with no historical trail; STORY-192 AC-001 backtests by showing
  "forecast vs actual day by day" but there is no
  `BudgetForecast` row to read from — Budget carries only the
  latest snapshot.
- **Problem**: Four derived fields pretend to be canonical,
  a referenced multiplier (`trend`) is undefined, and the
  historical series STORY-192's backtest needs is unstorable. The
  non_functional claim "Projection recomputed at least every hour"
  (line 10930) names no owner and no worker.
- **Fix**: introduce a `BudgetForecast(budget_id, as_of,
  projected, daily_burn, burn_7d, runs_out_on, trend,
  method_version)` model (see U2), move the derived fields off
  `Budget`, and add a story under FEAT-085 that defines `trend`
  and the refresh cadence.

### M5. No feature defines the `Request.cost → Budget.spent` pipeline

- `docs/requirements.yaml:10616` (STORY-184 NFR):
  > Dashboard reads roll up MTD spend from the Request stream
  > (materialized daily + intraday cache)
- `docs/requirements.yaml:10658, 10722, 10759, 10811, 10863…` —
  every EPIC-008 story carries
  > Counter reconciliation runs on a worker cadence; drift between
  > live counters and ledger alerts
  as NFR boilerplate.
- `Request.cost` is written by FEAT-141 (line 1744); `Budget.spent`
  is read by every dashboard, enforcement, forecast and billing
  line. **No EPIC-008 feature or story owns the write.** No story
  names the worker, the aggregation grain, the cache-hit handling,
  or the downgrade accounting rule.
- **Problem**: The entire enforcement + forecast + billing surface
  is grounded on an unspecified ingestion pipeline. Three features
  (FEAT-141, FEAT-066 cache savings, FEAT-086 budget counters) all
  depend on the same `cost` number and none hand it off formally.
- **Fix**: add a story under FEAT-086 (or a new feature under
  EPIC-008) that specifies — which fields of `Request`
  (`cost`, `input_cost`, `output_cost`, `cache_cost`) contribute
  to `Budget.spent`; reconciliation cadence and drift alert
  threshold; whether downgrade-served cost counts against the
  original or the served budget; and whether cached requests
  decrement the budget at all. See also X1, U1.

### M6. STORY-188 multi-row allocation has no atomicity or approval NFR

- `docs/requirements.yaml:10772–10789` (STORY-188 AC-001/002): one
  PATCH creates/updates N Budget rows (scope="project") and
  updates N Project.budget_id columns.
- The `non_functional` block (line 10809) inherits the generic
  shared-limiter lines; nothing states the allocation mutation is
  atomic, nothing states the new Budgets consume cycle_start /
  cycle_end from the parent, nothing states team-level Approval
  policy cascades to generated project budgets.
- **Problem**: partial-write windows will leave project budgets
  stranded, and a team whose BudgetPolicy requires Approval for
  edits can have child Budget rows created that bypass the
  policy.
- **Fix**: add NFR lines on STORY-188 that (a) require the
  allocation write to be a single transaction, (b) inherit the
  parent Budget's cycle, and (c) inherit / require the parent
  BudgetPolicy.

### M7. API path split on /api/budgets/* remains

- `docs/requirements.yaml:10700` (STORY-186): `POST /api/budgets` —
  scope in body.
- `docs/requirements.yaml:10791` (STORY-188):
  `PATCH /api/budgets/teams/:id/allocations`.
- `docs/requirements.yaml:10843` (STORY-189):
  `PATCH /api/budgets/teams/:id/policy`.
- `docs/requirements.yaml:11182` (STORY-197): `PUT /api/budgets/:id`.
- `docs/requirements.yaml:11246` (STORY-198): `DELETE /api/budgets/:id`.
- `docs/requirements.yaml:11302` (STORY-199):
  `PUT /api/budgets/:id/schedule`.
- **Problem**: Two coordinate systems on the same resource —
  `/teams/:id/…` is workable only when a Budget is uniquely
  determined by team-in-cycle, and breaks for `scope=project`,
  `scope=user`, `scope=org`. Clients discover this the hard way.
- **Fix**: canonicalize on `/api/budgets/:id/{policy,schedule,
  allocations}` and add a read-side
  `GET /api/budgets?scope=team&scope_id=…` that returns the
  per-team roll-up.

### M8. FEAT-091 overlap with EPIC-009 is not resolved in wording

- `docs/requirements.yaml:1255–1260` (FEAT-091 description):
  > Notify designated recipients when actual spend exceeds the
  > budget, distinct from soft-threshold alerts.
- `docs/requirements.yaml:11357` (STORY-200 NFR):

- **Problem**: "distinct from soft-threshold alerts" and

- **Fix**: after resolving M2, restate FEAT-091 as "pre-seeded

### M9. STORY-215/216/217/218 all mutate Billing.plan with no lifecycle model

- `docs/requirements.yaml:11769–11771` (STORY-215): trial start
  sets `trial_ends_at` (C2).
- `docs/requirements.yaml:11793–11795` (STORY-216 AC): "Prorated

- **Fix**: add `pending_plan: string, nullable` and

### M10. STORY-220 "tagged requests with cost-center" but Request cannot carry one

- `docs/requirements.yaml:11884` (STORY-220 AC-001 Given):
  > The workspace has tagged requests with cost-center
- `Request.tag` (line 21800) is a single nullable string;
  `Billing.cost_center` (line 23179) is one string per Org.
  There is no per-Request cost-center attribute.
- **Problem**: "CSV per cost-center with daily cost and a monthly
  total" requires partitioning rows by cost-center. Today every

  so the CSV has exactly one bucket. The showback intent is

- **Fix**: add `Request.cost_center: string, nullable` (indexed)
  and define the ingestion contract — request header, API key
  tag, project attribute, or AC-level header `X-Cost-Center`.
  See also U12.

### M11. STORY-193 (Grant waiver) lacks any UI specification

- `docs/requirements.yaml:10961–11004` (STORY-193): no `ui:`
  block. Every other state-mutating P1 story in EPIC-008 (STORY-197
  update, STORY-198 delete, STORY-199 rollover) has one.
- `docs/requirements.yaml:1207` (FEAT-086 description) lists
  "Grant one-time budget expansion" as a first-class
  screen-level capability and cross-references
  `proto/budgets-data.js` as the reference prototype.
- **Problem**: a high-impact mutation (raising a hard cap) has no
  screen, no typed-confirmation policy (see also m2), and no
  API-error surface beyond the AC's prose.
- **Fix**: add a `ui:` block to STORY-193 with `WaiverModal`
  inputs for amount, reason, `valid_until` and a confirm state,
  and extend the baseline `destructive_action_confirmation_policy`
  (line 5) to include waivers above a threshold.

### M12. STORY-185 team-detail endpoint does not surface waiver state

- `docs/requirements.yaml:10637–10646` (STORY-185 AC-001): the
  team detail returns `team: Budget, projects, users` but nothing
  about active waivers. `Budget` has no
  `effective_cap`/`active_waiver_amount` field; waivers live on
  BudgetEvent (line 22624, type="waiver", with `amount` and
  `valid_until`).
- **Problem**: "MTD spend" alone cannot tell an operator whether
  the current cycle is running above its *nominal* cap because of
  a waiver — and STORY-193 AC's "Enforcement resumes if the new
  cap is crossed" depends on being able to compute that effective
  cap. No endpoint serves it.
- **Fix**: either compute `effective_cap =
  Budget.amount + SUM(BudgetEvent.amount WHERE type='waiver'
  AND (valid_until IS NULL OR valid_until > NOW()))` in STORY-185,
  or denormalise it onto Budget as `effective_cap_current_cycle`.
  Call it out in STORY-185 AC-001 then.
