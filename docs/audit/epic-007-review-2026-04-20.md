# EPIC-007 — Safety and guardrails: requirements.yaml re-audit (2026-04-20)

Audit scope: `docs/requirements.yaml` lines 173–190 (epic), 1028–1160 (FEAT-069..FEAT-081), 9384–10563 (STORY-157..STORY-183) and the models cited by those stories: `SafetyEvent` (22309–22371), `DetectionRule` (22372–22425), `SafetyPosture` (22426–22457), `RedactionVaultEntry` (22458–22478), `PiiCategory` (22479–22497), `Approval` (23188–23231), `SafetyRuleVersion` (23526–23554), `SafetyRuleTemplate` (23555–23574), `TeamSafetyPolicy` (23575–23588+), `PiiUnredactRequest` (24366–24399), `Notification` (24690–24712), plus the cross-checked baselines `project.non_functional_baseline.security` (33), `...config_hot_reload` (21), `...streaming` (31), `...performance_slo` (25), `project.non_functional_baseline.notifications_model` (37–38) and `project.deployment_modes` (36).

This is a fresh pass. Previous findings in `docs/audit/epic-007-review.md` were not assumed to still apply; every item below was re-derived from the current file and its location re-cited. Findings from the prior audit that are now resolved per `docs/audit/fix-log.md` and confirmed against the file (e.g. `SafetyEvent.prompt_original` removed, `Approval.kind` gained `data-reveal`, `DetectionRule.method` enumerates eight methods, `PiiCategory.action` added, `DetectionRule.fire_count`/`fp_count` added, `SafetyEvent.sev` renamed to `severity`, `/api/safety/rules` standardized to `/api/detection-rules`, `RedactionVaultEntry` model introduced, `DetectionRule.deleted_at` and `installed_template_id` added, canonical notifications routing applied to STORY-159/177/178/180) are not re-raised.

This is a documentation review. No edits to `docs/requirements.yaml` were made.

---

## Summary

EPIC-007 has moved forward materially — the worst contract problems from the prior audit (persistent cleartext on `SafetyEvent`, `Approval.kind` overload for PII unredact, taxonomy drift between FEAT-073 and `DetectionRule.method`, missing `action` on `PiiCategory`, undefined `fp_rate` denominators, API-path drift) are now resolved in the file. The new class of defects is tighter and more localized: naming drift between stories and newly-added fields, model scope still at the org-less default for configuration that stories PATCH per-workspace, a handful of story invariants whose backing row or field still does not exist, and several uncited but clearly-required features that the rest of the spec already leans on.

Highest-impact themes in this pass:

1. **Template-install linkage drift.** STORY-173 writes `template_pack_id` to `DetectionRule`, but the field actually added in the recent fix is named `installed_template_id` (FK:SafetyRuleTemplate). Every install/uninstall step in the story references a column that does not exist; `SafetyRuleVersion.type` even carries a `template_install` value nothing writes. The template-pack → rule linkage is the first thing that reads off broken when a reader follows the story into the model.
2. **Per-workspace PII configuration still has no per-workspace model.** `PiiCategory` is a global registry with no `org_id`, but STORY-160 PATCHes a per-workspace strictness and STORY-159 assumes per-workspace `action` wiring. The `action` field was added to `PiiCategory` but that does not make the record org-scoped.
3. **Several story invariants still have no row to hold them.** STORY-167 tags an event `false_positive` with no field on `SafetyEvent`; STORY-169 grants a "one-time allow to the original fingerprint for the next 24h" with no grant model; STORY-178 routes to an OOO delegate with no delegation model.
4. **Request.prompt still durably stores cleartext.** STORY-159 AC-002 now explicitly asserts "Original prompts are never persisted", but `Request.prompt` (line 21803) is an unconstrained nullable string. The invariant the story added is already contradicted one model away.
5. **Pipeline surface remains undeclared.** No EPIC-007 feature and no EPIC-011 feature owns the ordering of safety relative to cache, residency, edge-auth and outbound provider call; `performance_slo` cannot be evaluated without it.
6. **Uncited-but-required features.** `SafetyPosture` has readers (STORY-183) and no writer. `RedactionVaultEntry` has a producer (STORY-159) but no lifecycle, rotation, residency-enforcement or expiry feature. Twelve PII categories are advertised (FEAT-070) with no seed/bootstrap feature. Cross-tenant safety abuse correlation (mandated by `notifications_model`) has no owner.

Counts: 6 critical, 14 major, 7 minor, 7 cross-epic, 5 missing/dangling references, 8 uncited-but-required features.

## Critical

### C-1. STORY-173 writes `template_pack_id` but the model field is `installed_template_id`

- Lines: `10159: - Installed rules are tagged with template_pack_id = "gdpr_pii"`; `10168: - All DetectionRule rows tagged with that template_pack_id are disabled (not deleted)`; `DetectionRule.installed_template_id` at `22411–22414` (`type: FK:SafetyRuleTemplate`); `SafetyRuleVersion.type` enum at `23536` includes `template_install` with no story writing that type.
- Problem: The install story says the tag is `template_pack_id = "gdpr_pii"` — a string value. The field that actually exists is `installed_template_id`, an FK to `SafetyRuleTemplate`. `SafetyRuleTemplate.id` is a string (`23559`), so a value like `"gdpr_pii"` could be stored, but (a) the field name on the story does not match the field name on the model, and (b) the story uses `template_pack_id` on both install *and* uninstall ACs, so the uninstall filter also reads off a column that does not exist. A human or a code generator following STORY-173 will attempt to persist `template_pack_id` and either break or silently dual-write.
- Reference check: `installed_template_id` exists at `docs/requirements.yaml:22411`; no `template_pack_id` field exists anywhere in the file.
- Fix: Rewrite STORY-173 ACs to use `installed_template_id` (and quote it as `DetectionRule.installed_template_id = "gdpr_pii"`). Independently, add an AC under STORY-173 that writes a `SafetyRuleVersion` of type `template_install` per installed rule — otherwise that enum value at `23536` is dead.

### C-2. `PiiCategory` is org-less but STORY-159/160 treat per-workspace strictness and action as configurable

- Lines: `PiiCategory` model `22479–22497` (fields `id, label, method, strictness, action, risk` — no `org_id`); STORY-159 AC-002 at `9510` (`PiiCategory "cc" has action "block"`); STORY-160 PATCH at `9549–9554` (`PATCH /api/safety/pii-config { body: { strictness } }`); STORY-160 data_models at `9562` lists `PiiCategory` as the mutated resource.
- Problem: `PiiCategory` has no `org_id` and no `unique` scoping statement, so it is a single global registry shared by every tenant. Yet STORY-160 PATCHes strictness "per workspace" (the title is "Set per-workspace PII strictness") and STORY-159 assumes per-workspace action. Mutating a global `PiiCategory` row through `/api/safety/pii-config` would change the setting for every tenant simultaneously — a cross-tenant leak of a safety-policy change, and a direct violation of `tenant_isolation` (line 34). The previous audit's C-5 fix added the `action` field but did not fix the org-scoping.
- Reference check: searched the file — no `PiiCategory.org_id` and no `PiiPolicy`/`PiiCategoryPolicy` record exists.
- Fix: Either (a) add `org_id: FK:Org` to `PiiCategory` and treat the seeded rows as per-org copies, or (b) introduce a new `PiiPolicy` record keyed on `(org_id, pii_category_id)` carrying `strictness` and `action`, leave `PiiCategory` as a global taxonomy, and repoint STORY-160's PATCH and STORY-174's override target at the new record. Update `TeamSafetyPolicy.overrides` (23585–23587) to reference the same keying.

### C-3. `Request.prompt` durably stores cleartext, contradicting STORY-159 AC-002's new invariant

- Lines: `Request.prompt` at `21803–21805` (`type: string, nullable: true`, no description); `Request.prompt_redacted` at `21806–21808`; STORY-159 AC-002 at `9517` (`Original prompts are never persisted; reversible spans are tokenized into RedactionVaultEntry and only FEAT-080 can reveal them`); `non_functional_baseline.security` at `33` (`never appear in logs, telemetry or persisted payloads`).
- Problem: STORY-159 now explicitly asserts original prompts are never persisted, but `Request.prompt` is an unconstrained nullable string, co-located with `prompt_redacted`. Every other part of the spec that writes the Request row (EPIC-004 request-log ingestion, EPIC-011 pipeline) will happily populate `prompt` because nothing forbids it. The invariant STORY-159 added is one model away from being false.
- Reference check: `Request.prompt` exists at `docs/requirements.yaml:21803`; no retention or redaction constraint on the field; `RedactionVaultEntry` (22458–22478) is the intended vault, but its connection to `Request` is only via STORY-159's prose.
- Fix: Either drop `Request.prompt` entirely (keep only `prompt_redacted`), or add an explicit description on the field stating it is populated only when `prompt_redacted IS NULL` *and* no redaction rule fired — and add a matching AC on STORY-159 and on the request-log ingestion story (EPIC-004 STORY-078 at 5786) that refuses to write `Request.prompt` when any `DetectionRule` or `PiiCategory` with `action != "allow"` fired on the request.

### C-4. STORY-169 grants a "one-time allow for the next 24h" with no model to hold the grant

- Lines: STORY-169 AC-002 at `9958–9965` (`A one-time allow is granted to the original fingerprint for the next 24h`); STORY-169 data_models at `9981–9983` (only `SafetyEvent`, `AuditLog`).
- Problem: The AC creates a time-boxed allow list scoped by `SafetyEvent.fingerprint`. Nothing persists this: there is no `SafetyOverrideGrant`/`SafetyAllowance` row, no field on `SafetyEvent` flagging it as granting-allow, no field on `DetectionRule` for per-fingerprint exceptions, and no TTL carrier anywhere. A safety decision on the next request cannot honor this grant because the only thing that was written is an AuditLog entry.
- Reference check: grep of the file confirms no grant/allowance/override model, and `SafetyEvent.fingerprint` (22351) is just a plain string.
- Fix: Introduce a `SafetyAllowanceGrant` (or similar) model keyed on `(org_id, fingerprint, expires_at)` and reference it from STORY-169 AC-002. Add a detector-path AC that consults the grant table before enforcing a block.

### C-5. STORY-167 "tagged false_positive" has no field on `SafetyEvent`

- Lines: STORY-167 AC-001 at `9820` (`The event is tagged false_positive`); `SafetyEvent` fields at `22309–22371` — `review_status` (nullable string, no enum), `reviewed_by_user_id`, `reviewed_at` — no `false_positive`/`fp` field and no enum document that spells "false_positive" as a legal `review_status` value.
- Problem: `DetectionRule.fp_count` is incremented when reviewers tag an event, but the tag itself has no home on `SafetyEvent`. Two consequences: (a) the same event can be counted multiple times into `fp_count` because there is no idempotency marker, and (b) the list UI cannot filter "show me FP events" because the tag does not exist as a column. The lifetime-monotonic `fp_count` added earlier is computed off a field that does not exist.
- Reference check: `SafetyEvent.review_status` exists but is `string, nullable: true` with no enum description; no `false_positive` field in the file.
- Fix: Either pin `review_status` to an enum that includes `false_positive` (and state the idempotency: incrementing `fp_count` happens at most once per event per review_status transition), or add a discrete `false_positive: bool` field, or introduce a `SafetyEventReview` row per (event, reviewer, verdict). Pick one and update STORY-167 AC-001 and STORY-169 AC-001 to match.

### C-6. STORY-161 AC-001 returns HTTP 451 without setting `Request.status_label`

- Lines: STORY-161 AC-001 at `9584` (`The request is rejected with 451`); compare STORY-159 AC-002 at `9514` (`The gateway returns 451 with Request.status_label = "safety_blocked"`); other policy blocks at `14639` (`status_label "ip_denied"`), `15381` (`"residency_violation"`), `15407` (`"geo_blocked"`).
- Problem: STORY-159 was updated (per fix-log M-4) to set `status_label = "safety_blocked"`; STORY-161 was not. Two blocked-at-edge 451s for two different safety classes now produce different `status_label` values (one `safety_blocked`, one blank). EPIC-004 request-log filters and every dashboard that reads `status_label` cannot tell the two apart, and cross-tenant abuse correlation (per notifications_model line 38) cannot aggregate jailbreak blocks without a stable label.
- Reference check: STORY-161 AC-001 at `docs/requirements.yaml:9584` contains only "451"; `Request.status_label` is a plain string with no enum, so values are not centrally registered.
- Fix: Add `Request.status_label = "safety_blocked"` (or a specific label like `jailbreak_blocked` if per-class labels are desired) to STORY-161 AC-001. Consider adding a brief enum description on `Request.status_label` at 21757 so these values do not drift further — or file that enum work under a separate EPIC-004 finding.

## Major

### M-1. `RedactionVaultEntry` has no lifecycle, retention, rotation, or residency-enforcement surface

- Lines: `RedactionVaultEntry` at `22458–22478` (fields: id, org_id, token, ciphertext, category, region, created_at); STORY-159 NFR at `9527` (`workspace-scoped, residency-pinned, CMK-bound`); STORY-182 at `10470–10491` (auto-expire on the `PiiUnredactRequest`, not on the vault).
- Problem: The vault is the canonical reversal store — but there is no `expires_at`, no `retention_days`, no `rotated_at`, no `deleted_at`, no `cmk_binding_id`, and no story that deletes entries on Org deletion or on residency reassignment. FEAT-080's one-time-reveal flow expires the token on `PiiUnredactRequest`, not the underlying vault row — so the vault grows unbounded and carries potentially decades-old cleartext ciphertext indefinitely, which directly conflicts with data-minimization expectations in `non_functional_baseline.security` (line 33).
- Fix: Add `expires_at`/`retention_days`, `rotated_at`, `cmk_binding_id` (FK:CmkBinding when present) and a `deleted_at` column to `RedactionVaultEntry`. Add a story either under FEAT-070 or FEAT-080 that (a) purges on retention policy, (b) fails the write on residency mismatch, and (c) invalidates all entries bound to a revoked CMK within seconds.

### M-2. `EPIC-007.description` (line 175) drifts from its actual feature set

- Lines: `173–190`. The description enumerates "PII detection and redaction, prompt-injection and jailbreak detection, classification, rule management" — four capabilities — but the epic owns thirteen features (FEAT-069..FEAT-081), including overrides, versioning/rollback, dry-run/backtest, templates, per-team policy, approvals, unredact, and the live flow diagram.
- Problem: Readers browsing the top of the epic cannot discover FEAT-074..081 from the description; every mode-availability statement ("declares its mode availability explicitly in its description" — baseline line 36) is therefore invisible at the epic level.
- Fix: Expand line 175 to enumerate all thirteen feature clusters, and add a per-feature mode declaration (see M-4 below).

### M-3. `SafetyEvent.review_status` enum undefined

- Lines: `SafetyEvent.review_status` at `22363–22365` (`string, nullable: true`); STORY-169 AC-001 at `9955` (`review status is "reviewed"`); STORY-164 AC-001 at `9666` (`reviewed_at is set` — does not touch `review_status`).
- Problem: Two stories write/read the same field but disagree on its values (one uses `reviewed`, the other does not set it). No enum description exists. The bulk-review path becomes indistinguishable from an empty review state.
- Fix: Pin `SafetyEvent.review_status` to an enum (`unreviewed|reviewed|false_positive|escalated` or similar) in the field description, and update STORY-164 AC-001 to set `review_status = "reviewed"` alongside `reviewed_at` and `reviewed_by_user_id` (see also M-5).

### M-4. Mode availability still not declared on any EPIC-007 feature

- Lines: baseline mandate at `36` ("Every feature that differs in scope, availability or data shape between the two modes declares its mode availability explicitly in its description"); FEAT-069..081 at `1028–1160` — no `mode:` field and no saas-vs-self_hosted sentences in any description.
- Problem: FEAT-077 template packs, FEAT-080 unredact (privacy_officer vs platform admin), FEAT-071/072 cross-tenant signals, FEAT-081 live flow (operator drill-in) and FEAT-079 data-reveal approvals all behave differently per mode. The baseline explicitly requires this to be declared at the feature level.
- Reference check: no `mode:` key on any FEAT-069..081; EPIC-013 features (e.g. lines 2104, 2114, 2124, 2134) carry it as the reference convention.
- Fix: Add `mode: both` to every FEAT-069..081 and, where behavior differs, a sentence in the description spelling out the saas/self_hosted difference.

### M-5. STORY-164 bulk-review leaves `reviewed_by_user_id` unset

- Lines: STORY-164 AC-001 at `9660–9667` (`All 50 SafetyEvent.reviewed_at is set; A single AuditLog entry records the bulk action with count`); STORY-169 AC-001 at `9948–9956` (`review status is "reviewed" with me as actor`).
- Problem: Single-event review writes the actor to `SafetyEvent.reviewed_by_user_id`. Bulk does not. Given the model has the column, the bulk path is either incidentally anonymous or will NULL out the actor — in both cases, audit reconstruction from event rows is worse after the bulk action than before.
- Fix: Extend STORY-164 AC-001 to set `reviewed_by_user_id` to the acting principal on every selected row.

### M-6. STORY-174 declares a 60s policy cache that conflicts with the 30s `config_hot_reload` baseline

- Lines: STORY-174 non_functional at `10254–10255` (`Policy evaluation is cached for 60s keyed by (team_id, category)`); `non_functional_baseline.config_hot_reload` at `21` (`propagate to every node within 30 seconds without restart`).
- Problem: A 60-second team-policy cache is twice the baseline SLA for policy propagation. A safety-policy tightening applied by STORY-177 (approve-apply) can take up to 60s to take effect on a team that has a cached policy — stale for up to 2x the permitted window, and silently so.
- Fix: Align the cache TTL with `config_hot_reload` (<= 30s) or invalidate the team cache explicitly on `TeamSafetyPolicy` writes / `policy.change.apply` AuditLog events. The simplest change is to lower the TTL to 15s and document the explicit invalidation hook.

### M-7. Action precedence across allow/block/redact/flag is still undefined

- Lines: FEAT-073 at `1074`; STORY-161 AC-002 at `9586–9595` (allowlist short-circuits a rule without a global rule); STORY-165 AC-001 at `9683–9692` (new allow-typed regex "no longer fires the default URL rule"); `SafetyEvent.action` enum at `22325` (`block|redact|flag|allow`).
- Problem: Four action values coexist, two stories imply allow overrides other verdicts, but the total order is nowhere stated. Simultaneous matches (one rule says `redact`, another says `block`) have no declared winner.
- Fix: Add a "rule precedence" paragraph to FEAT-073 (proposal: `allow` > `block` > `redact` > `flag`, ties resolved by `severity` then `id`) and an AC under STORY-165 covering simultaneous matches deterministically.

### M-8. FEAT-074 override matrix remains incomplete

- Lines: FEAT-074 at `1085–1086` (`block -> allow, flag -> block`); `SafetyEvent.action` at `22325` (`block|redact|flag|allow`).
- Problem: The four-value enum admits twelve directed transitions; FEAT-074 enumerates two. `redact -> allow` (the natural pairing with FEAT-080 unredact) and `allow -> flag` (escalation) are both unsupported despite being operationally common.
- Fix: Restate FEAT-074 as "permits any transition among action values subject to an audit reason", or enumerate the legal transitions explicitly and cover at least `redact -> allow` and `allow -> flag` in new STORY-169 ACs.

### M-9. STORY-178 SLA-breach routing to `trust_and_safety` misuses the cross-tenant sub-queue

- Lines: STORY-178 AC-001 at `10358–10364` (`on breach, the Approval is marked "sla_breached" and an AlertEvent fires, with audience routed to Org.security_contact (and Antirion trust_and_safety sub-queue in saas; collapsed to platform admin in self_hosted)`); `notifications_model` at `37–38` ("Cross-tenant signals (isolation breaches, abuse heuristics, sustained anomalies) always reach Antirion trust_and_safety in saas").
- Problem: Policy-approval SLA breach is an intra-tenant workflow signal, not a cross-tenant abuse heuristic. The notifications_model reserves `trust_and_safety` for cross-tenant abuse classes; routing a tenant-local SLA miss there dilutes that sub-queue and creates noise for Antirion staff proportional to the customer's approver discipline. STORY-159 AC-002 gets this right (the T&S fanout is conditioned on "sustained patterns"); STORY-178 does not.
- Fix: Drop the `trust_and_safety` fanout from STORY-178 AC-001. Keep `Org.security_contact` (tenant-local) and let the self_hosted collapse to platform admin follow the unset-contact fallback rule from the notifications_model. If SLA breach rates ever need a cross-tenant rollup, that aggregator belongs under a separate feature (see MR-2 / U-4 below), not in the per-event notification.

### M-10. STORY-157 / STORY-158 list filter enums are not pinned

- Lines: STORY-157 query parameters at `9427–9437` (`category: [string], severity: [string], action: [string]`); `SafetyEvent.severity` at `22320–22322` (`crit|high|med|low`); `SafetyEvent.action` at `22323–22325` (`block|redact|flag|allow`); no enum stated for `category`.
- Problem: The query contracts accept any string — the caller must guess the enum. `severity` and `action` are actually enumerated on the model, but the story does not reference those enums or their values; `category` has no enum anywhere. A client that filters on `severity=high` today silently becomes incompatible if the enum ever changes, because the breakage surfaces only in a zero-result list.
- Fix: Reference the model enums explicitly in the STORY-157 query description (e.g. `severity: see SafetyEvent.severity enum`) and either (a) introduce a `SafetyCategory` enumeration on the model or (b) document the canonical category strings in the FEAT-070/071/072 descriptions.

### M-11. STORY-166 AC-002 says "removed" but `DetectionRule.deleted_at` was added

- Lines: STORY-166 AC-002 at `9760–9767` (`The rule is removed; Open SafetyEvents linked to the rule retain their rule_id for history`); `DetectionRule.deleted_at` at `22415–22418` (`Soft-delete; SafetyEvent.rule_id remains valid for history after delete`).
- Problem: The fix-log shows the deleted_at soft-delete column was added to satisfy M-11 in the prior audit, but STORY-166's language ("removed", "click Delete and confirm") still reads like a hard delete and does not mention `deleted_at`. A reader following the story will build hard-delete semantics; a reader following the model will build soft-delete semantics. Also, no filter on STORY-157/STORY-165's rule list excludes `deleted_at IS NOT NULL`.
- Fix: Rewrite STORY-166 AC-002 to state `deleted_at` is set (not hard deleted). Add a one-liner to STORY-165's rule list and STORY-157's event list clarifying deleted rules remain joinable but are filtered from active pickers.

### M-12. STORY-168 references a "Drafts" filter with no GET endpoint declared

- Lines: STORY-168 AC-001 at `9848` (`The rule appears in the "Drafts" filter in the rule list`); STORY-165 APIs at `9701–9731` (POST and PATCH `/api/detection-rules` and `/api/detection-rules/:id`; no GET list).
- Problem: The story references a filter but no `GET /api/detection-rules?status=draft` endpoint exists in STORY-165, STORY-166 or STORY-168, and the rule-list UI on `/safety` is undeclared in the story's `ui:` block (STORY-168 has no `ui` block at all).
- Fix: Add `GET /api/detection-rules?status=draft|published&enabled=bool&cursor=…` to STORY-165 or a new list story, and give STORY-168 a `ui:` block referencing the rule list screen and `DraftsFilter` component.

### M-13. STORY-171 `scope` is an untyped string

- Lines: STORY-171 body at `9692–9696` (`scope: string, samples: string[]`); AC-001 at `10066` ("Last 24 hours"); AC-002 at `10073` ("20 text samples").
- Problem: `scope` carries two very different semantics — either a canned window (`"last_24h"`, `"last_7d"`) or a mode-toggle (`"paste_samples"`) — with no enum. Server-side validation is unenforceable; clients pass freeform.
- Fix: Pin `scope` to an enum (`last_24h|last_7d|sample`) and make `samples` mandatory iff `scope = "sample"`.

### M-14. STORY-163 "redacted content" CSV column still unbounded

- Lines: STORY-163 AC-001 at `9637–9644` (`A CSV with event id, time, category, rule, action, redacted content and request id is produced`); no stated guard against embedding the un-redacted span for any class.
- Problem: The CSV path is a natural exfiltration vector. Nothing in the AC forbids embedding `SafetyEvent.spans` raw tokens or the `RedactionVaultEntry.ciphertext` for debug. The audit log of the export is also absent (compare STORY-088 at 5986, which this story does not reference).
- Fix: Add an AC stating the export uses `prompt_redacted` only and never joins `RedactionVaultEntry`; add a `data_export` AuditLog entry with actor, filter and row count; require the `safety.export` scope on the route.

### m-7. STORY-176 Notification on approve/apply is missing

## Uncited-but-required features

These are capabilities the current requirements.yaml relies on but does not own anywhere — models without writers, baseline mandates without owners, stories whose ACs cannot be satisfied without a feature that isn't in the feature list. Each one is "conceptually cited" by the surrounding spec, but no FEAT-NNN implements it. Severities follow the scheme above.

### U-1. Safety pipeline stage ordering feature (Critical)

- **Capability**: An explicit, testable declaration of the total order — edge-auth → ingress translation → inbound safety → cache lookup → outbound auth → upstream → outbound safety → cache write → telemetry — with latency budgets per stage and fail-open/fail-closed rules per detector.
- **Why implied**: STORY-159 narrative at `9491–9493` (`before sending to upstream`), STORY-162 at `9616` (`A response is produced`), `performance_slo` at `25` (gateway-added latency <= 1 ms p95 — unverifiable without stage ordering), `streaming` at `31` (mid-stream retry rules depend on where safety sits).
- **Closest neighbor**: FEAT-141 (cost + idempotency) or FEAT-143 (telemetry). A new FEAT under EPIC-011 (`FEAT-???: Pipeline stage ordering and budgets`) is cleaner.
- **Severity**: Critical. Every other EPIC-007 feature implicitly sits inside a pipeline stage whose spec does not exist; cache/safety coherence, residency/safety coherence and `performance_slo` all unravel without it.

### U-2. `SafetyPosture` aggregator / writer (Major)

- **Capability**: A background worker that computes `redactor_coverage`, `allowlist_compliance`, `residency_enforcement`, `readiness` and the per-minute counters, writes `SafetyPosture` rows on a rolling window, and owns the derivation formulas referenced by the STORY-183 tooltip.
- **Why implied**: `SafetyPosture` model at `22426–22457`, STORY-183 AC-002 at `10523` (tooltip explains the score), STORY-183 reads `SafetyPosture` but no story writes it; baseline worker surface at line `18` ("metric aggregation, … runs in a dedicated worker fleet").
- **Closest neighbor**: FEAT-081 live flow visualization (reader). A companion FEAT — `Safety posture aggregation` — needs to exist as the writer.
- **Severity**: Major. STORY-183 cannot run in production without it; the data model is declaratively orphaned.

### U-3. `RedactionVaultEntry` lifecycle (retention, rotation, deletion, residency enforcement) (Major)

- **Capability**: Feature owning the vault's write path guarantees, retention sweep (delete entries older than N days), CMK rotation (re-wrap `ciphertext` under a new CMK and invalidate old wrap), deletion on Org delete/suspend, and refusal-to-write on residency mismatch.
- **Why implied**: `RedactionVaultEntry` at `22458–22478` (has no lifecycle columns, see M-1), STORY-159 NFR at `9527` ("residency-pinned, CMK-bound"), FEAT-133 (CMK) and FEAT-130 (residency) expect write paths to respect them.
- **Closest neighbor**: FEAT-070 (PII detection) or FEAT-080 (unredact). Neither story today owns the lifecycle.
- **Severity**: Major. Without it the vault grows forever and is implicitly reachable across residency boundaries and past CMK revocation.

### U-4. Cross-tenant safety abuse correlation feature (Major)

- **Capability**: Aggregator that rolls up per-tenant `SafetyEvent` rates (jailbreak, PII-egress, classification-toxic), fires above a platform-tuned threshold, and routes to `trust_and_safety` in saas / drops in self_hosted.
- **Why implied**: `notifications_model` at `37–38` (`abuse heuristics, sustained anomalies`), STORY-159 AC-002 at `9516` (conditional T&S fanout "for sustained patterns" — with no rule owning what "sustained" means).
- **Closest neighbor**: EPIC-007 (new FEAT), or a companion under EPIC-013 platform-observability.
- **Severity**: Major. The notifications_model explicitly names the outcome; currently nothing produces it and STORY-178 is mis-routing intra-tenant signals into the same sub-queue (M-9).

### U-5. PII category seeding / bootstrap feature (Major)
