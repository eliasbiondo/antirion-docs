# EPIC-002 Personal account — Re-audit (2026-04-20)

Fresh read-only audit of `docs/requirements.yaml` at `EPIC-002` (line 92) and its features `FEAT-013`–`FEAT-021` and stories `STORY-035`–`STORY-048`. All findings were re-derived from the current file; prior findings were not assumed to still apply. Baselines cross-checked: `project.destructive_action_confirmation_policy` (line 5), `project.ui_baseline` (line 7), `project.non_functional_baseline` (line 14), `project.notifications_model` (line 37). Models cross-checked: `User` (line 21002), `UserPrefs` (21096), `Session` (21115), `TrustedBrowser` (21144), `MfaFactor` (21178), `RecoveryCode` (21202), `PersonalToken` (21300), `NotificationPreference` (21326), `AuditLog` (23065), `EmailVerificationToken` (23265), `LoginAttempt` (23292), `DataExportJob` (23322), `AccountDeletionRequest` (23358), `Notification` (24690).

Every feature and story the epic cites resolves to a defined block in the current file. The prior fix pass visibly landed: `RecoveryCode`, `PersonalToken.status`, `MfaFactor.is_primary`, `EmailVerificationToken.consumed_at`, typed-handle confirmation on deletion, and the canonical `/api/users/me/*` route namespace are all in place. The surviving issues are a mix of (a) semantic gaps left by prior partial deferrals and (b) newly surfaced contradictions that did not appear in the previous pass.

Top themes:
1. Personal-token lifecycle has two incompatible revoke semantics — hard-delete from the token panel (`STORY-042`) vs. `status="rotating"` from deletion (`STORY-046/047`) — and the `PersonalToken.status="revoked"` enum value is never written by any story.
2. `AccountDeletionRequest` is created and can be cancelled, but no feature anywhere in the file implements the day-30 cutover. `RecoveryCode` rows are generated and regeneratable, but no feature consumes them at sign-in. Both models have a write-side that leaves their primary consumer undefined.
3. Destructive security events in EPIC-002 use three different notification mechanisms — `Notification` kind=security, plain "email", silent — which is inconsistent and partly invisible to `FEAT-102` channel routing.

## Critical findings

### C-1. `STORY-046` declares `AuditLog` in data_models but never writes one
- Location: `docs/requirements.yaml:4386`–`4390` (`data_models: AccountDeletionRequest, User, PersonalToken, AuditLog`) and ACs at `docs/requirements.yaml:4338`–`4359`.
- Still exists: `AuditLog` model at `docs/requirements.yaml:23065`.
- Quote (AC-001 `then`): `- An AccountDeletionRequest is created ... - User.status remains "active" until cutover ... - My PersonalToken rows have status set to "rotating" ... - A confirmation email is sent to me and to each workspace owner`.
- Problem: `project.notifications_model` at `docs/requirements.yaml:38` states "Every audit record on a notification-bearing event records the resolved audience list and the channel attempt". STORY-046 AC-001 is unambiguously a notification-bearing event (email to self + each workspace owner) and declares `AuditLog` as a data model, but no AC writes an audit row — unlike its sibling STORY-047 (`docs/requirements.yaml:4414`), which correctly emits `user.deletion.cancelled`. Implementers will ship STORY-046 against `AuditLog` absence. This is a hard contradiction with the baseline.
- Fix: add an AC to STORY-046: `- An AuditLog entry with action "user.deletion.requested" is written and records the resolved audience list (acting principal + each workspace owner) and the channel attempt per project.notifications_model.` Mirror the structure STORY-038 uses at `docs/requirements.yaml:3918`.

### C-2. PersonalToken revoke semantics are contradictory across STORY-042 and STORY-046/047
- Locations: STORY-042 AC-002 at `docs/requirements.yaml:4093`–`4101`, STORY-046 AC-001 at `docs/requirements.yaml:4349`, STORY-047 AC-001 at `docs/requirements.yaml:4413`; model enum at `docs/requirements.yaml:21315`–`21317`.
- Still exists: `PersonalToken.status` enum `active|rotating|revoked` at `docs/requirements.yaml:21317`.
- Quote (STORY-042 AC-002): `- The PersonalToken is deleted server-side - Requests using that token now return 401` — (STORY-046 AC-001): `- My PersonalToken rows have status set to "rotating"` — (STORY-047 AC-001): `- My PersonalToken rows with status "rotating" are restored to status "active"`.
- Problem: The model now carries a `status` column with a `revoked` value, installed for the deletion grace flow. But the user-initiated revoke path at STORY-042 still hard-deletes. Consequences: (a) `revoked` is an orphan enum value — no story writes it; (b) the hard-delete path cannot be reversed, while the deletion flow promises reversibility via `rotating → active`; (c) an auditor/DBA cannot tell "revoked by user" from "never existed" because the row is gone. This is unimplementable as a single consistent schema.
- Fix: pick one semantics and apply everywhere. Recommendation: soft-delete in STORY-042 AC-002 — `- The PersonalToken.status is set to "revoked" and PersonalToken.token_hash is nulled - Requests using that token now return 401`, and add `PersonalToken.revoked_at` to the model. Then either drop `rotating` (STORY-046 uses `revoked` directly and STORY-047 restores to `active`) or keep `rotating` as a distinct intermediate state and document precedence. Whichever is chosen, update the `status` enum description at `docs/requirements.yaml:21317` to match.

## Major findings

### M-1. `STORY-045` filter misses the exact signal the story is designed to surface
- Location: `docs/requirements.yaml:4285`–`4294` (AC-001) and `docs/requirements.yaml:4328` (non_functional).
- Still exists: `LoginAttempt.user_id` is nullable at `docs/requirements.yaml:23299`–`23300` and carries an `email_attempted` column at line 23301.
- Quote (AC-001): `- I see the last 30 LoginAttempt rows for my user with time, IP, user_agent, device, location and outcome` — (non_functional): `- Only the authenticated user's own attempts are returned`.
- Problem: Credential-stuffing and account-enumeration failures typically land with `outcome="invalid_credentials"` *before* the login path has resolved a user, so `user_id` is null while `email_attempted` matches the victim. A strict `user_id = me` filter silently drops exactly the attempts the story's narrative ("spot unfamiliar access to my account") exists to expose. The shortcut at line 4294 ("Something unfamiliar? Reset password") is meaningless if those attempts are invisible.
- Fix: rewrite AC-001 to `- I see the last 30 LoginAttempt rows whose user_id matches my id OR whose email_attempted matches my current or previously verified User.email, ordered by at DESC`. Update the non_functional clause correspondingly to name both predicates so a reviewer can verify the filter.

### M-2. `STORY-046` audience uses local language instead of canonical notifications_model roles
- Location: `docs/requirements.yaml:4350` (AC-001 `- A confirmation email is sent to me and to each workspace owner`).
- Still exists: `project.notifications_model` at `docs/requirements.yaml:37`–`38` defines canonical roles (a)–(e); `Org.owner_user_id` (added per the EPIC-003 pass) is the single canonical owner handle.
- Problem: "each workspace owner" is not a role in the canonical set. The closest matches are (b) "owning principal on the resource" (but the resource here is the requesting user, not the workspace) and (c) "Org.owner_user_id". Naming audiences in local language is exactly what the baseline forbids ("Stories cite these roles instead of inventing local language"). It also makes the audit-record obligation (C-1) harder to fulfil deterministically because "each workspace owner" is either `role="owner"` members, or `Org.owner_user_id`, or both, and the story does not say.
- Fix: rewrite the audience in AC-001 as `- A confirmation email is sent via FEAT-102 to (a) the acting principal and (c) Org.owner_user_id; if Org.owner_user_id is null, fall back to every User with role="owner" in the Org per project.notifications_model fallback chain.` Record that resolved list in the AuditLog entry added in C-1.

### M-4. Security events in EPIC-002 use three different notification mechanisms
- Locations: STORY-038 AC-001 at `docs/requirements.yaml:3917` (uses `Notification` model + FEAT-102); STORY-041 AC-001 at `docs/requirements.yaml:4055` ("An email notifies me"); STORY-046 AC-001 at `docs/requirements.yaml:4350` ("A confirmation email is sent").
- Still exists: `Notification` model at `docs/requirements.yaml:24690`; `FEAT-102` notification channels at `docs/requirements.yaml:1363`.
- Problem: Three sibling security actions inside the same epic describe delivery in three different shapes. STORY-038 names the `Notification` model + FEAT-102 routing; STORY-041 says "email" without model or feature; STORY-046 says "confirmation email … to each workspace owner" without model or feature. Implementers cannot infer a consistent code path. If "email" is shorthand for `Notification` + FEAT-102 channel=email, the stories should say so; if it is a direct-mail path bypassing the notification system, STORY-041/046 escape the FEAT-102 preference/quiet-hours machinery that STORY-043 exists to let users configure.

Rewrite STORY-041 AC-001 bullet to `- A Notification of kind "security" ("account.mfa_factor_removed") is delivered to the acting principal via FEAT-102 channel routing`, add `Notification` to its `data_models` (line 4064–4067). ### M-5. `STORY-047` cancellation emits no notification at all — asymmetric with STORY-046

### M-7. `STORY-043` quiet-hours semantics still contradict the `NotificationPreference` model shape
- Locations: STORY-043 AC-002 at `docs/requirements.yaml:4173`–`4180`, API body at `docs/requirements.yaml:4181`–`4189`, model at `docs/requirements.yaml:21326`–`21353`.

- Problem: The row-shape contradiction flagged in the prior audit was deferred for product input and the current file still reflects both sides. ## Uncited-but-required features

Capabilities that the current `docs/requirements.yaml` assumes but which no feature owns. Every item below was verified by a whole-file search — if a matching feature existed elsewhere, it would appear here as "resolves to" rather than "not present".

### U-1. Account-deletion cutover worker
- Severity: **Critical**.
- What is missing: a background feature that, at `AccountDeletionRequest.scheduled_for`, transitions status `pending` → `completed`, hard-deletes the `User` row (or sets `status="offboarded"` + `offboarded_at=now`), revokes every `PersonalToken` still marked `rotating`, cascades to `Session`/`TrustedBrowser`/`RecoveryCode`/`MfaFactor` and writes a terminal `user.deletion.completed` AuditLog entry.
- Why the yaml implies it: STORY-046 AC-001 at `docs/requirements.yaml:4347` says `An AccountDeletionRequest is created with status "pending" and scheduled_for now + 30 days`; line 4348 says `User.status remains "active" until cutover`; line 4349 says `My PersonalToken rows have status set to "rotating" (to be revoked at cutover)`. `AccountDeletionRequest.status` enum at `docs/requirements.yaml:23370` includes `completed`; `AccountDeletionRequest.completed_at` at `docs/requirements.yaml:23381` exists. Nothing in the file writes `completed` or flips `User.status` after the grace window. A whole-file search for `deletion.*completed` or `cutover` inside ACs returns no writer.
- Closest neighbor: FEAT-007 (User offboarding, `docs/requirements.yaml:395`) owns admin-driven offboarding, but its stories (STORY-018/019/020) are admin-initiated and do not consume `AccountDeletionRequest`. FEAT-167 (worker fleet, `docs/requirements.yaml:1991`) surfaces worker health but does not own this job. A new feature — plausibly `FEAT-NNN Personal account deletion cutover` under EPIC-002 — is the correct home; it would reuse FEAT-102 for the cutover notification and FEAT-167 for worker-health telemetry.
- Impact: without this, every `AccountDeletionRequest` that is neither cancelled nor manually purged stays in `pending` forever; the `completed` enum value is orphan; "my data is removed" (STORY-046 narrative `so_that`) is never delivered.

### U-2. Recovery-code consumption at sign-in
- Severity: **Critical**.
- What is missing: a sign-in AC/feature that lets a user present a recovery code instead of their MFA factor, marks the matching `RecoveryCode.consumed_at` and grants a session; and an accompanying "too few codes remaining" Notification that warns the user to regenerate.
- Why the yaml implies it: `RecoveryCode` model at `docs/requirements.yaml:21202`–`21220` has `consumed_at` nullable, is described as `Hashed, single-use recovery code issued for MFA bypass`. STORY-039 at `docs/requirements.yaml:3961`–`3968` generates 10 codes with `plaintext shown exactly once`. STORY-020 at `docs/requirements.yaml:3155`–`3158` is the admin-reset path. No story in EPIC-001 (sign-in, MFA verification) consumes a code on the user's behalf: STORY-001 at 2151 (sign in with password) and STORY-002 at 2225 (trust device) do not mention recovery codes. A whole-file search for `RecoveryCode.consumed_at` finds only STORY-039 AC-001 (regenerate-marks-all).
- Closest neighbor: FEAT-001 (sign-in) or FEAT-015 (personal security and MFA). The consume path naturally belongs under FEAT-001 as an alternate sign-in factor, with an accompanying story under FEAT-015 that surfaces "codes remaining" on the security panel.
- Impact: the only reason to generate codes is to use them when a user loses their MFA factor; with no consume path, a user who has lost their device and still has a printed code has no self-service recovery and must depend on STORY-020 admin reset, defeating the code's purpose.
