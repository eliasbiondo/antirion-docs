## Critical

### C1. STORY-106 AC-001 broken intra-project reference to STORY-379/STORY-380

- **Where**: `docs/requirements.yaml:7126`
  ```
  - Providers have configured rate limits (see STORY-379/380) and have live traffic
  ```
- **Problem**: The cited stories are break-glass key lifecycle, not provider rate-limit configuration:
  - `STORY-379` at `docs/requirements.yaml:17847` = *Auto-expire break-glass key* (FEAT-160)
  - `STORY-380` at `docs/requirements.yaml:17870` = *Revoke break-glass key manually* (FEAT-160)
- **What it should cite**: The provider rate-limit configuration lives in FEAT-161:
  - `STORY-382` (`docs/requirements.yaml:17916`) — *Configure per-provider default upstream rate limits*
  - `STORY-384` (`docs/requirements.yaml:18053`) — *Pre-emptively throttle outbound calls against the effective ceiling*
  - STORY-106's own `depends_on: [STORY-385]` is correct (STORY-385 is the overview rate-limit pressure card) and points at a story whose own `depends_on: [STORY-382, STORY-384]` confirms the right targets.
- **Severity**: Critical — the citation is inside an acceptance-criterion precondition, so readers using STORY-106 as a spec for the provider headroom strip will be directed to the wrong stories.
- **Fix**: Replace `(see STORY-379/380)` with `(see STORY-382 / STORY-384)` or, better, `(see FEAT-161)`.

### C2. Mid-stream retry / fallback semantics still silent in EPIC-005

- **Where**: No story in EPIC-005 covers streaming behaviour under fallback or canary auto-rollback. FEAT-051/STORY-121 (`docs/requirements.yaml:7893-7930`) describes fallback on 5xx / timeouts / rate-limits but never mentions streaming.
- **Baseline**: `project.non_functional_baseline.streaming` at `docs/requirements.yaml:31`:
  ```
  Mid-stream upstream failure retries per RoutingRule.retries only when no bytes have reached the client; once any bytes have been streamed, the same target is not retried — an optional fallback target may reissue the full request, otherwise the stream closes with an error event.
  ```
- **Cross-epic alignment**: `STORY-327` AC-002 (`docs/requirements.yaml:16285-16292`) mirrors the baseline verbatim and delegates reissue behaviour to "failover" — i.e. to EPIC-005's fallback chain. But EPIC-005 defines neither the "fallback target may reissue the full request" semantic nor the error-event closure on streams.
- **Problem**: The EPIC-005 rule config has no field that says *whether a fallback is allowed to reissue*, nor how canary auto-rollback (`STORY-118` AC-001 at `docs/requirements.yaml:7799-7808`) interacts with an in-flight stream whose bytes have partially reached the client. A rule author has no way to express "this stream cannot be safely reissued" (non-idempotent streaming tool calls).
- **Severity**: Critical — the runtime in EPIC-011 has already been aligned with the baseline (`STORY-327` AC-002), which makes the configuration gap on EPIC-005's side the only blocker. Without a config surface, every tenant gets the same default with no override.
- **Fix**: Add a story to FEAT-050 (or FEAT-051) that (a) defines `RoutingRule.fallback.reissue_after_bytes: bool` (or equivalent), (b) adds an AC to STORY-121 covering the mid-stream fallback path, and (c) adds an AC to STORY-118 covering canary auto-rollback on an in-flight stream.

### C3. RoutingRule has no structured sub-schema for canary, shadow, or retry policies that stories rely on

- **Where**:
  - `RoutingRule` fields `canary json (nullable)`, `shadow json (nullable)`, `retries int` — `docs/requirements.yaml:22028-22033`
  - Fields named by stories but not modelled:
    - `rampPct`, `errorBudget`, `autoRollback`, `canary.healthy` — `STORY-118` at `docs/requirements.yaml:7801, 7807`
    - `currentErrorRate`, `errorBudget` with a "persistence window" — `STORY-117` AC-003 at `docs/requirements.yaml:7742-7747`
    - `error_rate < 1% for 60m` — `STORY-125` AC-001 at `docs/requirements.yaml:8095`
    - `error_rate < 1%`, `p95_latency_ms < 1.2x primary`, `cost_per_request within 1.1x primary`, `all over 60m` — `STORY-126` AC-001 at `docs/requirements.yaml:8134`
    - `triggers ["429", "5xx"]`, `base-delay 200 ms` — `STORY-327` AC-001 at `docs/requirements.yaml:16277` (EPIC-011 story targeting the same RoutingRule)
  - Shadow narrative invents divergence/latency/cost-delta reporting — `STORY-119` AC-002 at `docs/requirements.yaml:7843-7844`
- **Problem**: All of the above assumes a structured shape on top of an opaque `json` field. Without explicit subfields:
  - API validators cannot be specified; two implementations will accept different inputs.
  - `/api/routing/rules/dry-run` (STORY-122) has no canonical diff to compute.
  - `RoutingDeploy.before` / `after` diffs (STORY-127) render as free-form JSON, defeating approval review (STORY-133).
  - Per-field overrides (e.g. an org-wide default `auto_rollback_window_s` inheriting into a rule) have nowhere to live.
- **Severity**: Critical. The prior audit also flagged this (C8) and the fix log lists it as deferred ("Structural model change; requires product input"). It remains deferred in the current YAML.
- **Fix**: Define the JSON sub-schemas inline on `RoutingRule.canary`, `.shadow`, and a new `.retry` (replacing the scalar `retries: int`). At minimum:
  - `canary: {ramp_pct, error_budget_pct, auto_rollback, auto_rollback_window_s, auto_graduate_window_s, manual_promote_lookback_s, healthy, current_error_rate, p95_latency_factor, cost_per_request_factor}`
  - `shadow: {sample_rate, target_model_id, divergence_metrics_enabled}`
  - `retry: {max, triggers, base_delay_ms, backoff, jitter}`
  Pick one authoritative source of truth per field and cite the field name from every story instead of inlining a number.

## Major

### M1. Three different canary evaluation windows across stories

- **Where**:
  - `STORY-118` AC-001 `docs/requirements.yaml:7802`: "Live error rate on canary reaches 3% over a **5-minute window**"
  - `STORY-124` AC-002 `docs/requirements.yaml:8049`: "The canary's error rate over **the last 15m** exceeds the rule's threshold"
  - `STORY-125` AC-001 `docs/requirements.yaml:8095`: "error_rate < 1% **for 60m**"
  - `STORY-126` AC-001 `docs/requirements.yaml:8134`: "all over **60m**"
- **Problem**: One canary, four narratives, three distinct windows (5 / 15 / 60 min). Auto-rollback uses 5 min; guarded-promote uses 15 min; auto-graduate uses 60 min. No story states which window governs which purpose, and `RoutingRule.canary` has no field for any of them.
- **Severity**: Major — previously raised (prior M1) and still deferred per fix-log.
- **Fix**: Name the windows explicitly on the canary sub-schema from C3 (`auto_rollback_window_s`, `auto_graduate_window_s`, `manual_promote_lookback_s`) and have each AC reference the named field rather than inlining a number.

### M2. FEAT-050 lists six strategies but only three have stories

- **Where**:
  - FEAT-050 description `docs/requirements.yaml:840`: "Single, weighted, failover, least-loaded, canary and shadow routing strategies"
  - `RoutingRule.strategy` enum `docs/requirements.yaml:22018`: `single|weighted|failover|least-loaded|canary|shadow`
  - Stories for FEAT-050: `STORY-118` (canary), `STORY-119` (shadow), `STORY-120` (reference drawer). None cover `single`, `weighted`, `failover`, or `least-loaded` selection semantics.
  - `STORY-114` AC-001 (FEAT-049) creates a weighted rule, and `STORY-116` AC-001 edits a failover rule — but neither defines the runtime selection function, inputs (which load signal? per-provider or per-target?), or tie-breaking.
- **Problem**: `least-loaded` ships in the enum and the reference drawer but has no story fixing its load metric, tie-break or test cases. `single` is degenerate but still needs a definition.
- **Severity**: Major — previously raised (prior M2) and still deferred per fix-log.
- **Fix**: Add a story per strategy under FEAT-050 defining the selection function, a worked example, and a test-case. A `least-loaded` story is mandatory because no other surface defines its input metric.

### M4. CatalogPolicy and RoutingRule overlap on retry/fallback policies with no precedence rule

- **Where**:
  - `CatalogPolicy.fallback_on_5xx = {enabled, max_hops}` at `docs/requirements.yaml:21681-21683`
  - `CatalogPolicy.retry_on_429 = {enabled, backoff_min_ms, backoff_max_ms}` at `docs/requirements.yaml:21684-21686`
  - `RoutingRule.fallback json` at `docs/requirements.yaml:22021` and `RoutingRule.retries int` at `docs/requirements.yaml:22026`
  - `STORY-107` AC-001 `docs/requirements.yaml:7183` lists the CatalogPolicy fields verbatim; `STORY-114` AC-001 lets the user set `fallback` and `retries` per-rule at `docs/requirements.yaml:7592-7593`.
- **Problem**: Two layers, same policy surface, no story declares precedence. Does a per-rule override replace, cap, or compose with the catalog default? `STORY-107` AC-002 (`docs/requirements.yaml:7193`) only asserts "Downstream resolvers pick up the change within the config-reload SLA" — not how resolvers merge the two layers.
- **Severity**: Major — previously raised (prior M4) and still deferred per fix-log.
- **Fix**: Add a story to FEAT-049 (or FEAT-045) fixing precedence. The natural rule is "catalog as default; rule overrides when present; effective `max_hops = min(rule.fallback.length, catalog.fallback_on_5xx.max_hops)`". Document the chosen merge on both models.

### M5. ModelAlias.fallback_model_id (single) vs RoutingRule.fallback (chain) — same word, different shape

- **Where**:
  - `ModelAlias.fallback_model_id: FK:Model (nullable)` at `docs/requirements.yaml:21714-21716`
  - `RoutingRule.fallback: json (nullable)` at `docs/requirements.yaml:22021-22023` (chain per `STORY-329` AC-001 at `docs/requirements.yaml:16363`)
  - FEAT-047 description at `docs/requirements.yaml:806` describes "aliases that resolve to a model with a fallback"
- **Problem**: Two very different "fallbacks". When a request targets an alias that is itself the primary of a routing rule with its own chain, the interaction is unspecified. Is the alias resolved to a single model at resolve-time and the rule chain applied on top? Or are the alias-fallback and the rule-fallback merged into one chain at runtime?
- **Severity**: Major — previously raised (prior M5) and still deferred per fix-log.
- **Fix**: Either align the names (alias-fallback becomes single-element chain, or both are renamed) or add an AC to FEAT-047 stating the resolution order: alias is resolved to its primary eagerly; alias.fallback only fires when the alias cannot resolve (e.g. primary is disabled/deleted), independent of routing-rule fallback.

### M6. Canary auto-promote silently rewrites `RoutingRule.fallback`

- **Where**: `STORY-124` AC-001 at `docs/requirements.yaml:8045`:
  ```
  - The original primary is added to the fallback chain
  ```
- **Problem**: Promote-canary is a 1-click action whose side effect mutates the rule's `fallback` field. The confirmation dialog (AC-001 line 8040) does not surface the mutation; the chain grows unboundedly across promotions; and the mutation can quietly exceed `CatalogPolicy.fallback_on_5xx.max_hops` (see M4). `RoutingDeploy.before/after` captures the diff at `docs/requirements.yaml:23449-23453` but nothing in the story asserts the diff is surfaced to the actor.
- **Severity**: Major — previously raised (prior M7) and still deferred per fix-log.
- **Fix**: Add an explicit "Add prior primary to fallback chain" toggle (default on) to the promote dialog, cap the chain at `CatalogPolicy.fallback_on_5xx.max_hops` with a visible truncation warning, and require the promote confirmation dialog to show the chain delta.

### M7. Cache × canary/shadow and safety × canary/shadow interactions are still unspecified

- **Where**: No story in FEAT-050 or FEAT-053 references EPIC-006 (cache) or EPIC-007 (safety).
- **Problem**:
  - Shadow + cache: if the primary is served from the cache (`STORY-142` at `docs/requirements.yaml:8631`), does the shadow still issue a real upstream call? If yes, the shadow comparison is biased by counting cache-hits as "primary faster"; if no, the shadow misses the cached portion of traffic.
  - Canary + cache: a freshly-introduced canary target gets a cold cache while the primary is warm, biasing `p95_latency_ms` in STORY-126's threshold check. Should warm-cache requests be excluded from the canary evaluation window?
  - Canary/shadow + safety: when an EPIC-007 rule blocks a request, does that count as a canary "error"? As a fallback trigger (FEAT-051)? As a divergence event on a shadow? Nothing is stated.
- **Severity**: Major — previously raised (prior M10) and still deferred per fix-log.
- **Fix**: Add ACs to STORY-118/STORY-119 (and/or a new story under FEAT-050) covering each interaction and cross-link bi-directionally from EPIC-006/FEAT-060 and EPIC-007/FEAT-073.

### M8. No catalog-sync feature — every model is a manual entry

- **Where**: FEAT-046 description `docs/requirements.yaml:796` / `STORY-108` at `docs/requirements.yaml:7237-7288`. The sole creation path is manual `POST /api/models`.
- **Problem**: Provider catalogs change often. A purely manual path guarantees drift between Antirion's catalog and what the upstream provider actually serves. No story covers:
  - periodic reconciliation with provider catalog APIs
  - reconciliation reports (Antirion catalog vs. upstream inventory)
  - auto-deprecation when an upstream marks a model as retired
- **Severity**: Major — previously raised (prior M11) and still deferred per fix-log.
- **Fix**: Add a new feature (or extend FEAT-046) for provider-catalog sync. Define which providers expose an enumerable catalog API, the sync cadence, the reconciliation surface on `/models`, and plug auto-deprecation into FEAT-055's runbook.

### M9. FEAT-058 and FEAT-059 still scope-mismatched to "Model catalog and routing"

- **Where**:
  - EPIC-005 title at `docs/requirements.yaml:140`: "Model catalog and routing"
  - EPIC-005 description at `docs/requirements.yaml:141` acknowledges FEAT-058 (prompt/template registry) and FEAT-059 (model evaluation) but does not justify their placement.
  - FEAT-058 at `docs/requirements.yaml:918-928` and FEAT-059 at `docs/requirements.yaml:929-937` are inside this epic.
- **Problem**: Prompt/template registry is neither catalog nor routing. Model evaluation is a quality discipline that *uses* the catalog but isn't a catalog-management concern. The prior C7 (broken FEAT-072 reference) was a direct symptom of the stretched scope — FEAT-059's "translation regression suite" wound up citing a safety feature because nothing in EPIC-005 is actually about translation.
- **Severity**: Major — previously raised (prior M12) and partially applied (description acknowledges the features but they remain in the wrong epic).
- **Fix**: Split into a dedicated "Prompts and evaluation" epic (or move FEAT-058 alongside content-governance and FEAT-059 alongside observability/quality). Either the epic boundary follows the title, or retitle to "Model catalog, routing, prompts and evaluation".

## Uncited-but-required

Features that are not referenced anywhere in `requirements.yaml` but that the surrounding context strongly implies must exist for EPIC-005 to be coherent. One subsection per proposed missing capability. Each cites the line(s) that create the expectation, names the closest-neighbour feature, and assigns a severity.

### U1. Provider catalog sync (critical)

- **Missing capability**: A periodic reconciliation between Antirion's `Model` table and upstream provider catalog APIs, producing a diff surface ("new models available", "model marked legacy upstream", "capability changed") and feeding auto-deprecation through FEAT-055.
- **Why implied**:
  - `FEAT-046` (`docs/requirements.yaml:793-802`) only supports manual add, and `STORY-108` AC-001 (`docs/requirements.yaml:7246-7263`) requires a human to type in provider/id/tier/status.
  - `FEAT-055 STORY-129` AC-001 (`docs/requirements.yaml:8247-8258`) assumes deprecation is a *local* decision driven by the workspace admin, but upstream providers mark models legacy without asking.
  - The baseline hot-reload SLA (`docs/requirements.yaml:21`) only covers propagation to edge nodes, not catalog freshness against upstream.
- **Closest neighbour**: Extend `FEAT-046` or add a new `FEAT-XXX Catalog sync` in EPIC-005.
- **Severity**: Critical — without it, tenants silently call deprecated upstream models.

### U2. Shadow comparison data model & aggregation (major)

- **Missing capability**: A `ShadowComparison` (or equivalent) model that captures per-request divergence, latency delta, and cost delta between primary and shadow, plus an aggregation feature that computes the divergence metrics surfaced by `STORY-119` AC-002.
- **Why implied**:
  - `STORY-119` AC-002 (`docs/requirements.yaml:7843-7844`) promises "divergence rate, latency delta and cost delta" once ≥100 comparisons have accumulated.
  - `RoutingRule.shadow` is opaque `json` at `docs/requirements.yaml:22031-22033`; `RoutingDecision` at `docs/requirements.yaml:22048-22073` captures only the served decision, not the shadow arm.
  - Baseline `observability` (`docs/requirements.yaml:32`) emits spans per request — spans can carry this data but the story promises a queryable divergence surface, not just a trace.
- **Closest neighbour**: New `FEAT-XXX Shadow traffic analytics` in EPIC-005 (adjacent to FEAT-050) and a `ShadowComparison` entry in the models block.
- **Severity**: Major.

### U3. Canary metric aggregation pipeline (major)

- **Missing capability**: A feature that computes the per-rule, per-arm metrics (`error_rate`, `p95_latency_ms`, `cost_per_request`, rolling window) used by STORY-125 auto-graduation and STORY-126 thresholds.
- **Why implied**:
  - `STORY-125` AC-001 (`docs/requirements.yaml:8093-8101`) and `STORY-126` AC-001 (`docs/requirements.yaml:8128-8136`) reference these metrics as already-computed.
  - `FEAT-143` (`docs/requirements.yaml:1762-1770`) owns telemetry but does not describe canary-arm splits or the evaluator loop.
  - The canary sub-schema gap from C3 means the evaluator has no place to read thresholds from.
- **Closest neighbour**: Extend `FEAT-053` (Canary promotion and graduation) with an explicit evaluator story, or cross-link to a new `FEAT-XXX Routing metrics pipeline`.
- **Severity**: Major.

### U4. RoutingRule validator / match-expression grammar (major)

- **Missing capability**: A formal grammar / validator for `RoutingRule.match` and `RoutingRule.targets`, owning: match syntax, allowed operators, target-weight validation, target-model existence, alias resolution, and circular-chain detection.
- **Why implied**:
  - `STORY-114` AC-002 (`docs/requirements.yaml:7562-7570`): "Weights must sum to 100".
  - `STORY-116` AC-002 (`docs/requirements.yaml:7696-7703`): "At least one target is required".
  - `STORY-122` AC-002 (`docs/requirements.yaml:7951-7959`): "'Match syntax error: <detail>'".
  - No story defines the grammar, error model, versioning, or where the validator lives in the stack.
- **Closest neighbour**: Extend `FEAT-049` (Routing rule management) with an explicit validator story, or new `FEAT-XXX Routing rule grammar & validator`.
- **Severity**: Major.

### U5. Routing rule change notifications (major)

- **Missing capability**: Notification of affected teams when a RoutingRule edit, rollback, or canary-promote lands.
- **Why implied**:
  - `FEAT-054` (`docs/requirements.yaml:879-889`) captures every change as a `RoutingDeploy` but no story notifies anyone (contrast with `STORY-129` for model deprecation which does notify).
  - `STORY-128` AC-001 (`docs/requirements.yaml:8199-8208`) emits an `AuditLog` entry but no `Notification`. A rollback on a P0 routing rule should likely notify `Org.tech_contact` and rule owners.
  - `project.notifications_model` (`docs/requirements.yaml:37`) mandates canonical audiences for event-notification stories; routing-rule change events are not named.
- **Closest neighbour**: Add ACs to STORY-127/128/124 (Notification of change), or a new `FEAT-XXX Routing change notifications` sourcing from FEAT-102.
- **Severity**: Major.

### U6. Readiness-hydration of routing/catalog state (major)

- **Missing capability**: A feature owning gateway-node readiness with respect to routing state: "node is not ready until it has loaded the current RoutingRule set, CatalogPolicy, ModelAlias resolution map, and circuit-breaker state from shared storage".
- **Why implied**:
  - `project.non_functional_baseline.health_checks` (`docs/requirements.yaml:23`) requires the readiness probe to confirm "config loaded, migrations applied, circuit-breaker state hydrated".
  - `project.non_functional_baseline.zero_downtime_deploy` (`docs/requirements.yaml:24`) requires rolling deploys to drop zero requests; a node that starts taking traffic before its routing table is loaded would violate this.
  - No feature in EPIC-005 (or anywhere in the file, based on the surrounding context) owns the startup hydration path for routing/catalog specifically. FEAT-142 covers circuit-breaker transitions (`docs/requirements.yaml:1753-1761`) but not routing rule hydration.
- **Closest neighbour**: Extend `FEAT-049` or `FEAT-050` with a readiness story; or place under EPIC-011 if the team prefers "runtime" framing.
- **Severity**: Major — the baseline demands it, and its absence is invisible until a cold-start incident.

### U7. Preview-access allowlist management (minor)

- **Missing capability**: Stories to grant/revoke preview access when `CatalogPolicy.preview_access = "allowlisted"`.
- **Why implied**:
  - `CatalogPolicy.preview_access` enum at `docs/requirements.yaml:21690-21692` is `admin|allowlisted|all`.
  - `STORY-107` only displays/edits the enum value; nothing describes the allowlist store or its edit flow.
- **Closest neighbour**: Extend `FEAT-045` (Model catalog) with a `PreviewAllowlist` entity and story.
- **Severity**: Minor.

### U8. Golden-dataset versioning workflow (minor)

- **Missing capability**: Stories to bump a `GoldenDataset.version`, promote a version, or roll back.
- **Why implied**:
  - `GoldenDataset.version: int` at `docs/requirements.yaml:24770` implies a versioning lifecycle.
  - `STORY-139` (`docs/requirements.yaml:8564-8585`) only creates a dataset (presumably v1).
  - No story covers editing records, promoting a new version to production-compare, or rolling back.
- **Closest neighbour**: Extend `FEAT-059` with a versioning story (or accept that `GoldenDataset.version` stays at 1 and drop the field).
- **Severity**: Minor.
