## Top-three highest-impact items

1. **C-4 triple-storage of the org IP allowlist** — since the prior audit, `ApiAccessConfig` gained a third field (`org_ip_allowlist_mode` at line 22935) that duplicates `OrgIpAllowlist.mode`, so the configuration surface is now spread across five fields on two models with two write endpoints. A single edge enforcer cannot decide which row is authoritative.
2. **C-2 FEAT-133 CMK revocation contract is self-contradicting** — STORY-309 AC-001 ("Writes fall back to the emergency key or are paused", line 15484) still contradicts the mode-gated feature description at line 1675 and the per-story fail-closed NFR copy-pasted on STORY-308/309/310 (lines 15467, 15491, 15514). A security-critical action has three incompatible contracts in the same feature.
3. **U-1 audit-signing-key lifecycle has no owning feature** — `AuditLog.signing_key_id` (line 23109) is required to seal every audit row per STORY-266 AC-001 and baseline security (line 33), but no feature in EPIC-010 (or anywhere else) defines how that signing key is issued, rotated, escrowed or revoked. The tamper-evidence contract cannot be honored without it.

### M-1. STORY-263 tenant-contact fallback ignores the baseline self_hosted rule

- Locations: baseline `docs/requirements.yaml:38` ("When a tenant-contact field is unset in self_hosted the event falls back to platform admin; when unset in saas the event falls back to Org.owner"); STORY-263 AC-002 `docs/requirements.yaml:13903-13912`.

Other EPIC-010 stories (STORY-285 line 14892, STORY-289 line 14999, STORY-290 line 15026, STORY-300 line 15258, STORY-305 line 15382) correctly write "falling back to Org.owner in saas and platform admin in self_hosted". ## Cross-epic

### X-1. FEAT-107 billing half vs EPIC-008 FEAT-095/096/098 mode split

See C-1. FEAT-107's plan/commitment/invoices surfaces are unreachable in self_hosted per EPIC-008's explicit 404 rule, but FEAT-107 itself has no mode declaration. Either delete the billing half from FEAT-107 and link to EPIC-008, or add the saas-only clause.

### X-2. FEAT-127 DR vs EPIC-013 FEAT-170 cluster/fleet health

### X-3. FEAT-119 delete-org vs FEAT-177 install lifecycle in self_hosted

In self_hosted the install is one-tenant-per-install, so STORY-274 "Delete organization" (`docs/requirements.yaml:14375-14428`) deletes the sole tenant — effectively decommissioning the install. FEAT-177 (`docs/requirements.yaml:2088-2098`) owns install-level lifecycle but STORY-274 neither declares mode availability nor defers to FEAT-177. What happens to the install when the only Org is `scheduled_for_deletion` is unspecified.

### X-4. FEAT-109 SCIM cascade (STORY-252) depends on FEAT-003/FEAT-004/FEAT-015 but doesn't declare it

### X-5. FEAT-133 CMK data_class enum does not cover EPIC-003 BYOKCredential.ciphertext

Add `provider_credentials` to the enum, or declare that BYOK ciphertext inherits `all` and add an AC that proves the binding on rotation.

### X-6. FEAT-113 PII toggle (STORY-257) vs EPIC-007 FEAT-070 PII detection

### X-7. STORY-280 audit_sink_degraded vs EPIC-009 alerting audience conventions

See M-3. EPIC-009 alerts carry explicit audience resolution; STORY-280 raises an Alert with no audience or mode split, which drops the alert into EPIC-009's default routing — which for tenant-scoped events is per FEAT-099/FEAT-102 but which for operator-plane events should route to platform operator per the baseline. ## Uncited-but-required features

These are features that are not referenced anywhere in `docs/requirements.yaml` but that the surrounding text and models strongly imply must exist. Each subsection states (1) the missing capability, (2) the evidence that implies it, (3) the closest existing neighbor, (4) severity.

### U-1. Audit-log signing-key lifecycle (issue, rotate, escrow, revoke) — **Critical**

1. **What**: A feature that owns the lifecycle of the audit signing key (or keys) used to seal every AuditLog row per the tamper-evidence contract. It must define how the key is created at install time, rotated on a schedule, rolled over without breaking hash-chain verification, escrowed for break-glass verification, and revoked when compromised.
2. **Why the yaml implies it**:
   - `docs/requirements.yaml:23106-23112` — AuditLog now has `prev_hash`, `hash`, and `signing_key_id` fields. `signing_key_id` is nullable only "on genesis rows prior to signing-key issuance".
   - `docs/requirements.yaml:14035-14052` — STORY-266 AC-001 requires "Each row stores prev_hash and a hash over (prev_hash, payload, signing_key_id)" and AC-002 requires a signed export whose verification "reports the break" when a row is tampered.
   - `docs/requirements.yaml:33` — baseline security: "Audit log writes are append-only with tamper-evident hashing".
3. **Closest neighbor**: FEAT-115 (audit sinks) handles *delivery* but not key lifecycle; FEAT-133 (CMK) handles tenant data-at-rest encryption but the audit-signing key is a separate concept (the CMK encrypts data at rest; the audit-signing key seals individual rows). A new FEAT-NNN is needed, most naturally a sibling of FEAT-115 inside EPIC-010, or a sibling of FEAT-133 if the root-of-trust lives in KMS.
4. **Severity**: **Critical** — the tamper-evidence contract cannot be implemented without it, and STORY-266 is blocked by a dependency that has no owner.

### U-2. SCIM conflict inbox model + workflow — **Major**

1. **What**: A persistent ScimConflict (or ScimSyncError) model and the worker that writes it, plus the retry/ignore workflow surfaces used by STORY-251.
2. **Why the yaml implies it**: STORY-251 AC-001 (`docs/requirements.yaml:13327-13334`) requires a UI of persistent per-user conflict rows with email, reason, last attempt, and Retry / Ignore actions. ScimConfig (`docs/requirements.yaml:22878-22905`) stores only sync stats. No ScimConflict model exists in the models section.
3. **Closest neighbor**: FEAT-109 already owns the SCIM surface — the model should be added to that feature's data_models rather than a new FEAT.
4. **Severity**: **Major** — STORY-251 cannot be implemented without it.

### U-3. Restore-request ticket (saas FEAT-127 contract) — **Major**

1. **What**: A ticket template, SLA and routing definition for the "request restore" button that FEAT-127 description reserves for tenants in saas, plus the Antirion-side triage.
2. **Why the yaml implies it**: FEAT-127 description (`docs/requirements.yaml:1620-1623`): "tenant-visible surfaces are read-only timelines plus a 'request restore' button that opens an Antirion-side ticket". No STORY in FEAT-127 backs this; all five stories assume execute-restore is tenant-available.
3. **Closest neighbor**: FEAT-181 (Antirion inbound support queue) is the receiving end, but the ticket *type*, SLA, and template for "tenant requests production restore" are nowhere defined. A new STORY under FEAT-127 ("Request production restore in saas") plus an AC extension in FEAT-181 would suffice.
4. **Severity**: **Major** — the saas role split in FEAT-127 description has no backing story, blocking C-3.

### U-4. Cancel-delete during 30-day grace (FEAT-119) — **Major**

1. **What**: A story with AC, API, and AuditLog entry for cancelling a scheduled org deletion during the 30-day grace window.
2. **Why the yaml implies it**: STORY-274 AC-002 (`docs/requirements.yaml:14400-14402`): "The Org is marked scheduled_for_deletion with a 30-day grace … During grace, any owner can cancel". No follow-up AC, no API path, no AuditLog entry is defined for the cancel action.
3. **Closest neighbor**: STORY-274 itself. A new STORY-NNN under FEAT-119 (or an AC-004 on STORY-274 covering `POST /api/danger/delete/cancel` with confirm-slug) would close the gap.
4. **Severity**: **Major** — ignoring it means an org that starts a delete cannot cleanly reverse it despite the story claiming they can.

### U-5. Break-glass runtime state (ingress-pause, dry-run-all) persistence — **Major**

1. **What**: A feature / model that persists the "ingress paused until T" and "dry-run all enabled" state, so the state survives node restart and propagates to every edge replica within the baseline's config_hot_reload SLA.
2. **Why the yaml implies it**:
   - `docs/requirements.yaml:14315-14369` — STORY-273 AC-001/AC-002 write the pause/dry-run toggles via `/api/danger/ingress/pause` and `/api/danger/dry-run`, but cite only AuditLog in data_models.
   - `docs/requirements.yaml:16` — baseline statelessness: "Gateway and control-plane API nodes hold no request, session, routing or budget state in local memory or disk beyond the lifetime of a single request". The paused-ingress state must therefore live in Redis / Postgres, not on a node.
   - `docs/requirements.yaml:21` — baseline config_hot_reload: "Routing rules, model catalog, safety and budget policies, rate limits, alert rules and feature flags propagate to every node within 30 seconds".
3. **Closest neighbor**: a sibling of FeatureFlag or a field on Org (`ingress_paused_until`, `dry_run_all_enabled`, `dry_run_all_reason`, `dry_run_all_set_by`). A named "Break-glass state" model is the cleaner option; the model should then be read into the routing hot-path with a config_hot_reload-aligned refresh.
4. **Severity**: **Major** — the stories write to a surface that has no persisted shape and no propagation contract.

### U-6. Webhook auto-disable threshold configuration — **Minor**

1. **What**: A configuration surface (model field, API path, story) for the "configured consecutive-failure threshold" at which a webhook auto-disables per STORY-300.
2. **Why the yaml implies it**: STORY-300 AC-001 (`docs/requirements.yaml:15252-15253`): "A Webhook reaches the configured consecutive-failure threshold". Webhook model (`docs/requirements.yaml:23007-23039`) has no `failure_threshold` field. No story configures the threshold; no org-level default exists.
3. **Closest neighbor**: field on Webhook (per-webhook override) and/or on ApiAccessConfig (org default). A small AC on STORY-259 or an extension of STORY-260 would cover configuration.
4. **Severity**: **Minor** — a sensible default ships without configuration, but the story's wording promises configurability.

### U-7. GeoPolicy model and configuration surface for FEAT-131 — **Major**

1. **What**: A GeoPolicy (or CountryPolicy) model for the country allowlist/blocklist and a configuration story that writes it.
2. **Why the yaml implies it**: STORY-306 AC-001 (`docs/requirements.yaml:15400-15407`) references "the blocklist contains 'XX'" and "an allowlist", but STORY-306 data_models (`docs/requirements.yaml:15416-15417`) lists only Request. No GeoPolicy model exists in the models section, and no configuration API path is declared anywhere in FEAT-131.
3. **Closest neighbor**: a sibling of OrgIpAllowlist. FEAT-131 should be expanded with a configuration STORY that writes to `/api/org/geo-policy` and persists on a new model.
4. **Severity**: **Major** — the enforcement story cannot be implemented without the storage model.

### U-8. DlpExportConfig model and configuration surface for FEAT-132 — **Major**

1. **What**: A DlpExportConfig model that persists the S3 bucket, prefix, IAM role, region, and redaction profile for the DLP export target, plus a configuration story.
2. **Why the yaml implies it**: STORY-307 (`docs/requirements.yaml:15421-15442`) says "S3 credentials are configured" but data_models lists only Request (line 15438-15439). AuditSink (`docs/requirements.yaml:23040-23064`) supports `type=s3` but is scoped to audit delivery, not redacted-body export. No DlpExportConfig exists.
3. **Closest neighbor**: either a new DlpExportConfig model or a reuse of AuditSink with a `format=dlp` convention. If AuditSink is reused, FEAT-132 must say so explicitly.
4. **Severity**: **Major** — blocks STORY-307.

### U-9. Ingestion-health alerting (baseline → FEAT-134 or FEAT-115 neighbor) — **Minor**

1. **What**: An AlertRule and runbook for "ClickHouse ingestion gaps" that the baseline's data_durability clause says must exist.
2. **Why the yaml implies it**: `docs/requirements.yaml:30` — baseline data_durability: "ClickHouse ingestion targets >= 99.99% row durability for the request log; ingestion gaps trigger the ingestion-health alert". "The ingestion-health alert" is singular and definite, implying a specific named AlertRule. No STORY in EPIC-010 or EPIC-009 declares this rule by name.
3. **Closest neighbor**: EPIC-009 FEAT-099 (alert rule management). AlertRule.scope_kind now supports `platform`, so an ingestion-health AlertRule with `org_id=null` fits. A STORY under FEAT-099 or FEAT-134 should install the canonical rule.
4. **Severity**: **Minor** — the baseline implies it but the detection can be added during implementation without blocking a separate story.

### U-10. Org-level signing-secret lifecycle and distribution — **Major**

1. **What**: A defined storage and distribution contract for the "org signing secret used for webhook HMAC defaults" that STORY-275 AC-003 rotates.
2. **Why the yaml implies it**: STORY-275 AC-003 (`docs/requirements.yaml:14460-14469`): "An org signing secret exists (used for webhook HMAC defaults) … A new signing secret is generated and shown once / The previous secret continues to validate for a 5-minute replay window". No field on Org carries this secret; Webhook.secret_hash (line 23025) is per-webhook, not org-scoped. There is no OrgSigningSecret model.
3. **Closest neighbor**: either a new OrgSigningSecret model with rotation fields (`secret_hash`, `previous_secret_hash`, `previous_valid_until`, `rotated_at`, `rotated_by_user_id`) or a set of fields on Org. The webhook delivery code needs to know which org-level secret to apply when a Webhook is created without its own secret.
4. **Severity**: **Major** — STORY-275 AC-003 cannot be implemented without a storage contract.

### U-11. Integration credentials encryption binding — **Minor**

1. **What**: An NFR or binding that declares Integration.credentials must be KMS-wrapped at rest.
2. **Why the yaml implies it**: Integration.credentials (`docs/requirements.yaml:23135-23137`) is `json nullable` with no encryption hint. The baseline security clause (line 33) says secrets live in KMS and never appear in persisted payloads. The OAuth tokens this field stores fit "secrets" exactly.
3. **Closest neighbor**: same remediation as M-9 for AuditSink.credentials — rename to `credentials_ciphertext`, add a CmkBinding data_class, and add an NFR to FEAT-116. No new FEAT is needed.
4. **Severity**: **Minor** — the baseline covers it, but the model field's loose type invites plaintext storage.
