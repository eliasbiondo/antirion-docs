## Critical findings

### C-1. STORY-017 typed-confirm flow references `Team.slug`, which the `Team` model does not have

- STORY-017 AC-001 (`docs/requirements.yaml:3003`):
  > "I click 'Delete team' and type the team slug to confirm"
- STORY-017 AC-004 (`docs/requirements.yaml:3027–3033`):
  > "I type a value that does not match the team slug and click Confirm"
- STORY-017 API (`docs/requirements.yaml:3047–3049`):
  > `request.body.confirm_slug: string`
- `Team` model (`docs/requirements.yaml:21253–21276`) has `id, org_id, name, color, lead_user_id, created_at` — and **no `slug` field**.

The guard "Confirm button stays disabled … Type the team slug to confirm" cannot be implemented against a column that does not exist, and the API body `confirm_slug` has nothing to diff against. `Org.slug` exists (line 20936) but that is the workspace slug, not a team's. The `destructive_action_confirmation_policy` baseline (line 5) explicitly enumerates "delete team" as requiring typed-slug confirmation, so dropping the typed confirmation is not an acceptable resolution.

*Fix*: add `Team.slug (string, unique within (org_id, slug))` and generate it on team create (STORY-013), mirroring `Org.slug`. Then STORY-017 AC-004's disabled-button check has a field to compare against.

### C-2. STORY-017 AC-001 asserts a soft-delete on `Team` but `Team` has no soft-delete column

- STORY-017 AC-001 (`docs/requirements.yaml:3005`):
  > "The Team is soft-deleted"
- `Team` model (`docs/requirements.yaml:21253–21276`) has no `deleted_at` / `archived_at` / `status` field.
- Comparable model: `Project.deleted_at` at `docs/requirements.yaml:22592–22595` (nullable timestamp with description "Soft-delete timestamp; Projects remain visible in audit history after delete").

A soft-delete without a column either silently downgrades to a hard delete (violating AC-001's "team budgets are released back to the Org" semantics — a hard delete would also wipe `Request.team_id` FKs and budget attribution history) or is implemented via an undocumented side table. STORY-017 AC-002/AC-003/AC-005 explicitly check pre-existing references (ApiKeys, members, Projects) to guard the delete, which is only coherent if the team row survives the delete with a marker.

*Fix*: add `Team.deleted_at (timestamp, nullable)`, mirror `Project.deleted_at`'s description, and state in STORY-017 AC-001 that the Team row is retained with `deleted_at` set.

## Major findings

### M-1. FEAT-012 declares a rejection contract with no story to enforce it

- FEAT-012 description (`docs/requirements.yaml:447`):
  > "The platform admin role is … rejected by the role API if requested in saas mode. Role assignment attempts that violate these constraints produce an AuditLog 'role.assignment_rejected' entry."
- STORY-031..STORY-034 (`docs/requirements.yaml:3576–3694`) cover listing built-in roles, creating / editing / deleting custom roles — none exercises the "assigning platform_admin in saas is rejected" path, and none writes `role.assignment_rejected` to `AuditLog`.
- STORY-012 AC-002 (`docs/requirements.yaml:2749–2756`) covers "non-owner cannot change owner" but not the platform-admin-in-saas rejection.

The only testable mode-gated rule in the epic has no acceptance criteria. Any implementation can ship without this guard and still pass the ACs.

*Fix*: add an AC to STORY-012 (or a new story under FEAT-012) for "attempt to assign role='platform_admin' in saas mode returns 400 and writes an AuditLog row with action='role.assignment_rejected'", citing `project.deployment_modes` and `User.role` enum.

### M-2. STORY-015 bulk add-members silently re-assigns `User.team_id` without warning or audit line

- STORY-015 AC-002 (`docs/requirements.yaml:2889–2897`):
  > "I add 2 existing Users to the team … Both Users' team_id is updated to this team … the team's member count becomes 5"
- `User.team_id` (`docs/requirements.yaml:21010–21012`) is a single nullable FK, so a user is a member of at most one team.

A User already on another team is silently evicted from that team when added here; no warning, no AC about the source team's `member count`, no `AuditLog` line on the team the user was taken *from*. This is the only story in the epic that mutates team membership for already-assigned users at bulk, and it produces silent side effects on other teams' budgets.

*Fix*: STORY-015 AC-002 should state "Users already on another team are moved; the source team is debited accordingly and both teams record an AuditLog 'team.member.move' entry." Either that, or reject the add when the user is already on another team and require STORY-014 "Change a user's team" as the single move-path.

### M-3. STORY-020 "Reset MFA" does not notify the target user and has no `Notification` data_model

- STORY-020 AC-001 (`docs/requirements.yaml:3162–3171`): admin deletes all `MfaFactor` rows for a user; only `AuditLog` is written.
- STORY-023/024 (`docs/requirements.yaml:3295–3404`) set the pattern: a security-relevant admin action against a user's identity emits `Notification.kind="security"` to the target user and records the resolved audience in `AuditLog`.
- `project.notifications_model` (`docs/requirements.yaml:37–38`) requires any notification-bearing event to name its canonical audience.

MFA reset is the same class of event (admin-initiated identity change on a target user) yet STORY-020 is silent on user notification. The target user would not know their factors were wiped until their next sign-in.

*Fix*: add AC to STORY-020 "A Notification of kind 'security' ('mfa.reset') is emitted to the target User (owning principal) per project.notifications_model" and add `Notification` to `data_models`.

### M-4. STORY-022 flag-user AuditLog does not record the resolved audience list that STORY-023/024 require

- STORY-022 AC-001 (`docs/requirements.yaml:3249–3251`) emits a `Notification.kind="security"` to `Org.security_contact` with the canonical fallback chain — good — but the paired `AuditLog` line is:
  > "An AuditLog entry records actor, reason and action 'user.flag'"
- STORY-023 AC-001 (`docs/requirements.yaml:3317`) and STORY-024 AC-001 (`docs/requirements.yaml:3377`) each state:
  > "records the resolved audience list"
- `project.notifications_model` (`docs/requirements.yaml:38`):
  > "Every audit record on a notification-bearing event records the resolved audience list and the channel attempt."

STORY-022's flag is a notification-bearing event, yet its AC does not mandate recording the audience list. Same gap on STORY-022 AC-002 ("user.unflag"): even if no notification is emitted there, the audit line should still be explicit about audience (empty set) to stay diffable against the policy.

*Fix*: STORY-022 AC-001 should add "and records the resolved audience list", and AC-002 should state the emitted audience (or `audience_snapshot=[]`) to keep routing correctness diffable per the baseline.

### M-5. STORY-022 unflag does not require a cross-notification signalling "flag cleared"

- STORY-022 AC-001 emits `Notification.kind="security"` on flag.
- STORY-022 AC-002 (`docs/requirements.yaml:3252–3260`) returns User.status to "active" with only an `AuditLog` line; no Notification is emitted.

The original flag produced a user-visible security notification to `Org.security_contact`. Clearing it leaves the same audience uninformed that the incident is resolved. `notifications_model` collapse rules treat the flag class consistently; the unflag event should either re-notify the same audience with a resolution state or explicitly declare the notification audience as `[]` with a rationale.

*Fix*: add "A Notification of kind 'security' ('user.flag_cleared') is emitted to the audience of AC-001 per project.notifications_model" to STORY-022 AC-002, or add a non_functional line stating that unflag is intentionally not notification-bearing.

### M-6. STORY-018 offboarding remains a click-confirm even though its effect is irreversible in practice

- STORY-018 AC-001 (`docs/requirements.yaml:3082–3088`):
  > "I click 'Offboard' on the user and confirm"
- The same AC revokes all Sessions, revokes all type='user' ApiKeys, sets `User.status = "offboarded"`.
- `project.destructive_action_confirmation_policy` (`docs/requirements.yaml:5`) enumerates the list of actions requiring typed-name confirmation — "delete team, delete org, delete project, delete alert rule with active fires, delete model with live traffic, purge cache, rotate org secret" — and does not include "offboard user".

In enumerated effects, offboarding is equivalent to "delete team + rotate org secret for the target user's keys" and is functionally irreversible (you cannot un-revoke a Session or un-rotate an ApiKey). The policy either does not cover it intentionally (in which case STORY-018 is fine) or this is a gap. Previously flagged (m-8, deferred) — still open.

*Fix*: either add "offboard user" to `project.destructive_action_confirmation_policy` and STORY-018 AC-001 (type the target's email to confirm, API body `confirm_email: string`), or add an explicit non_functional line on STORY-018 stating that offboarding is intentionally click-confirm because STORY-019 blocks the irreversible tail (shared-key transfer) as a prerequisite.

## Minor findings

### m-1. STORY-012 AC-002 "The Role is unchanged" should be "User.role is unchanged"

`docs/requirements.yaml:2755–2756`. The mutation target is a User's `role` field, not the `Role` model. Using "Role" (capitalized) invites the reader to think a `Role` row is involved. STORY-012 AC-003 (`docs/requirements.yaml:2764–2765`) uses "the role is unchanged" (lowercase) — mixing the two forms within a single story.

*Fix*: replace with "User.role is unchanged" in both AC-002 and AC-003 for consistency.

### m-2. STORY-016 "Team.members" is not a declared collection on the `Team` model

`docs/requirements.yaml:2969`. The `Team` model has no `members` field; membership is represented by `User.team_id`. STORY-015 correctly talks about "Users' team_id" but STORY-016 reaches for a collection name that does not exist.

*Fix*: restate as "My User.team_id becomes null" to match the actual mutation.

### m-3. `Invite.status` has no declared enum despite the stories using four values

- `Invite.status` (`docs/requirements.yaml:21248–21249`): `type: string` with no description.
- Stories reference `status = "pending"` (STORY-007 AC-001, line 2519), `"accepted"` (STORY-008 AC-001, line 2579), `"revoked"` (STORY-010 AC-001, line 2665). STORY-008 AC-002 (line 2581–2589) implies an `"expired"` state via `expires_at` but never asserts a status transition.

Not a contradiction — the values used are internally consistent — but the other lifecycle enums in the epic (e.g. `User.status`, `ApiKey.status`) have inline enum descriptions; `Invite.status` should match for symmetry and to make the valid transitions greppable.

*Fix*: add description `pending|accepted|revoked|expired` to `Invite.status`.

### m-4. STORY-005 non_functional block carries boilerplate unrelated to JIT provisioning

`docs/requirements.yaml:2424–2427`. The first two lines are the generic "Membership and role changes are audited … propagate within the config-reload SLA" line that appears on dozens of stories; only the third line ("SSO assertion signature, audience and NotOnOrAfter validated on every callback") is JIT-specific. The prior audit closed the same class of issue on STORY-002 (m-1); the same trim applies here.

*Fix*: drop the two generic lines on STORY-005 and keep the SSO-specific guarantee, or add JIT-specific guarantees (e.g. "JIT writes are idempotent per SSO subject", "default role is validated against `User.role` enum before write").

### m-5. STORY-026 "An initial credential is issued" leaves the ApiKey.type ambiguous

`docs/requirements.yaml:3457`. The `ApiKey.type` enum is `service|user|ephemeral` (`docs/requirements.yaml:21378–21379`). A service account is expected to own `type='service'` keys (consistent with STORY-019's filter), but STORY-026 AC-001 does not assert it — an implementation could issue a `user`-typed key to a service account User and STORY-026 would still pass.

*Fix*: restate as "An initial ApiKey with `type='service'` is issued and shown once".

### m-6. STORY-023 AC-001 does not specify that `LoginAttempt.user_id` is populated on lock

`docs/requirements.yaml:3316`. `LoginAttempt.user_id` is nullable (`docs/requirements.yaml:23298–23300`). On a 5th failure against a known user the user_id is obviously known; leaving it optional invites an implementation that only populates it on success. The sibling AC-002 for IP throttle (line 3327) says "No LoginAttempt targeting a real user is created on that try" — that correctly implies user_id is null there, so the symmetry should be made explicit for AC-001.

*Fix*: add "with `user_id` set to the locked user" to AC-001's LoginAttempt bullet.

### m-7. STORY-004 domain-enforcement check overlaps `Org.sso_enforced` vs `SsoConfig.require_for_domains` without declaring precedence

- STORY-004 AC-002 (`docs/requirements.yaml:2353–2360`):
  > "The workspace enforces SSO for domain 'acme.example' … I see 'This workspace requires SSO sign-in for @acme.example'"
- `Org.sso_enforced` (`docs/requirements.yaml:20989–20991`): "When true, password sign-in is blocked for users in `require_for_domains`."
- `SsoConfig.require_for_domains` (`docs/requirements.yaml:22858–22859`): the per-domain list.

Both are necessary to compute the blocked state, but no story states the evaluation order (does unsetting `Org.sso_enforced` short-circuit `require_for_domains`, or does a non-empty `require_for_domains` imply enforcement?). Minor because one reading is natural; still worth pinning.

*Fix*: STORY-004 AC-002 non_functional should cite "Password sign-in is blocked iff `Org.sso_enforced=true` AND user's email domain is in `SsoConfig.require_for_domains`."

### m-8. STORY-007 role field is accepted as an unvalidated string on the invite

`docs/requirements.yaml:2545`. The POST body passes `role: string`. STORY-032/033 prevent privilege escalation at role-edit time, but STORY-007 has no AC rejecting an invite with an unknown or off-limits role name (e.g. `platform_admin` in saas, or a custom role from a different Org). The Invite lifecycle inherits whatever role string is persisted.

*Fix*: add AC to STORY-007: "Invites with role not in the Org's resolvable role list return 400" — covers both unknown names and saas-mode `platform_admin`.

## Uncited-but-required features

### U-1. (Critical) `SessionPolicy` admin-edit surface — no feature lets anyone configure lockout or session timeouts

- **Capability**: an admin screen plus API to read and update the per-Org `SessionPolicy` row (`idle_timeout_minutes`, `absolute_max_hours`, `lockout_threshold`, `lockout_window_minutes`, `lockout_duration_minutes`).
- **Why implied**: STORY-023 AC-001 (`docs/requirements.yaml:3309`) reads `SessionPolicy.lockout_threshold` and `lockout_window_minutes`; STORY-023 non_functional (`docs/requirements.yaml:3353`) states "Thresholds configurable via org-level SessionPolicy … with safe defaults"; the `SessionPolicy` model exists at `docs/requirements.yaml:23757–23777`. No story in EPIC-001 (or elsewhere in the document) exposes a write path. Without it, the fields are seeded once and never changeable — "configurable" is not delivered.
- **Nearest neighbor**: no good fit inside EPIC-001. FEAT-007 (User offboarding), FEAT-009 (Lockout) and FEAT-012 (Role management) are the closest; logically this belongs under a new FEAT in EPIC-001 ("Account-security policy") or adjacent to EPIC-010's admin surfaces. A new FEAT is warranted.
- **Severity**: Critical. A configuration contract that has no writable surface is functionally broken, and FEAT-009's promise of "configurable threshold" (`docs/requirements.yaml:417`) goes unmet.

### U-2. (Critical) Session idle-timeout and absolute-max enforcement — fields exist but no feature enforces them

- **Capability**: on every authenticated request, validate `Session.last_active_at` against `SessionPolicy.idle_timeout_minutes` and `Session.created_at` against `SessionPolicy.absolute_max_hours`; expire the session on either breach.
- **Why implied**: `SessionPolicy.idle_timeout_minutes` and `absolute_max_hours` are declared (`docs/requirements.yaml:23763–23766`) and `Session.last_active_at` / `expires_at` exist (`docs/requirements.yaml:21135–21138`). FEAT-003 (Sign-out and session revocation, `docs/requirements.yaml:346–354`) owns *manual* revocation only — STORY-006 covers user-initiated sign-out but not timeout-driven expiry. No STORY writes to `Session.expires_at` based on the policy.
- **Nearest neighbor**: FEAT-003. Either extend it with a new STORY (and an AC "Sessions with `last_active_at` older than `SessionPolicy.idle_timeout_minutes` are rejected with 401 and a forced re-authentication") or create a sibling feature under EPIC-001.
- **Severity**: Critical. The two fields are load-bearing for any real-world deployment's compliance story; without an enforcer they are decorative.

### U-3. (Major) Transactional email delivery — invite and password-reset emails have no owning feature

- **Capability**: deliver transactional product emails (invite, invite resend, password reset, email verification) to recipients who may not have a User row yet, with templating, retry, and DKIM/SPF-clean delivery.
- **Why implied**: STORY-003 AC-001 (`docs/requirements.yaml:2296`) "A reset email is sent to my address"; STORY-007 AC-001 (`docs/requirements.yaml:2518`) "An invite email is sent to alex.chen@acme.example"; STORY-009 AC-001 (`docs/requirements.yaml:2616`) "A new invite email is sent". FEAT-102 Notification channels (`docs/requirements.yaml:1363–1369`) routes *event* notifications per-user through `NotificationPreference`, but invite and reset recipients don't have a `NotificationPreference` yet — they are not signed-in users. No feature in the document defines transactional email delivery as a capability; the provider, template inventory, retry policy, bounce handling and dead-letter are all unspecified.
- **Nearest neighbor**: FEAT-102 is the closest but structurally wrong for this (it is user-addressed, not email-addressed). A new feature under EPIC-001 or EPIC-010 is warranted ("Transactional email delivery"), or FEAT-102 must be expanded to cover pre-signin recipients by email address.
- **Severity**: Major. Three stories depend on email being sent; a release that ships STORY-003/007/009 cannot pass its ACs without this capability, yet nothing owns the work.

### U-4. (Major) TrustedBrowser list / revoke surface for end users

- **Capability**: a user-facing list of their own `TrustedBrowser` rows with a per-row "Revoke" action; automatic revocation triggered on password change, MFA reset and admin session revocation.
- **Why implied**: STORY-002 non_functional (`docs/requirements.yaml:2275`): "Trust is revoked on password change, MFA reset or admin session revocation." `TrustedBrowser` model (`docs/requirements.yaml:21144–21161`) has a fingerprint and expiry but no story writes to it beyond STORY-002's creation path. There is no STORY that lists a user's trusted browsers, no STORY that revokes one, and no non_functional or AC in STORY-006 (revoke other sessions), STORY-018 (offboard), STORY-020 (reset MFA) or STORY-038 (change password, EPIC-002) that cascades to `TrustedBrowser`.
- **Nearest neighbor**: FEAT-015 (Personal security and MFA, EPIC-002) for the self-service list/revoke; FEAT-003 (Sign-out and session revocation) for the cascade on password/MFA/session events.
- **Severity**: Major. Without a cascade, the promise "trust revoked on password change" in STORY-002 is untestable; without a list surface, a user whose browser was fingerprinted cannot see or revoke the grant.

### U-5. (Major) `platform_admin` grant and install-bootstrap path in self_hosted mode

- **Capability**: the mechanism by which the first `platform_admin` principal is seeded during a self_hosted install, and subsequent grants/revocations of `platform_admin`.
- **Why implied**: FEAT-012 (`docs/requirements.yaml:447`): "The platform admin role is a reserved built-in that exists only in self_hosted mode, cannot be granted to an ordinary workspace member, cannot be created as a custom role … and is rejected by the role API if requested in saas mode." `User.role` enum includes `platform_admin` (`docs/requirements.yaml:21035`). STORY-031..034 cover built-in listing and custom role CRUD but nothing covers platform_admin grant. EPIC-013 covers platform-operator administration but does not define the `platform_admin` seeding path either.
- **Nearest neighbor**: FEAT-177 (install wizard, EPIC-013 per `docs/requirements.yaml:1504` reference). Either FEAT-012 or FEAT-177 must own this; today neither does.
- **Severity**: Major. Without a seeding path there is no way to bootstrap a self_hosted install; without a grant story the rejection contract in FEAT-012 has no counterpart on the grant side.

### U-6. (Major) Credential-rotation notification to the service account's human owner

- **Capability**: emit a `Notification.kind="security"` to `User.human_owner_user_id` when a service account's credential is rotated (STORY-028), revoked, or when the account itself is deleted (STORY-029).
- **Why implied**: `project.notifications_model` (`docs/requirements.yaml:37–38`) names the "owning principal on the resource" (audience b) as a canonical notification role. `User.human_owner_user_id` (`docs/requirements.yaml:21075–21078`) exists specifically to identify that person for service-account Users. STORY-023 / STORY-024 use this exact pattern for user lockout/unlock notifications. STORY-028 (rotate) and STORY-029 (delete) both pivot on machine-owned credentials whose compromise or removal would directly affect the human owner — yet neither emits a notification.
- **Nearest neighbor**: FEAT-011 (Service account lifecycle). Add a notification AC to STORY-028 and STORY-029 per the notifications_model pattern. (If notifications_model is intentionally scoped away from service-account events, then `User.human_owner_user_id` is the contract that says otherwise and should be reconciled.)
- **Severity**: Major. The ownership field exists specifically to drive this notification; leaving it unused for rotation/delete is a silent violation of the notifications_model policy.

### U-7. (Minor) `LoginAttempt` retention / sweep

- **Capability**: a retention policy and background worker that prunes or archives `LoginAttempt` rows older than N days.
- **Why implied**: STORY-001 AC-002 (`docs/requirements.yaml:2177–2183`) and STORY-023 AC-001/AC-002 (`docs/requirements.yaml:3316, 3322–3327`) each write a `LoginAttempt` row per sign-in attempt. Under a credential-stuffing attack the IP-throttle path fires up to 20 rows in 5 minutes per hostile IP; over time this table grows without bound. `project.non_functional_baseline.workers` (`docs/requirements.yaml:18`) mandates retention sweeps as a first-class worker responsibility, but no feature owns this sweep for `LoginAttempt`. The `RetentionConfig` model (`docs/requirements.yaml:22988`, owned by EPIC-010) has no `login_attempts_days` field.
- **Nearest neighbor**: EPIC-010 retention configuration (FEAT-123 area). Either add `RetentionConfig.login_attempts_days` plus a sweep worker, or make it explicit in FEAT-009 that `LoginAttempt` is kept forever for forensic reasons (in which case sharding/archive must be spec'd instead).
- **Severity**: Minor. Does not block ACs but the gap is inevitable at any meaningful traffic level.

### U-8. (Minor) `User.status = "inactive"` has no lifecycle story transitioning into or out of it

- **Capability**: a background sweep that transitions dormant Users (no `last_active_at`/`last_login_at` within a configured threshold) into `inactive`, and/or a user-visible path to return them to `active` on next sign-in.
- **Why implied**: `User.status` enum (`docs/requirements.yaml:21038`) declares `active|pending|flagged|inactive|offboarded`. Four of the five values have at least one transition story: `active` (STORY-001/007), `pending` (STORY-007), `flagged` (STORY-022), `offboarded` (STORY-018). `inactive` has no writer and no reader anywhere in the document.
- **Nearest neighbor**: FEAT-007 (User offboarding) or FEAT-008 (User activity and flagging) could extend to cover the dormancy case, or the enum value should be removed if there is no plan to use it.
- **Severity**: Minor. Either the feature is missing or the enum value is dead code.
