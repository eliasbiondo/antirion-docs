## Summary

EPIC-011 is the runtime data plane: every line of `non_functional_baseline` applies. The earlier audit's most damaging items (story-level p95 budgets that violated the baseline, the swapped FEAT IDs in `shared_limiters`, the dangling `STORY-394`/`FEAT-101`/`STORY-134` cross-refs, the streaming-retry contradictions and the limiter-algorithm split) have been resolved in the current text and are not re-flagged here.

What remains, after a fresh pass, is mostly structural debt that the prior audit deferred:

1. **Pipeline ordering is still nowhere written down.** The epic title is "Gateway request pipeline" but no FEAT and no story define where cache, safety, budget admission and concurrency caps sit in the request path. `Request.cached`, `Request.safety_flag`, `Request.pii_flag` are nullable columns with no producer feature.
2. **Failover/retry/circuit-breaker ownership remains split three ways** — FEAT-051 (config), FEAT-139 (retry+failover), FEAT-142 (circuit). The descriptions claim a clean split, but `STORY-121` AC-002 and `STORY-336` AC-001 still both write `circuit_state` and both raise `circuit.open`.
3. **Several models referenced conceptually have no row.** `IngressDialect`, `RateLimit`, `IdempotencyKey`, `RetryPolicy`, per-phase `Timeout`, per-model circuit state, break-glass-key marker, AccessLog/rejection metadata.
4. **Mode-availability still missing on `FEAT-158`/`FEAT-159` for the parts that diverge** — the `mode: both` field is now present (good), but the descriptions still describe behavior that materially differs between modes without separating the concerns.
5. **Several stories assume capabilities not provided by any other feature**: per-tenant concurrency cap (baseline line 34), readiness-check circuit-breaker hydration (baseline line 23), residency-aware cache lookup contract, dialect-shape error contract, OTel↔Request id contract.

Counts: **3 critical**, **9 major**, **8 minor**, **6 cross-epic**, **6 missing/dangling**, **9 uncited-but-required**.

The top three highest-impact items:

- **C1 / U1** — No feature owns canonical request-pipeline ordering. Every downstream feature (cache, safety, budget, idempotency) silently chooses its own position. Implementations will diverge on tail-latency, cost-attribution, billing and safety semantics.
- **C2** — `STORY-336` AC-003 demands per-model circuit state, but `ProviderHealth.circuit_state` is keyed by `(provider_id, region)` only. There is no row to hold per-model breaker state.
- **M1** — Failover / circuit-breaker / retry are still co-owned by `STORY-121` (FEAT-051), `STORY-329` (FEAT-139) and `STORY-336` (FEAT-142). `STORY-121` AC-002 still raises `circuit.open` despite FEAT-051 being declared "config-only".

## Critical

### C1. No feature defines the canonical gateway pipeline order

- Cites: EPIC-011 description (line 261), `Request` model fields `pii_flag`, `safety_flag`, `cached`, `cost` (lines 21796–21799, 21763, 21765), baseline `config_hot_reload` listing "safety and budget policies" (line 21).
- Quoted (line 261): `The runtime that serves traffic — ingress dialects, translation, edge auth, rate limits, retries and timeouts, runtime failover, outbound provider auth, cost attribution, idempotency, health and circuit breaker, telemetry.`
- Quoted (line 21796): `pii_flag: bool` and (line 21798): `safety_flag: bool`.
- Verification of absence: there is no FEAT under EPIC-011 titled "pipeline order" or "request path", and no story under any of FEAT-135..FEAT-161 defines whether (a) cache lookup runs before or after safety, (b) budget admission runs before or after routing, (c) input safety runs pre-translation or post-translation, (d) idempotency lookup runs before or after edge auth. `STORY-345` (line 17000) reads `CacheEntry` only as an outage fallback — implying cache is *not* read on the happy path, which contradicts the entire EPIC-006 promise.
- Problem: The four pipeline stages with material correctness consequences (idempotency, cache, safety, budget admission) have no declared position. Tail-latency, cost-attribution correctness, billing semantics and safety guarantees all depend on the order. (E.g. cache hit before safety = leaks unsafe content from history; safety before cache = wastes safety latency on cache hits; budget admission after routing = wastes routing latency on a request that will be rejected.)
- Fix: Add a new feature in EPIC-011 (or extend the epic description) that fixes the canonical order, e.g.: ingress → edge auth (FEAT-138) → size limit (FEAT-147) → rate-limit (FEAT-138) → idempotency lookup (FEAT-141) → input safety (FEAT-070/FEAT-071) → cache lookup (FEAT-060) → budget admission (FEAT-084) → routing (FEAT-049/FEAT-050) → translation (FEAT-137) → outbound auth (FEAT-140) → outbound limiter (FEAT-161) → upstream call → output safety → cost attribution (FEAT-141) → telemetry (FEAT-143).

### C2. `STORY-336` AC-003 demands per-model circuit state but `ProviderHealth` cannot hold one

- Cites: STORY-336 AC-003 (line 16659), `ProviderHealth` model (line 23232).
- Quoted (line 16659–16666): `Per-model vs per-provider scope … A rule specifies circuit scope "per-model" … One model on a provider is failing while others are healthy … Only the failing model's circuit opens; others stay closed`.
- Quoted (line 23232 onwards): `ProviderHealth … circuit_state … closed|open|half-open` — fields are `provider_id`, `region`, `status`, `circuit_state`, `error_rate_5m`, `last_latency_ms`, `consecutive_failures`, `last_probed_at`, `cooldown_ends_at`. No `model_id`, no per-model scope.
- Problem: The AC is unimplementable against the declared schema. A per-model breaker has nowhere to live; multiple per-model breakers cannot co-exist on the same `(provider_id, region)` row.
- Fix: Either (a) add `model_id: FK:Model nullable` to `ProviderHealth` and make the row's effective scope `(provider_id, region, model_id)` with `model_id IS NULL` for provider-scope breakers, or (b) split into `ProviderHealth` and a new `ModelCircuitState(provider_id, model_id, circuit_state, …)`. STORY-336 AC-003 must then say which row writes which.

### C3. `STORY-329` AC-002 returns 502 directly without invoking dialect-shape rule

- Cites: STORY-329 AC-002 (line 16372), STORY-346 (line 17047).
- Quoted (line 16379): `The client receives 502 with error code "all_targets_failed"`.
- Quoted (line 17047 onwards): STORY-346 requires every graceful-degradation response to be dialect-shaped (OpenAI, Anthropic or Antirion-native).
- Problem: STORY-329 AC-002 runs *before* graceful degradation (it fires at the end of the fallback chain when no cached response is configured). It does not call out the dialect-shape rule, so a multi-dialect client (e.g. an Anthropic SDK that called `/v1/messages`) will see a non-dialect-conforming 502. Either route `all_targets_failed` through `FEAT-148`'s graceful degradation path (and then STORY-346 applies), or restate the dialect-shape rule on STORY-329 AC-002.
- Fix: Add an AC on STORY-329 mandating dialect-shape response for `all_targets_failed`, or add an AC on STORY-345/346 that all-targets-failed flows through `FEAT-148` even when no cached response exists.

## Major

### M1. FEAT-051 is declared "config-only" but `STORY-121` still does runtime work

- Cites: FEAT-051 description (line 851), STORY-121 AC-002 (line 7915), STORY-336 AC-001 (line 16640).
- Quoted FEAT-051 (line 851): `Per-rule trigger taxonomy and fallback-chain configuration on errors, timeouts and rate-limits; runtime (retry execution, circuit open/close transitions, cluster-wide state) is owned by FEAT-139 and FEAT-142.`
- Quoted STORY-121 AC-002 (line 7915–7923): `Circuit breaker opens … The circuit breaker for "X" is open and the first fallback "Y" is tried directly … An AlertEvent "circuit.open" is generated`.
- Quoted STORY-336 AC-001 (line 16640–16649): `Breaker opens on error-rate threshold … The circuit state for that provider becomes "open" … An AlertEvent "circuit.open" is generated`.
- Problem: Both stories own the same runtime event (`circuit.open` AlertEvent fan-out) and both gate the same routing decision. STORY-121 contradicts the "config-only" promise the FEAT-051 description now makes. Two stories that fire identical alerts on the same condition will double-page on-call.
- Fix: Strike STORY-121 AC-002 (the runtime portion) and rewrite STORY-121 to test only the *configured* fallback chain semantics, with a forward-link to STORY-329/336 for runtime behavior. Or: strike STORY-336 AC-001's `AlertEvent` fan-out and reference STORY-121 AC-002 as the canonical fan-out site.

### M7. `STORY-385` `binding_layer` freshness is unspecified

- Cites: STORY-385 AC-001 (line 18142), `ProviderRateLimitWindow.binding_layer` field (line 21657).

### M8. `STORY-339` "/metrics returns 404 from public ingress" reveals the listener exists

- Cites: STORY-339 AC-002 (line 16781).

## Uncited-but-required features

This section lists capabilities that the current EPIC-011 text strongly implies must exist for the epic to be coherent, but for which no FEAT or model is declared.

### U1. Pipeline-stage ordering feature (Critical)

1. **Capability**: A canonical, single-owner declaration of the EPIC-011 request-pipeline order (auth → size limit → rate-limit → idempotency → input safety → cache → budget admission → routing → translation → outbound auth → outbound limiter → upstream → output safety → cost attribution → telemetry).
2. **Why implied**: EPIC-011 description (line 261) lists nine ordered concerns ("ingress dialects, translation, edge auth, rate limits, retries and timeouts, runtime failover, outbound provider auth, cost attribution, idempotency, health and circuit breaker, telemetry") but no FEAT and no story states the order. `Request.pii_flag`, `safety_flag`, `cached`, `cost` (lines 21796–21765) are nullable columns with no producer-feature claim.
3. **Closest neighbor**: A new FEAT under EPIC-011, e.g. `FEAT-NNN — Gateway pipeline ordering`. No existing feature is a clean fit (FEAT-138/139/140 each own one phase).
4. **Severity**: Critical. Implementations will diverge on tail-latency, cost-attribution correctness, billing semantics and safety guarantees.

### U2. Rejected-request audit / `AccessLog` model (Major)

1. **Capability**: A model row that captures pre-`Request` rejections (auth failure, size cap violation, IP allowlist violation) with enough fields to satisfy EPIC-004 analytics, key-anomaly detection, and incident forensics — without polluting `Request` with `status=401` rows that were never assigned a model/provider.
2. **Why implied**: STORY-323 AC-001 (line 16091) says `No Request row is written beyond the rejection metadata` — but "rejection metadata" is undefined, no model exists, and `Request` (line 21722) has `model_id`/`provider_id` declared non-null. STORY-324 AC-002 (line 16170) records an `AuditLog` entry on IP allowlist failure, but `AuditLog` is for security events, not analytics. STORY-344 AC-001 (line 16968) says a `Request` row is written for 413 — contradictory: some rejections are written to `Request`, some are not.
3. **Closest neighbor**: A new `AccessLog` model. Or relax `Request.model_id`/`provider_id` to nullable and document the rejection rows.
4. **Severity**: Major. Affects analytics, billing reconciliation and incident response.

### U3. Per-tenant concurrency-cap feature (Major)

1. **Capability**: Per-tenant bounded concurrency over the gateway data plane (org-scoped), distinct from per-key rpm/tpm rate limits and from per-node concurrency caps.
2. **Why implied**: `non_functional_baseline.tenant_isolation` (line 34) requires `Per-tenant quotas bound concurrency, CPU-seconds, queue slots and storage so a noisy tenant cannot degrade others.` `concurrency_and_backpressure` (line 19) covers per-node only. FEAT-138 covers rpm/tpm only. Streaming connections (`non_functional_baseline.streaming`, line 31) count against per-node caps but the AC text never asserts a per-tenant cap.
3. **Closest neighbor**: FEAT-138 (which already enforces edge auth/rate-limit) or a new `FEAT-NNN — Per-tenant concurrency and backpressure`. Distinct from FEAT-161 (which is *outbound* limiter, not inbound concurrency).
4. **Severity**: Major. Without it, a single tenant streaming hot can degrade every co-tenant on the same node, contradicting the baseline.

### U4. Readiness-probe and circuit-breaker hydration feature (Major)

1. **Capability**: The gateway data plane's readiness contract — readiness blocks until config is loaded, migrations are applied, dependencies are reachable, and circuit-breaker state is hydrated from the shared store.
2. **Why implied**: `non_functional_baseline.health_checks` (line 23) explicitly requires `(dependencies reachable, config loaded, migrations applied, circuit-breaker state hydrated)`. No EPIC-011 story owns this contract. FEAT-142 covers steady-state breaker transitions (STORY-336 AC-002 cooldown→half-open) but not startup hydration. FEAT-167 (EPIC-012) tracks worker-fleet health but is not the data-plane readiness probe.
3. **Closest neighbor**: A new story under FEAT-142, or a new FEAT under EPIC-011 owning health/readiness across the data plane.
4. **Severity**: Major. A node that joins the load balancer with `circuit_state="closed"` despite a real breaker being open will trigger the very provider hammering FEAT-142 exists to prevent.

### U5. Residency-aware cache lookup contract (Major)

1. **Capability**: A request-path AC that requires cache lookup to be partitioned by the resolved residency region of the requesting tenant.
2. **Why implied**: `non_functional_baseline.multi_region_and_dr` (line 29) — `Cross-region failover honors residency per FEAT-130 and fails closed rather than crossing a residency boundary.` STORY-345 AC-001 (line 17014) added the residency check on the *outage* fallback path. There is no equivalent assertion on the *happy* path of cache lookup. `CacheEntry` (line 22176) carries `org_id` (good) but the lookup story does not name a region partition or `ResidencyConfig.region` predicate.
3. **Closest neighbor**: A new AC on FEAT-060 STORY-142, or a new EPIC-011 story tying cache lookup to FEAT-130 residency on the happy path.
4. **Severity**: Major. Cross-region cache reads on the happy path silently violate residency, even with the outage path correctly fixed.

### U6. `ApiKey.kind`/`Request.break_glass` extension feature (Major)

1. **Capability**: A first-class marker on `ApiKey` that the key is a break-glass key (so edge auth can skip standard scope checks and tag every request `break_glass=true`), and a `Request.break_glass` column to record it.
2. **Why implied**: STORY-378 AC-001 (line 17836) — `Every request is tagged break_glass=true`. `ApiKey.type` (line 21377) lists only `service|user|ephemeral`; no `break_glass` value. `Request` (line 21722) has no `break_glass` column. Without these, the audit and incident-postmortem flows that rely on filtering "show me every request issued under a break-glass key during the incident window" cannot run.
3. **Closest neighbor**: Extend `ApiKey.type` enum to include `break_glass`, add `Request.break_glass: bool` column. Or move the entire feature to EPIC-003 (per M2) and let FEAT-022 own the schema change.
4. **Severity**: Major.

### U7. Pipeline-stage timeout model (Major)

1. **Capability**: Per-phase timeout configuration distinct from `RoutingRule.timeout_ms` (which is whole-request only).
2. **Why implied**: STORY-328 (line 16307) requires three distinct timeouts — connect, first-byte, stream-idle — but `RoutingRule` (line 22000) only has `timeout_ms: int` and `retries: int`. The story assumes a model that supports the per-phase split; no model row provides it. Fallback re-issue (STORY-329) per project.streaming baseline (line 31) also has no field for "deadline budget remaining after fallback re-issue".
3. **Closest neighbor**: Either extend `RoutingRule` with `connect_timeout_ms`, `first_byte_timeout_ms`, `stream_idle_timeout_ms`, or add a new `TimeoutPolicy` model that `RoutingRule` references.
4. **Severity**: Major.

### U8. `IngressDialect` / endpoint-state model (Major)

1. **Capability**: A model that tracks which ingress endpoints are enabled per org, which dialects are supported, and the current admin enable/disable state.
2. **Why implied**: STORY-317 (line 15812) enables/disables an ingress endpoint via `AuditLog "endpoint.disable"` — but no model row records the endpoint's current state. After a node restart, the disable is unrecoverable from the audit log alone (audit log is append-only). FEAT-152 (line 1834) — `Defines scope and basic ops for Assistants/Responses dialects if in scope for the workspace plan` — implies a per-workspace dialect-set; nothing models it.
3. **Closest neighbor**: A new `IngressEndpointState(org_id, endpoint, enabled, disabled_at, disabled_by_user_id, reason)` model. Or extend `ApiAccessConfig` with a `disabled_endpoints: json` field.
4. **Severity**: Major. Without it, a disabled endpoint silently re-enables on rolling restart, contradicting `non_functional_baseline.zero_downtime_deploy` (line 24).

### U9. `IdempotencyKey` model (Minor)

1. **Capability**: A first-class model row for in-flight and completed idempotency-key records, with body-hash, response snapshot, TTL and tenant scope.
2. **Why implied**: STORY-334 (line 16540) asserts `Idempotency entries stored in Redis with 5-minute TTL (configurable up to 24h)` (line 16593). No model row exists. The baseline `shared_limiters` (line 20) names FEAT-141 as the owner of cluster-wide Idempotency-Key lookup, but no schema row backs it.
3. **Closest neighbor**: A new `IdempotencyKey(id, org_id, key, body_hash, status, response_snapshot, ttl_at)` model. Redis is the runtime store but the schema row is the contract.
4. **Severity**: Minor. The Redis-only storage is implementable, but every other shared-store contract in the document has a schema row (e.g. `ProviderRateLimitWindow` for the limiter; CacheEntry for cache); this one is the odd one out.
