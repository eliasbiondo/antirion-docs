# EPIC-006 Caching — re-audit 2026-04-20

Scope: `docs/requirements.yaml` — EPIC-006 (line 159), FEAT-060..FEAT-068 (lines 938–1027), STORY-142..STORY-156 (lines 8631–9383), the cache models (`CacheConfig` 22074, `SemanticCacheConfig` 22111, `PromptSegment` 22138, `CacheEntry` 22176, `CacheEvictionEvent` 22231, `InvalidationRule` 22258, `CacheOverride` 22283, `CacheWarmupJob` 23498), and cross-references into EPIC-005 routing (STORY-118 line 7787, STORY-119 line 7816), EPIC-007 safety (STORY-159 line 9487), EPIC-008 budgets, EPIC-010 residency (FEAT-130 line 1651), EPIC-011 gateway pipeline (FEAT-141 line 1744, FEAT-144 line 1771, FEAT-148 line 1799, STORY-332 line 16454, STORY-341 line 16827, STORY-345 line 17000), EPIC-012 worker fleet (FEAT-167 line 1991), and the project baseline (lines 1–73).

All line numbers are from the current `docs/requirements.yaml` (24,929 lines). This is a fresh pass — earlier findings from `docs/audit/epic-006-review.md` are not assumed to still hold; each one was re-checked against the current file before being kept, dropped, or restated.

## Summary

EPIC-006 has absorbed most of the original schema-level fixes (CacheEntry now carries `org_id`/`user_id`/`provider_id`/`byok_credential_id`/`semantic_cache_config_id`/`streamed`; CacheConfig has `eviction_policy`/`max_size_mb`; `CacheEvictionEvent` model exists). What remains is a layer of **drift between updated and un-updated stories**, **schema fields that no story uses**, **schema descriptions that promise contracts the schema does not realize**, and **eviction reasons that have no firing path**. Counts: **8 Critical**, **15 Major**, **11 Minor**, **7 Cross-epic**, **10 Missing references**, **9 Uncited-but-required features**.

Top three highest-impact items:

1. **C-1 — STORY-148 BYOK-rotation cache invalidation cites a broken FK chain** (`Request.api_key_id -> ApiKey.owner -> BYOKCredential`). `ApiKey.owner` is not a field — only `ApiKey.owner_user_id` exists, and it is `FK:User`, not `FK:BYOKCredential`. The fallback path resolves to nothing, so any CacheEntry written before `byok_credential_id` was added is unreachable from this purge.
2. **C-5 — Cache writes are not suppressed when `SafetyEvent.action = "block"`.** STORY-159 (line 9487) returns 451 for blocked prompts; no story in EPIC-006 says the cache must skip writing the response. Combined with the cache key being computed over the redacted prompt (STORY-142 AC-001, line 8649), a blocked outcome can become a cache entry on the next non-blocked variant.
3. **C-3 — Latency NFR drift across the epic.** STORY-142 (8703) and STORY-147 (8976) cite the corrected "500 µs p95 sub-budget"; the other 11 stories (143, 144, 145, 146, 148, 149, 150, 151, 152, 153, 156) still cite the legacy "Cache lookups complete within 5 ms p95 on a hit". Cache-lookup latency is now a 10× internal contradiction in the same epic.

## Critical

### C-1. STORY-148 cites a broken FK chain for BYOK-rotation cache invalidation

- Location: STORY-148 AC-001 line 8995; `ApiKey` model 21354–21440; `BYOKCredential` model 21464–21510.
- Quote (8995): `An async purge deletes CacheEntry rows where byok_credential_id = the rotated credential (or, for rows without byok_credential_id, rows matched via Request.api_key_id -> ApiKey.owner -> BYOKCredential)`.
- Problem: `ApiKey` has no field named `owner` and no FK pointing to `BYOKCredential`. The closest field is `ApiKey.owner_user_id` (line 21365), which is `FK:User`. There is no schema path from a `Request` (which carries `api_key_id`) to the `BYOKCredential` that authenticated the upstream call. The fallback path described in the AC therefore resolves to nothing — CacheEntries written before the schema acquired `byok_credential_id` (and any rows where the field is null because resolution happened post-cache-write) are silently never invalidated. The "automatic" promise in the story narrative is hollow for that population.
- Fix: either (a) add `byok_credential_id` to `Request` and rewrite AC-001 to use `Request.byok_credential_id` directly; or (b) declare the matching path as `(org_id, provider_id) joined to BYOKCredential.provider_id`, accepting a coarser purge; or (c) require backfill of `CacheEntry.byok_credential_id` before STORY-148 can ship and document the migration.

### C-2. CacheConfig description claims a (model_id, scope, scope_id) key but the schema has neither field

- Location: `CacheConfig` model 22074–22110.
- Quote (22082): on `model_id`, `Not globally unique — one config per (model_id, scope, scope_id); scope-level overrides live in CacheOverride`.
- Problem: `CacheConfig` has fields `id, model_id, backend, key_strategy, ttl_sec, max_entry_kb, max_items, max_size_mb, eviction_policy, enabled, note`. No `scope` or `scope_id` field exists. The description promises a composite key that the model cannot enforce. A reader implementing this is forced to either (a) treat `model_id` as effectively unique again (re-introducing C-2 from the previous audit) or (b) invent the missing fields. Either way the description is unimplementable as stated.
- Fix: add `scope: string` (enum `model|team|project|env|byok`) and `scope_id: uuid nullable` to `CacheConfig`, with a partial-unique index over `(model_id, scope, scope_id)`; or rewrite the description to declare the keying contract that the schema actually realizes.

### C-3. Cache-lookup latency NFR has drifted across stories — 5 ms vs 500 µs in the same epic

- Location: STORY-142 line 8703 (`<= 500 µs p95`), STORY-147 line 8976 (`<= 500 µs p95`); STORY-143 line 8758, STORY-144 line 8820, STORY-145 line 8861, STORY-146 line 8901, STORY-148 line 9002, STORY-149 line 9027, STORY-150 line 9086, STORY-151 line 9146, STORY-152 line 9190, STORY-153 line 9245, STORY-156 line 9382 (`<= 5 ms p95 on a hit`).
- Problem: Two of the 15 stories cite the corrected sub-budget; the other 11 cite the legacy 5 ms value. The 5 ms number is 10× the gateway-added p95 baseline (`performance_slo` line 25: `<= 1 ms p95 end-to-end`), so it is not just inconsistent with STORY-142 but inconsistent with the project SLO. A reader cannot tell which budget the implementation is held to.
- Fix: rewrite the NFR line on the 11 stale stories to match STORY-142/STORY-147, or hoist it onto the epic so individual stories do not need to re-state it.

### C-4. Eviction reasons `safety_policy_change` and `manual_purge` have no firing path

- Location: STORY-156 AC-001 line 9368 (`reason (policy|ttl|size|safety_policy_change|manual_purge)`); `CacheEvictionEvent.reason` line 22246; STORY-275 AC-002 lines 14450–14459 (workspace-wide cache purge).
- Problem: STORY-275 is the only "manual purge" surface in the file and it writes only `AuditLog` action `"cache.purge"` (line 14459). No story produces a `CacheEvictionEvent` with `reason = "manual_purge"`, so the AC-002 evictions tab will show zero rows for purges. Similarly, `safety_policy_change` is not emitted by any EPIC-007 story — DetectionRule edits (EPIC-007), PiiCategory changes (EPIC-007) and TeamSafetyPolicy changes nowhere invalidate cached entries. Two of the five enumerated reasons are dead enum values.
- Fix: add an AC to STORY-275 that fans out a `CacheEvictionEvent` per purged entry (or one batched roll-up event) with `reason = "manual_purge"`; add a story under FEAT-064 (or in EPIC-007) that emits `CacheEvictionEvent` with `reason = "safety_policy_change"` when `DetectionRule` or `PiiCategory` changes invalidate matching entries.

### C-5. Cache write is not suppressed when SafetyEvent.action = "block"

- Location: STORY-159 AC-002 lines 9506–9527 (PII block path); EPIC-006 stories — none mention safety-block suppression.
- Quote (9514): `The gateway returns 451 with Request.status_label = "safety_blocked"`.
- Problem: STORY-159 returns 451 to the client and records a SafetyEvent of action "block". No EPIC-006 story states that a blocked request must not be written to the cache. With the cache key formula being `(org_id, resolved_model_id, resolved_provider_id, redacted_prompt)` (STORY-142 AC-001, line 8649), and redaction running before the block decision, a blocked variant can leave a cache key reservation that is later treated as a hit, or a near-miss prompt can poison the cache. No story makes the contract explicit.
- Fix: add an AC to STORY-142 (or a new story under FEAT-060) stating that when `SafetyEvent.action ∈ {block}` for a request, no `CacheEntry` row is written and no entry is updated. State the invariant as: "writes to CacheEntry are conditional on Request.status < 400 and SafetyEvent.action ∉ {block}".

### C-6. Streaming × cache replay semantics are still undefined despite the new `streamed` field

- Location: `CacheEntry.streamed` line 22206; `non_functional_baseline.streaming` line 31; STORY-341 (FEAT-144) line 16827–16872; STORY-345 AC-001 line 17009; no EPIC-006 story.
- Problem: `CacheEntry.streamed: bool` was added (presumably in response to the prior audit's C-6) but no story consumes it. The contract is undefined for: (a) does a streaming request (`Request.streamed=true`) produce a `CacheEntry` at all, given the baseline forbids buffering (line 31)? (b) on a hit for a stream-shaped request, does the gateway re-frame the stored body as SSE chunks with the 15 s keep-alive cadence (STORY-341 AC-001, line 16845)? (c) what is `Request.ttft_ms` (line 21782) for a cache hit on a streamed request? (d) does FEAT-148 graceful degradation (STORY-345) re-frame as SSE if the original request was streamed? Adding the field without an owning story leaves the implementation under-specified.
- Fix: add a story under FEAT-060 that defines: write path may buffer on the worker side after the stream completes (never on the forward path); hit-path replays as SSE with `cached: true` SSE comment; `ttft_ms` is set to the cache-lookup latency; cross-references STORY-341 and STORY-345.

### C-7. STORY-142 cache-key formula only makes sense for backend = exact

- Location: STORY-142 AC-001 line 8649; CacheConfig.backend description line 22085 (`exact|prompt_prefix|semantic`).
- Quote (8649): `The cache key is computed over (org_id, resolved_model_id, resolved_provider_id, redacted_prompt) - not over aliases or raw prompts`.
- Problem: The formula is universal in the AC ("the cache key is computed over …") but only correct for `backend = exact`. For `backend = prompt_prefix`, the key is over a prefix or canonicalized template fragment. For `backend = semantic`, there is no prompt-substring key — there is an embedding lookup with a similarity threshold (`SemanticCacheConfig.threshold`, line 22125). Stating the exact-cache key formula as the universal rule contradicts the epic's own three-backend taxonomy (line 161) and the model's `backend` enum.
- Fix: split AC-001 by backend — three sub-ACs, one per backend value. Or factor the cache-key derivation into a separate per-backend description and have AC-001 cite it.

### C-8. STORY-148 does not pin invalidation timing to the BYOK rotation lifecycle

- Location: STORY-148 AC-001 lines 8993–8996; `BYOKCredential.successor_id` (line 21490) and `rotation_grace_ends` (line 21494).
- Problem: STORY-148 says the purge runs "when the rotation completes" but does not say what "completes" means — start of grace, end of grace, or successor activation? `BYOKCredential` supports a zero-downtime rotation window where both old and successor credentials are valid until `rotation_grace_ends`. If invalidation fires at the start of grace, the cache loses entries that the successor credential could have kept producing for. If invalidation fires only after grace ends, the old credential's responses are still served from cache for the entire grace window — the security claim "responses produced with an old credential are not served" (story narrative line 8985) is violated. The story is silent.
- Fix: state the firing point explicitly. Recommended: invalidation fires at the moment `BYOKCredential.status` transitions to `revoked` (after `rotation_grace_ends`); during the grace window, hits served from cache are tagged with the rotated credential, not the successor, and a SafetyEvent / AuditLog entry records the served-from-old-credential disclosure.

## Major

### M-1. STORY-152 wire type for `override_value` disagrees with the model

- Location: STORY-152 API body line 9176 (`override_value: string`); `CacheOverride.override_value` line 22297 (`type: json`).
- Problem: The story declares the override value as a string on the wire, but the model stores JSON. A team that wants to override `ttl_sec: 600` must send `override_value: "600"` and have it parsed, or the model field is being misused. Either way the contract is wrong.
- Fix: change the API body to `override_value: json` (matches model) or change the model field to a discriminated `value_string|value_number|value_bool` shape (matches strict OpenAPI consumers).

### M-2. STORY-156 declares no GET endpoint, but AC-002 requires browsing the eviction history

- Location: STORY-156 lines 9350–9383.
- Quote (AC-002, 9374–9376): `I open the Evictions tab on a CacheConfig / I see the last 24h of events with reason breakdown`.
- Problem: No `api:` block is declared on STORY-156 at all. The UI cannot read `CacheEvictionEvent` rows without a server endpoint.
- Fix: add `GET /api/cache/configs/:id/evictions?from=&to=` returning `events: [CacheEvictionEvent]` with `reason_breakdown: object`.

### M-3. CacheConfig PATCH lives at two different paths

- Location: STORY-142 PATCH at `/api/cache/models/:id` (line 8680); STORY-155 PATCH at `/api/cache/configs/:id` (line 9334).
- Problem: Same model (`CacheConfig`) edited from two URLs. Either the implementation has two parallel routers (and they will drift) or one of these is dead.
- Fix: pick one — `/api/cache/configs/:id` is the more honest naming because `CacheConfig` is not strictly per-model after the (model_id, scope, scope_id) keying lands (C-2). Update STORY-142.

### M-4. CacheOverride has only POST — no GET, PATCH, or DELETE

- Location: STORY-152 lines 9168–9184.
- Problem: A team can create an override but cannot list, change, disable or delete one. The model has `edited_by_user_id` (22304) and `edited_at` (22307) implying a mutation surface that no story owns.
- Fix: extend STORY-152 (or split into two stories) to add `GET /api/cache/overrides`, `PATCH /api/cache/overrides/:id`, `DELETE /api/cache/overrides/:id`.

### M-5. STORY-148 has no `depends_on` on the BYOK-rotation source story

- Location: STORY-148 lines 8979–9003.
- Problem: STORY-148 fires on BYOK rotation but does not declare dependency on STORY-056 (BYOK rotation per the EPIC-003 audit log). Reviewers cannot infer ordering.
- Fix: add `depends_on: [STORY-056]` to STORY-148.

### M-6. STORY-148 emits no Notification despite being security-sensitive

- Location: STORY-148 lines 8987–8996; `notifications_model` lines 37–38.
- Problem: A BYOK credential rotation that triggers a cache purge is a security-sensitive event class. The notifications baseline requires every event-notification story to cite canonical roles per deployment mode. STORY-148 only writes an AuditLog. STORY-159 (line 9516) is the model story for this kind of fanout (security_contact, privacy_officer, with saas trust_and_safety fanout).
- Fix: add an AC emitting `Notification kind = security` to `Org.security_contact` per the notifications_model collapse rules.

### M-7. STORY-149 emits no Notification despite being a compliance change

- Location: STORY-149 lines 9013–9027.
- Problem: A retention-policy shrink that purges cache entries is a compliance event. No Notification is raised. Audience would naturally be `Org.privacy_officer` and `Org.compliance_officer`-equivalent roles per the notifications_model.
- Fix: add an AC emitting `Notification kind = security|compliance` to `Org.privacy_officer` (and Antirion-staff `dpo` queue in saas).

### M-8. STORY-151 (edit/delete invalidation rule) emits no Notification despite being a policy change

- Location: STORY-151 lines 9088–9147.
- Problem: An invalidation rule is a cache policy. EPIC-007 policy-change stories (e.g. STORY-159) emit Notifications because `Approval.kind` includes `policy-change` (line 23198). STORY-151 has no Notification AC and no Approval flow.
- Fix: at minimum, write an `AuditLog` action `"cache.invalidation_rule.edit"` (model has `target_type` 23089), and emit a Notification to `Org.security_contact` for rules whose `action ∈ {drop-user, drop-team}`.

### M-9. InvalidationRule has no `org_id` — rules are tenant-unscoped at the schema level

- Location: `InvalidationRule` model 22258–22282.
- Problem: Every other tenant-owned model in the file carries `org_id` (e.g. CacheEntry 22182, CacheEvictionEvent 22237, BYOKCredential 21470, Notification 24696→user_id→User.org_id). InvalidationRule does not. With STORY-150's POST endpoint accepting only `name, match, action, scope` (lines 9059–9062) and no implicit tenant binding, rules are effectively global — a rule in tenant A could fire on a tenant-B event if `match: json` is broad enough. Direct contradiction with `tenant_isolation` (line 34).
- Fix: add `org_id: FK:Org` (non-null) to `InvalidationRule`. Add `team_id: FK:Team, nullable: true` for team-scoped rules. Index on `(org_id, scope)`.

### M-10. CacheConfig has no `org_id`; tenant isolation is transitive through Model.org_id

- Location: `CacheConfig` 22074–22110.
- Problem: `CacheEntry` was hardened with a direct `org_id` (per the prior audit's C-1 fix, now line 22182). `CacheConfig` was not. Tenant isolation is transitive through `Model.org_id`, which is fragile (e.g. if a Model is reassigned, all its CacheConfig overrides belong to the wrong tenant). Operators querying configs by org need a join.
- Fix: add `org_id: FK:Org` (non-null) directly to `CacheConfig`.

### M-11. SemanticCacheConfig has no `org_id`

- Location: `SemanticCacheConfig` 22111–22137.
- Problem: Tenant isolation is transitive through `team_id` (which has org_id). Same fragility as M-10. Reassigning a team to another org silently relocates all semantic cache configs.
- Fix: add `org_id: FK:Org` (non-null).

### M-12. PromptSegment has no `org_id`

- Location: `PromptSegment` 22138–22175.
- Problem: Transitive through `model_id` only. Same critique.
- Fix: add `org_id: FK:Org` (non-null).

### M-13. CacheOverride has no `org_id`

- Location: `CacheOverride` 22283–22308.
- Problem: `target_id: uuid` (22293) is polymorphic across `team|key|project|env|byok` (22291) — there is no way to enforce that all overrides for an org are listable without joining through `target_id` per scope. Same critique as M-9..M-12.
- Fix: add `org_id: FK:Org` (non-null).

### M-14. STORY-150 POST `scope: string` has no enum and no implicit tenant binding

- Location: STORY-150 API line 9063 (`scope: string`); InvalidationRule.scope line 22271 (no enum description).
- Problem: Combined with M-9 (no org_id), an `InvalidationRule` is created with a free-text scope and no tenant. The acceptance tests cannot validate the scope value.
- Fix: declare `scope: string` enum on the model (`org|team|project|user|key`) and on the API body; include `org_id` (server-derived from the auth token).

### M-15. STORY-147 typed-slug confirmation is conditioned on >100 entries but `confirm_slug` is unconditional in the API body

- Location: STORY-147 AC-002 lines 8924–8930; API body line 8962 (`confirm_slug: string`).
- Problem: The AC says typed confirmation is required when "the matching set exceeds 100 entries", but the API contract makes `confirm_slug` mandatory always. Either the AC threshold is wrong, or the API field should be optional with server-side enforcement at the threshold. Inconsistency between AC and wire schema.
- Fix: pick one — recommended: keep `confirm_slug` mandatory always (matches `project.destructive_action_confirmation_policy` line 5: every cache purge is "high-impact destructive") and rewrite AC-002 to drop the >100 conditional.

## Cross-epic

### X-1. STORY-345 graceful-degradation does not specify which cache backend is acceptable

- Location: STORY-345 AC-001 line 17013–17020.
- Quote (17013): `The org has graceful degradation set to "serve-cached"`.
- The AC says "A matching CacheEntry exists" with residency match, but does not say whether the match must be `backend = exact` or whether `prompt_prefix` / `semantic` (with what threshold) is acceptable in degraded mode. If semantic is allowed under degradation, a near-paraphrase with cosine 0.85 could be served as the "outage response" — the operational tradeoff is undocumented.

### X-2. STORY-275 manual cache purge writes only AuditLog, never CacheEvictionEvent

- Location: STORY-275 AC-002 lines 14454–14459.
- Quote (14457–14459): `Every CacheEntry in the workspace is deleted / The next matching request misses and regenerates / An AuditLog entry records action "cache.purge"`.
- STORY-156 enumerates `manual_purge` as an eviction reason; STORY-275 produces no `CacheEvictionEvent` rows. The Evictions tab will be empty after a Danger-Zone purge of millions of entries.
- Fix: add an AC to STORY-275 that emits one `CacheEvictionEvent` per purged entry (or a roll-up event with `count` / `aggregate=true` semantics added to the model).

### X-3. EPIC-007 redaction policy changes do not invalidate cache

- Location: STORY-159 (EPIC-007) lines 9487–9527; no EPIC-006 story.
- The cache key uses `redacted_prompt` (STORY-142 AC-001, line 8649), so when redaction strictness or `DetectionRule.method` changes (DetectionRule supports versioning per the previous EPIC-007 audit), entries generated under the prior policy continue to be served — but their `redacted_prompt` no longer corresponds to anything the new policy would produce. Stale cached responses persist.
- Fix: emit `CacheEvictionEvent` with `reason = "safety_policy_change"` when `DetectionRule` or `PiiCategory.action` changes; pin a `safety_policy_version` on `CacheEntry` to bound the staleness window.

### X-4. EPIC-008 budget enforcement on cache hits is only addressed for warmup

- Location: STORY-154 AC-001 line 9265 (`upstream provider charges (for managed or BYOK) still apply and still decrement budget counters by Request.cache_cost on subsequent hits`); no general story.
- The warmup story now states that subsequent hits decrement the budget by `Request.cache_cost`. The general cache-hit case (a regular request that hits cache) is silent — does it decrement? Per the cost formula in STORY-332 (line 16512) the cost is non-zero on a hit. STORY-145 AC-001 (line 8838) says the cache_cost field is populated, but no story says EPIC-008 (`Budget`/`BudgetEvent`) decrements on hits.

### X-5. EPIC-005 shadow / canary × cache is undefined

- Location: STORY-118 (canary) lines 7787–7815; STORY-119 (shadow) lines 7816–7852; no EPIC-006 story.
- Shadow traffic (STORY-119 AC-001 line 7836: response is discarded) — does it write to the cache? If yes, shadow traffic prefills the primary's cache with shadow-model responses. If no, shadow path must skip cache lookup too. Canary traffic similarly: if canary hits the same key as primary, a 10% canary ramp can prefill 100% of cache reads with canary responses. Neither story addresses cache interaction. The previous audit's X-1 / X-2 was logged as "cache-key pinning applied" via STORY-142 AC-001 (which now uses `resolved_model_id`), but that only addresses the alias-vs-resolved question — it does not say shadow/canary suppresses writes.

### X-6. CacheEvictionEvent is not consumed by EPIC-009 alerting

- The model exists, but no AlertRule schema or story expresses "alert when eviction rate spikes" or "alert when safety_policy_change evicted > N entries". A spike caused by a runaway invalidation rule would only be visible in the per-config Evictions tab (STORY-156 AC-002). Operability gap.

## Uncited-but-required features

These are capabilities the surrounding context strongly implies must exist, but which no FEAT in the file currently owns. One subsection per proposed missing feature.

### U-1. Streaming-aware cache replay (Critical)

- What is missing: a feature owning the contract for serving cached SSE/chunked responses. This includes (a) whether streaming requests may write to cache; (b) on a hit, how the stored body is re-framed as SSE chunks with the keep-alive cadence; (c) how `Request.ttft_ms` is set; (d) interaction with FEAT-148 graceful degradation.
- Why the yaml implies it: `CacheEntry.streamed: bool` (line 22206) was added to the schema, with the description `True when the stored body originated from a streaming request (affects replay framing)`. That description points at framing logic that no feature owns. `non_functional_baseline.streaming` (line 31) prohibits buffering on the forward path. STORY-341 (line 16827) defines SSE behavior with 15 s keep-alive. STORY-345 (line 17000) serves cached responses on outage. None of them define cache-stream interaction.
- Closest neighbor: FEAT-060 (cache config). A new story under FEAT-060, or a new FEAT-069-equivalent dedicated to "cache replay semantics", would be the cleanest home.
- Severity: Critical — the streamed flag is dead schema until a feature consumes it.

### U-2. Safety-policy change → cache invalidation (Critical)

- What is missing: a feature that fires a `CacheEvictionEvent` (with `reason = "safety_policy_change"`) when a `DetectionRule` or `PiiCategory.action` changes.
- Why the yaml implies it: STORY-156 AC-001 (line 9368) enumerates `safety_policy_change` as a valid eviction reason. STORY-159 (line 9487) describes inline redaction whose policy changes can render cached entries stale. The cache key is computed over `redacted_prompt` (STORY-142 AC-001, line 8649), so a redaction-policy change desyncs the key from current behavior. No story or feature owns the firing path.
- Closest neighbor: FEAT-064 (invalidation rules) or a new EPIC-007 cross-link. A new story under FEAT-064 is the smallest fix.
- Severity: Critical — silently serves responses produced under deprecated safety policies.

### U-3. Cache-write suppression on safety blocks (Critical)

- What is missing: a feature that prohibits cache writes when `SafetyEvent.action ∈ {block}`.
- Why the yaml implies it: STORY-159 AC-002 (line 9514) returns 451 with `Request.status_label = "safety_blocked"`. No EPIC-006 story states the cache write is suppressed. Without this, a near-prompt that just barely passes filtering can be poisoned by an earlier blocked variant's response.
- Closest neighbor: FEAT-060. A single AC on STORY-142 would suffice, or a new explicit story.
- Severity: Critical — security/correctness.

### U-4. Invalidation-event catalog (Major)

- What is missing: a model + feature defining the canonical set of events that `InvalidationRule.match` can fire on.
- Why the yaml implies it: STORY-150 (line 9041) cites `session.logout`. STORY-151 narrative (line 9102) cites `role.change`. `InvalidationRule.match: json` (line 22266) accepts arbitrary JSON, so the UI cannot offer autocomplete or validate at write time. FEAT-102 (notifications) is the analogue for notifications and presumably owns its own event catalog; cache invalidation has none.
- Closest neighbor: FEAT-064 (invalidation rules). Add an `InvalidationEvent` model and an `events:` list under FEAT-064, or cross-reference whichever feature owns the org-wide event registry.
- Severity: Major — without a catalog, every rule is opaque to the UI and to authors.

### U-5. Scope-keyed CacheConfig keying (Major)

- What is missing: a feature that owns the `(model_id, scope, scope_id)` keying contract that CacheConfig's description claims to support.
- Why the yaml implies it: CacheConfig description on `model_id` (line 22082) asserts the keying. CacheOverride.scope (line 22291) enumerates `team|key|project|env|byok`. No feature owns how a request resolves its effective CacheConfig at lookup time when multiple scopes apply. Precedence is documented in CacheOverride description (22284) but only for overrides, not for config selection itself.
- Closest neighbor: FEAT-065 (team and key cache overrides). Either expand FEAT-065 to own per-scope CacheConfig resolution, or add a new feature for "cache config resolution and precedence".
- Severity: Major — the description promises a keying that the schema and stories do not deliver.

### U-6. Cache-event Notification routing (Major)

- What is missing: a feature defining audience and channel for cache invalidation, purge and over-quota events.
- Why the yaml implies it: STORY-148 (BYOK rotation invalidation) and STORY-149 (retention shrink purge) have no Notification ACs even though notifications_model (lines 37–38) requires them for security and compliance event classes. FEAT-102 owns notification-channel routing for user-addressable roles, but no cache-event story names an audience.
- Closest neighbor: FEAT-102. A list of cache-event Notification kinds and audiences should be added to its event catalog.
- Severity: Major — events occur and are not surfaced to the right humans.

### U-7. Cache invalidation worker job ownership (Major)

- What is missing: a feature explicitly assigning async cache purges (STORY-148, STORY-149, STORY-150) to a worker fleet.
- Why the yaml implies it: `non_functional_baseline.workers` (line 18) enumerates "cache eviction" among worker-fleet responsibilities. FEAT-167 (worker fleet health) owns worker-fleet observability. No FEAT explicitly says "cache invalidation runs on the worker fleet with idempotent handlers, retries, dead-letter, etc.". STORY-148 says "an async purge deletes" but does not declare the runtime.
- Closest neighbor: FEAT-064 or FEAT-167. A non_functional sentence on FEAT-064 referencing FEAT-167 would close it.
- Severity: Major — operational owner of the purge pipeline is undefined.

### U-8. Per-region cache topology (Major)

- What is missing: a feature owning cache replication and residency-honoring failover.
- Why the yaml implies it: `non_functional_baseline.multi_region_and_dr` (line 29) requires every stateful dependency to be deployed across at least three AZs with automatic failover and residency-honoring failover. Cached responses are stateful. FEAT-130 (residency runtime enforcement, line 1651) rejects outbound calls that would cross residency, and STORY-345 (line 17014) requires the cached entry to be in the request's resolved residency region. No feature owns the cache topology, eviction-on-region-pin, or cross-AZ replication of cache shards.
- Closest neighbor: FEAT-068 (eviction policies) or a new EPIC-010 feature. The cache-topology contract is missing.
- Severity: Major — cache durability and residency-pinning behavior are unowned.

### U-9. Cache-event AlertRule integration (Minor)

- What is missing: an AlertRule category for cache eviction spikes, hit-rate cliffs, and runaway invalidation-rule firings.
- Why the yaml implies it: `CacheEvictionEvent` exists; `InvalidationRule.firings_24h` (22281) exists; STORY-153 produces hit-rate metrics. None feed any AlertRule.
- Closest neighbor: EPIC-009 alerting. A cache-event category on AlertRule would close it.
- Severity: Minor — operability gap, not correctness.
