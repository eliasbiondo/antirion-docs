### C1. STORY-089 JSONL/Parquet export bypasses the RBAC, audit and notification controls STORY-088 now enforces

- **Where**: lines 6467–6498 (STORY-089) vs. 6404–6466 (STORY-088).
- **Quoted (STORY-088 AC-001, line 6419)**: `The caller holds the "requests.export" scope`.
- **Quoted (STORY-088 AC-001, line 6426)**: `An AuditLog entry records actor, filter set, row count and action "requests.export"`.
- **Quoted (STORY-088 AC-002, line 6436)**: `On completion a Notification is delivered to the acting principal via FEAT-102 channel routing`.
- **Quoted (STORY-089 AC-002, line 6492–6493)**: `Parquet files are written to s3://bucket/prefix/ and I get a completion notification`.
- **Quoted (STORY-089 data_models, line 6494–6495)**: `data_models: - Request`.
- **Problem**: STORY-089 declares `depends_on: [STORY-088]` (line 6475–6476) but its ACs do not inherit or restate STORY-088's gates. There is no `requests.export` scope check, no AuditLog write (STORY-089 data_models omits both AuditLog and Notification), and the completion notification is described as an unrouted "completion notification" rather than FEAT-102 channel-routed. JSONL and Parquet exports are strictly higher-volume / higher-risk than CSV — exporting to a customer-owned S3 bucket is a durable exfiltration channel — so the absence of these gates is a regression from the harder control path STORY-088 won. The S3 credential path (`s3://bucket/prefix/`) is also not reconciled with `project.non_functional_baseline.security` ("Secrets live in the configured KMS and never appear in logs, telemetry or persisted payloads"): the write target credentials have no model, no owner and no audit shape.
- **References**: STORY-088 (line 6404) and FEAT-102 (line 1363) both exist. AuditLog (line 23065) and Notification (line 24690) both exist.
- **Fix**: Add an AC to STORY-089 requiring (a) `requests.export` scope; (b) an AuditLog entry with action `"requests.export"` and context including format and destination; (c) completion Notification delivered via FEAT-102 to the acting principal. Add `AuditLog, Notification` to `data_models`. Consider splitting "Parquet → S3" into its own story with a declared `SupportedExportTarget` or `AuditSink`-style model (Webhook / Integration) so the destination credential is a first-class, audited resource.

### C2. STORY-102 Tool-use analytics AC cannot be served from its declared data model

- **Where**: lines 6960–6981 (STORY-102), 765–771 (FEAT-043), 21787–21789 (Request.tool_count), 21972–21999 (ToolInvocation).
- **Quoted (STORY-102 AC-001, line 6976)**: `I see top 20 tools with invocations, failure rate and p50/p95 latency`.
- **Quoted (STORY-102 data_models, line 6977–6978)**: `data_models: - Request`.
- **Problem**: "Top 20 tools" requires grouping by tool name/id and aggregating per-tool invocation outcome and latency. The current `Request` model stores only `tool_count: int` (line 21787) — a count, not a tool dimension. Per-tool aggregation belongs on `ToolInvocation` (line 21972), whose fields are exactly `tool_name, tool_id, status (ok|error|timeout), latency_ms`. STORY-102 does not list `ToolInvocation` in `data_models`, so a reader implementing against this story would build the view off `Request.tool_count` and fail. STORY-102 also has no `api:` block (see M5), so the endpoint shape that would have forced the data dependency is also absent.
- **References**: `ToolInvocation` model exists at line 21972 (`grep "^- name: ToolInvocation"` returns line 21972). `Request.tool_count` exists at line 21787. FEAT-043 exists at line 765.
- **Fix**: Add `ToolInvocation` to STORY-102 `data_models`. Add an `api:` block defining `GET /api/analytics/tool-use` returning per-tool rows (tool_name, invocations, failure_rate, p50_ms, p95_ms). If per-tool rollups are expected from MetricSeries, extend MetricSeries.scope to include `tool` and add rollup columns.

### C3. STORY-103 streaming analytics cannot be served from the current rollup schema

- **Where**: lines 6982–7011 (STORY-103), 21790–21795 (Request.client_cancelled / mid_stream_dropped), 21828–21870 (MetricSeries).
- **Quoted (STORY-103 AC-002, line 7006)**: `I see counts of client_cancelled and mid-stream drops over time`.
- **Quoted (STORY-103 non_functional, line 7010–7011)**: `Dashboards and analytics read from pre-aggregated rollups computed by the worker fleet; raw-log fallback only within the last minute`.
- **Problem**: `Request.client_cancelled` and `Request.mid_stream_dropped` are present as booleans per the fix-log C4 (lines 21790–21795), so single-row truth exists. But "counts over time" means time-bucketed aggregates, and the declared NFR locks the UI to `MetricSeries` rollups for any non-last-minute window. The current `MetricSeries` columns (line 21828–21870) are `count, spend, spend_input, spend_output, spend_cache, errors, p50/p95/p99_ms, cache_hit_rate, input_tokens, output_tokens` — no `client_cancelled_count`, no `mid_stream_dropped_count`, no `ttft_p95_ms`. The AC-001 TTFT filter (`ttft > 500ms`) also has no aggregate backing. Implemented as written, STORY-103 either violates its own NFR (by running raw-log scans) or returns empty metrics over any range older than 1 minute.
- **References**: `MetricSeries` exists at 21828; Request fields at 21790–21795; STORY-103 at 6982.
- **Fix**: Either (a) extend MetricSeries with `client_cancelled_count, mid_stream_dropped_count, ttft_p50_ms, ttft_p95_ms` columns and update the worker-fleet rollup spec, or (b) weaken the NFR for this story to permit raw-log aggregation on streamed-only rows (bounded by range and `streamed=true`). Add an `api:` block that names the endpoint and the time-bucket granularity.

## Uncited-but-required features

Each subsection proposes a feature that EPIC-004 implicitly depends on but which is not defined anywhere in `docs/requirements.yaml`. For each: (1) the missing capability, (2) the yaml line(s) that create the expectation, (3) the closest existing neighbor (or "net-new FEAT required"), (4) severity.

### U1. Request-log ingestion pipeline (Critical)

1. **Missing capability**: An owned feature that declares the durable pipeline from a live gateway request to a persisted `Request` row (line 21722) queryable by every surface in EPIC-004. Scope includes queue contract, at-least-once semantics, idempotent handlers, backfill on ClickHouse ingestion gap, ingestion-health alert wiring.
2. **Why implied**: Baseline line 18 names `request-log ingestion` as a worker-fleet activity. Baseline line 30 says "ClickHouse ingestion targets >= 99.99% row durability for the request log; ingestion gaps trigger the ingestion-health alert." Epic description line 120 explicitly says: "Telemetry is produced by FEAT-143 and the request-log ingestion worker." The pipeline is named but unowned — FEAT-143 covers the *telemetry* emission, not the ingestion consumer. Every EPIC-004 surface (FEAT-029..044) reads `Request` rows; every one of them silently presumes the pipeline.
3. **Closest neighbor**: None. FEAT-167 (line 1991) surfaces *health* of worker queues but does not define any queue's contract. FEAT-143 (1762) is the producer side. A net-new FEAT (e.g. "FEAT-0NN Request-log ingestion") under EPIC-004 or EPIC-011 is required.
4. **Severity**: **Critical**. Every story in EPIC-004 depends on this pipeline; under the current text a reader cannot tell which team owns ingestion latency, dead-letter handling, or backfill.

### U2. MetricSeries rollup / pre-aggregation worker (Critical)

1. **Missing capability**: An owned feature that computes `MetricSeries` rows (line 21828) on a defined cadence, handles late-arriving data backfill, reconciles against ClickHouse raw logs, and honours retention. The feature must declare bucket granularity (1m, 5m, 1h, 1d rollups), scope projection (line 21840: `org|team|provider|model|project|api_key|env`) and failure modes.
2. **Why implied**: 26 story-level NFR lines (5435, 5467, 5642, 5719, 5789, 5835, 5867, 5928, 5971, 6002, 6026, 6145, 6182, 6236, 6302, 6351, 6466, 6498, 6625, 6719, 6805, 6843, 6866, 6888, 6914, 6936, 6958, 6980, 7010) say "Dashboards and analytics read from pre-aggregated rollups computed by the worker fleet; raw-log fallback only within the last minute." STORY-094 non_functional (line 6783) reaches further: "Query uses pre-aggregated MetricSeries when the range exceeds 7 days." Baseline line 18 names `metric aggregation`. No feature owns the worker.
3. **Closest neighbor**: FEAT-167 (line 1991) describes the queue health only. The proposed FEAT belongs under EPIC-004 (consumer owner) or EPIC-011 (telemetry pipeline). Net-new FEAT required.
4. **Severity**: **Critical**. Every analytics NFR pins itself to rollups the file never declares a producer for. The current dangling promise also blocks C3 (streaming counts) from being servable.

### U3. Anomaly detector evaluation worker (Critical)

1. **Missing capability**: A feature that periodically evaluates `AnomalyDetector` (line 21948) configurations, produces `Anomaly` rows (line 21899) when thresholds fire, respects `AnomalyDetector.cooldown_seconds` (line 21964) and `AnomalyDetector.last_fired_at` (line 21966), and resolves the audience per `project.notifications_model` (line 37) before dispatch.
2. **Why implied**: STORY-095/096/097 (lines 6784–6867) configure detectors; STORY-085 (line 6238) reacts to already-fired rows; STORY-085 AC-003 (line 6264–6274) requires the worker to write `Anomaly.audience_snapshot`, dispatch to trust_and_safety in saas and drop in self_hosted. Baseline line 18 names `anomaly scoring`. No feature owns the evaluation loop, the per-category threshold semantics, the cooldown enforcement, or the audience-resolution step at fire time.
3. **Closest neighbor**: FEAT-033 (line 666) describes the *signals* (consumer-facing surface). FEAT-041 (line 746) configures the detectors (settings). Neither declares the evaluation worker. Net-new FEAT required — most naturally "FEAT-0NN Anomaly detector evaluation" under EPIC-004.
4. **Severity**: **Critical**. STORY-085 AC-003 names contractual worker behavior but no feature owns that worker. This unownedness also blocks C4 (Anomaly→Alert bridge) from being implementable: you can't bridge fires you don't own producing.

### U4. Prompt full-text search index management (Major)

1. **Missing capability**: A feature that owns the prompt full-text search backing store: index creation, incremental indexing from the ingestion pipeline, PII-safe indexing (redacted vs unredacted prompt per viewer entitlement), retention parity with `RetentionConfig.request_body_days` (line 22994), reindex on redaction-policy change, and outage mode.
2. **Why implied**: Six story-level NFR lines (5643, 5720, 5790, 5836, 5868, 6403) say "Full-text search on prompt handled by a dedicated index (e.g. Postgres tsvector or an OpenSearch cluster)". FEAT-035 (line 685) *uses* the index but its description is "Search requests across id, user, model, and prompt text" — it does not own the index lifecycle. STORY-087 AC-003 (line 6386) says "when a row is redacted for the caller, only Request.prompt_redacted is matched and rendered" — this demands two distinct indexable fields with viewer-conditional query planning, a non-trivial capability nobody owns.
3. **Closest neighbor**: FEAT-035 (685) and FEAT-113 (1479, retention). The feature could be added under EPIC-004 next to FEAT-035, or under EPIC-010 alongside FEAT-113. Net-new FEAT required.
4. **Severity**: **Major**. FEAT-035 will either ship as a naive table scan (violating its NFR) or implement a full index pipeline without a spec — both are bug factories. The PII-conditional indexing is also a compliance risk.

### U5. Async request-data export job lifecycle (Major)

1. **Missing capability**: A feature that owns the async export pipeline for filtered request data: a job-tracking row (scope, status, output format, destination, expiry), the worker that materializes the archive, retry/poison-queue handling, download-URL signing, expiry sweep, and the FEAT-102 notification delivery on completion.
2. **Why implied**: STORY-088 AC-002 (line 6428–6436) describes "A modal offers an async export" with FEAT-102 delivery; STORY-089 AC-002 (line 6492) describes Parquet → S3 with a "completion notification". `DataExportJob` (line 23322) exists but its `scope` enum (line 23329: `user|org`) targets account-level archives — its semantics do not cover the filtered-request-log exports. No EPIC-004 feature owns the `RequestExportJob` equivalent, the worker, or the URL signing contract.
3. **Closest neighbor**: FEAT-036 (line 695) promises the *export surface* but says "Export the filtered request set to CSV" — the async lifecycle is not declared as its responsibility. Either extend FEAT-036 to own the job model and worker, or add a net-new feature.
4. **Severity**: **Major**. Today two stories (STORY-088, STORY-089) reference an async export with completion notifications but there is no model, no queue contract, no retry policy and no expiry sweep. This is the exact shape of a promise that gets built twice and drifts.

### U6. ToolInvocation producer pipeline (Major)

1. **Missing capability**: A feature that declares where `ToolInvocation` rows (line 21972) are written — during the request hot path, as a post-hoc worker step consuming streaming event deltas, or both — including per-tool latency/tokens accounting, status classification (`ok|error|timeout`, line 21996), and volume bounds.
2. **Why implied**: `ToolInvocation` exists as a model (21972) with a description "A single tool (function) call recorded inside a Request for tool-use analytics" — and is referenced by `Request.tool_count` (line 21787, "Number of tool invocations on this request (see ToolInvocation)"). FEAT-043 / STORY-102 (line 6960) needs per-tool aggregation. No story writes the row; no feature owns the writer. See C5/R1.
3. **Closest neighbor**: FEAT-146 (line 1785, Tool-calling passthrough) is the closest runtime owner. Extending FEAT-146 with a story "Emit ToolInvocation per tool-use turn" is likely the smallest-fix answer. Net-new FEAT is probably not needed.
4. **Severity**: **Major**. Without a writer-story, FEAT-043's single-story deliverable cannot work; C2 and C5 depend on this being named.

### U7. Share-link visit authorization and visit-audit (Major)

1. **Missing capability**: A feature that declares the visit-time enforcement for `ShareLink` (line 23405) — how `audience` values (`workspace|public-read`, line 23420) are enforced at the edge, how redacted prompts render under a workspace viewer who lacks the un-redact scope, when AuditLog rows are written on visit (baseline security line 33 requires tamper-evident audit), and how `ShareLink.open_count` / `last_opened_at` (lines 23433, 23436) are incremented concurrently without contention.
2. **Why implied**: STORY-092 AC-001 (line 6644) says "Opening the link from a signed-in workspace member restores the filters" — implying but not specifying visit-time authorization. STORY-093 AC-002 (line 6713) says "ShareLink.open_count increments and last_opened_at updates" — a concurrency-sensitive counter with no feature owning the write path. FEAT-039 (line 725) owns creation; nothing owns visit.
3. **Closest neighbor**: FEAT-039 (725). Extending FEAT-039 with a visit-authorization story is the obvious path, but the missing story today means a reader cannot tell whether ShareLinks are a new auth surface or a lightweight add-on.
4. **Severity**: **Major**. A signed-link auth surface with no declared enforcement is a security-design vacuum; especially for `public-read` audience the posture is undefined.

### U8. Replay isolation mechanics (Minor)

1. **Missing capability**: A feature that owns the synthetic-tag / budget-accounting / anomaly-exclusion semantics of replayed requests. Questions it must answer: does a replay decrement the caller's Budget (EPIC-008)? Is it excluded from MetricSeries cost rollups? How is the `replay_of_request_id` lineage traversed for spike detection? Which tag value is reserved?
2. **Why implied**: STORY-090 non_functional (line 6567–6568) states: "Replays are subject to the caller's budget and rate limits, not the original's" and "Replays are excluded from anomaly spike detection by a synthetic tag". `Request.replay_of_request_id` (line 21776) exists. Neither the budget exclusion/inclusion nor the synthetic-tag-producing mechanism is owned by any feature — baseline and model carry the promise, the behavior has no spec.
3. **Closest neighbor**: FEAT-037 (line 705) owns the replay UI. Either FEAT-037 should gain a story on the isolation semantics, or an EPIC-008 feature (Budget) should extend its spec to define replay inclusion/exclusion. Likely no net-new FEAT — scope extension on FEAT-037 plus a cross-reference to the budget feature.
4. **Severity**: **Minor**. The gap is narrow but real: the NFR as written cannot be implemented without a coordination decision between EPIC-004 and EPIC-008.
