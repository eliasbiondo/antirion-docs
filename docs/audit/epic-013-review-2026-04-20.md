# EPIC-013 re-audit (2026-04-20) — Platform operator administration

Read-only re-audit of `docs/requirements.yaml` scoped to EPIC-013 (line 304)
and its features FEAT-170 through FEAT-182 plus STORY-403 through STORY-441.
Cross-referenced against `project.deployment_modes` (line 36),
`project.notifications_model` (line 37), `project.non_functional_baseline`
(line 13), the `models:` section (line 20927+), and sibling epics
EPIC-001, EPIC-004, EPIC-007, EPIC-009, EPIC-010 and EPIC-012.

Line numbers refer to the state of `docs/requirements.yaml` on the branch
`feat/issue-99` at the time of this audit. This is a fresh pass — findings
were re-derived from the current file rather than carried over from
`docs/audit/epic-013-review.md`. Findings already applied per
`docs/audit/fix-log.md` are only restated when the current YAML proves the
fix is missing or was reverted.

No edits were made to `requirements.yaml`.

## Summary

The epic has been materially improved since the prior audit — `AuditLog`
has first-class operator-plane rows (line 23075), `Org` carries `state`
and suspension fields (line 20977-20988), `License.tier` exists
(line 24266), `IncidentTimelineEntry` and `LicenseIncident.renewal_confirmation_id`
are modelled, STORY-422 cites STORY-411 correctly, and the `/api/bootstrap`
collision has been resolved via `/api/operator/install/bootstrap`.

What remains is four clusters of defect:

1. **The saas operator principal is still absent.** EPIC-013 promises
   Antirion-staff-driven operator surfaces in saas, but no model declares
   an Antirion-staff identity. STORY-407 still returns `"platform admin"`
   as the qualifying role in both modes, contradicting FEAT-012 which
   pins `platform admin` to self_hosted only. Every `FK:User` field that
   audits saas operator actions (PlatformIncident.created_by_user_id,
   FleetDeployment.initiated_by_user_id, FleetDeployment.acting_user_id,
   LicenseIncident.last_actor_user_id) is either non-nullable with no
   saas User to point at, or silently fails. This was deferred in the
   prior audit (C4); it surfaces now in at least six places.
2. **Operator-namespace gating is incomplete.** STORY-408 enforces
   `/operator/*` and `/api/operator/*` 404-for-tenants, but FEAT-170's
   fleet-health stories route at `/settings/fleet/*` and `/api/fleet/*`
   (403 on deny — existence is disclosed), and FEAT-181 / FEAT-182
   route at `/antirion/*` which the gating contract never mentions. The
   gating rule's secrecy promise is bypassable via two separate URL
   namespaces.
3. **Schema-vs-AC drift in peripheral models.** WorkerQueue still lacks
   `consumer_count`, `dlq_depth`, `retry_count` rendered by STORY-405;
   PhoneHomeCallLog carries only a `payload_digest` but STORY-432
   promises an "inspect sample" render of the payload; License has no
   per-tier grace-window field despite FEAT-180 saying it is
   "configurable per license tier at issue time"; Org has no NPS / CSAT
   fields although STORY-439 and STORY-441 render both; FleetDeployment
   has non-nullable actor fields an auto-rollback cannot populate.
4. **Cross-epic capability gaps.** FEAT-167 (EPIC-012) still overlaps
   FEAT-170 with a different URL namespace; the operator paging
   integration named by `notifications_model` has no owning feature;
   the cross-tenant abuse-heuristic producer read by STORY-422 has no
   producer feature; the operator-plane audit viewer complementing
   FEAT-120 is absent.

Counts: **5 critical**, **12 major**, **10 minor**, **8 cross-epic**,
**13 missing/dangling references**, **12 uncited-but-required features**.

Severities used below:

- **Critical**: data-model / contract mismatches that block
  implementation, privileged-action audit coverage gaps, or broken
  mode/role gating.
- **Major**: broken intra-epic references, drafting contradictions,
  schema-vs-AC mismatches that cannot be resolved without product
  input.
- **Minor**: naming inconsistencies, missing-but-non-blocking fields,
  local drafting issues.
- **Cross-epic**: issues that surface in EPIC-013 but are really
  attributes of another epic's data or features.

## Critical

### C1. `/settings/fleet/*` and `/api/fleet/*` bypass the operator gating contract in STORY-408

*Citing lines*: STORY-403 AC-001 line 19001 (`I am on /settings/fleet`);
STORY-403 `GET /api/fleet/replicas` errors `[403]` line 19038-19048;
STORY-404 `POST /api/fleet/replicas/:id/drain` errors `[403, 404, 409]`
line 19106-19115; STORY-405 AC-001 line 19147 (`I am on /settings/fleet/queues`);
STORY-405 `GET /api/fleet/queues` line 19183-19189 (403 only); STORY-406
AC-001 line 19234 (`/settings/fleet/config`); STORY-408 AC-001 line
19370 (`Every /operator/* route returns 404 rather than 403 so existence
is not disclosed`).

*Problem*: EPIC-013's gating promise (STORY-408) is that operator
surfaces are invisible to tenant principals — 404, not 403, so the
existence of the surface is not leaked. But FEAT-170 stories — which
are operator-only by EPIC-013 membership and by their `navigation: operator`
declaration (line 18991) — route through the `/settings/*` tenant
namespace and reply 403 on deny. A tenant principal that probes
`GET /api/fleet/replicas` gets a 403, which is a direct disclosure that
the handler exists. The gating rule in STORY-408 explicitly says this
is forbidden. Identical problem on the UI side: `/settings/fleet` under
the tenant shell.

*Fix*: Either (a) move FEAT-170 stories to `/operator/fleet*` and
`/api/operator/fleet/*` so they inherit the STORY-408 404 contract, or
(b) extend STORY-408 AC-001 to explicitly gate `/settings/fleet/*` (and
name every operator-only path under `/settings/*` — currently none are
called out). Option (a) is simpler and removes the tenant-vs-operator
URL ambiguity from EPIC-012 as well (see C5 / M4 below).

### C2. STORY-407 returns `"platform admin"` as the qualifying role in saas, contradicting FEAT-012

*Citing lines*: STORY-407 AC-001 line 19315
(`GET /api/bootstrap returns deployment_mode in {saas, self_hosted}
and my_roles including "platform admin" only if I qualify`) and line
19316 (`The shell renders the "Operator" top-level nav item only when
my_roles contains "platform admin"`); FEAT-012 description line 447
(`The platform admin role is a reserved built-in that exists only in
self_hosted mode, cannot be granted to an ordinary workspace member,
cannot be created as a custom role in either mode, and is rejected by
the role API if requested in saas mode`).

*Problem*: FEAT-012 declares `platform admin` is self_hosted-only. In
saas mode no principal can hold the role — the role API rejects it.
Yet STORY-407 makes the saas-side "Operator" nav conditional on
`my_roles` containing `"platform admin"`. Under the current contract,
the Operator nav never renders in saas for anyone — including Antirion
staff, who are the declared saas operators per EPIC-013 line 306. Every
downstream saas operator surface (FEAT-170 fleet, FEAT-172 roster,
FEAT-173 incidents, FEAT-174 releases, FEAT-176 abuse triage, FEAT-181
support queue, FEAT-182 CRM) is thus unreachable in saas mode. Prior
audit m1 (STORY-409 saas-given "I hold platform admin") is the same
defect in a story scenario; C2 is the root cause — STORY-407 needs the
canonical saas operator-role name, and that name is nowhere defined
(see C5 and U1).

*Fix*: Introduce a canonical operator-role identifier (e.g.,
`"antirion_staff"` in saas, `"platform_admin"` in self_hosted) and
rewrite STORY-407 AC-001 to expose `operator_role` separately from the
tenant `my_roles` list, or return the mode-correct role string in
`my_roles`. Must be coordinated with C5 / U1 (operator principal
identity).

### C3. `/antirion/*` namespace (FEAT-181, FEAT-182) has no declared gating contract

*Citing lines*: STORY-436 AC-001 line 20738 (`/antirion/support/queue`);
STORY-439 AC-001 line 20840 (`/antirion/crm/orgs/:id`); STORY-441 AC-001
line 20908 (`/antirion/crm/pipeline`); STORY-408 AC-001/002/003 lines
19362-19387 (names only `/operator/*` and `/api/operator/*`).

*Problem*: FEAT-181 and FEAT-182 render on the "Antirion-staff operator
console" (descriptions at 2133 and 2143). Their routes live at
`/antirion/*`, which is a third URL namespace separate from both
`/settings/*` and `/operator/*`. FEAT-171's gating contract
(STORY-407/408) says nothing about `/antirion/*`, so there is no
declared contract that tenant principals get 404 on those paths, no
declared contract that `/antirion/*` is absent from non-Antirion
self_hosted installs (FEAT-181/182 say the features "run only on
Antirion's own operator console"), and no AC exists anywhere that
probes `/antirion/*` for leakage. An implementation reading the spec
could emit a /antirion handler on every install and pass all existing
tests. This was flagged as M4 in the prior audit, marked `deferred
— needs gating contract update` in `fix-log.md`; the gating contract
update still has not been made.

*Fix*: Extend STORY-408 AC-001 to state `/operator/*`, `/api/operator/*`
AND `/antirion/*`, `/api/antirion/*` are gated identically, with the
additional rule that `/antirion/*` returns 404 on any install that is
not the Antirion-operated console (independent of mode). This may
require the baseline-level "installed_on" dimension raised in prior
X5 — see also C4 below.

### C4. FEAT-181 `mode: saas` and FEAT-182 `mode: saas` misrepresent the feature's location

*Citing lines*: FEAT-181 line 2134 (`mode: saas`); FEAT-181 description
line 2133 (`in self_hosted mode the feature is installed only on
Antirion's own operator console and is never rendered on customer
installs`); FEAT-182 line 2144 (`mode: saas`); FEAT-182 description
line 2143 (`In self_hosted mode for Antirion's internal tracking of
its own customers this CRM runs only on Antirion's operator console
and carries one row per self_hosted install that has opted in to the
"license" phone-home channel under FEAT-179; on a customer install
itself, this surface is never rendered`).

*Problem*: `mode: saas` is a machine-readable gate per
`project.non_functional_baseline.deployment_modes` (line 36). Any tool
that reads `mode:` to decide presence will conclude FEAT-181/182 are
absent in self_hosted — including on Antirion's own self_hosted
console, where the descriptions themselves say the features *do* run
(for internal tracking). The tag is load-bearing and currently wrong.
Prior audit M2, deferred in `fix-log.md` as `Needs baseline schema
upgrade`; the upgrade has not been applied.

*Fix*: Either (a) extend the `mode:` taxonomy to include a value like
`antirion_console_only` (present on the Antirion-operated console
regardless of install mode; absent on customer installs in either
mode), or (b) add a sibling key `installed_on: antirion_console` and
set `mode: both, installed_on: antirion_console` for FEAT-181/182.
Must be coordinated with C3's `/antirion/*` gating rule.

### C5. No "Antirion staff" / operator-principal model — every saas EPIC-013 audit row has no actor shape

*Citing lines*: EPIC-013 description line 306; STORY-407 AC-001 line
19315; STORY-412 AC-001 line 19625 (`the acting operator (Antirion
staff id in saas, platform admin in self_hosted)`); STORY-417 AC-001
line 19940 (`Antirion SRE in saas, platform admin in self_hosted`);
FEAT-181 description line 2133; FEAT-182 description line 2143. Models:
`User` line 21002 (tenant-scoped, `belongs_to Org`); `AuditLog.actor_user_id`
line 23080 (FK:User, nullable); `PlatformIncident.created_by_user_id`
line 24177 (FK:User, non-nullable); `FleetDeployment.initiated_by_user_id`
line 24232 (FK:User, non-nullable); `FleetDeployment.acting_user_id`
line 24234 (FK:User, non-nullable); `LicenseIncident.last_actor_user_id`
line 24891 (FK:User, non-nullable).

*Problem*: In saas mode the operator is "Antirion staff". There is no
field on `User`, no dedicated `OperatorPrincipal` / `AntirionStaff`
model, and no documented convention for how an Antirion-staff account
is represented in a saas AuditLog. Consequences:

- `AuditLog.actor_user_id` is nullable (line 23082) so operator-plane
  rows can leave it null — but then the identity of the acting saas
  staffer is lost, which contradicts STORY-412's explicit requirement
  that the audit records the acting operator's id.
- `PlatformIncident.created_by_user_id` is *non-nullable* — an incident
  declared by Antirion staff in saas has no valid User row to attach.
- `FleetDeployment.initiated_by_user_id` / `acting_user_id` are both
  non-nullable — auto-rollback (STORY-417) has no human driver, and
  saas-initiated rollouts have no tenant-User identity either.
- `LicenseIncident.last_actor_user_id` is non-nullable but the incident
  is self_hosted-only; still, STORY-435 AC-002 shield flows track
  Antirion staff actions (the renewal-confirming side) that are not
  on the install.
- Edge gating (STORY-408) claims to identify a saas operator, but no
  model tells the edge how.

*Fix*: Add an explicit `OperatorPrincipal` model (`id, kind=antirion_staff|platform_admin,
external_id, name, active, created_at`) with its own identity lifecycle,
and change every `created_by_user_id` / `initiated_by_user_id` /
`acting_user_id` to a polymorphic ref or add a parallel `*_operator_id`
nullable FK. Audit actor resolution should read `OperatorPrincipal`
first, falling back to `User`. This is also the foundation for C2
(STORY-407 role string) and is the root of prior audit C4.

### M4. `PhoneHomeCallLog` lacks a retained payload, but STORY-432 "inspect sample" needs one

*Citing lines*: `PhoneHomeCallLog` line 24837 — fields are `channel,
endpoint, payload_digest, response_code, latency_ms, at, prev_hash,
hash` only; no body; STORY-432 AC-001 line 20627 (`a "inspect sample"
action that renders the payload through the same redaction rules as
the original send`).

*Problem*: The acceptance criterion promises an inspect-sample render.
The model only stores a SHA-256 digest of the payload, so the payload
cannot be re-rendered from the log row. There is no separate
`PhoneHomePayloadSample` store, and no retention-window column on
PhoneHomeCallLog.

*Fix*: Either add `payload_sample: text nullable` with an explicit
retention column (`sample_retained_until: timestamp`) and document that
samples outside the retention window appear as "expired" in the UI; or
remove the "inspect sample" AC and clarify that audit of payload
content is deliberately not possible post-send.

### M5. FEAT-170 vs FEAT-167 URL-namespace collision

*Citing lines*: STORY-399 narrative line 18817 (`I am on /operator/workers —
the worker fleet is platform infrastructure, so the dashboard lives
under the EPIC-013 operator shell, not under tenant /settings`);
STORY-403 AC-001 line 19001 (`I am on /settings/fleet`); STORY-405
AC-001 line 19147 (`I am on /settings/fleet/queues`); STORY-406 AC-001
line 19234 (`I am on /settings/fleet/config`); STORY-405 NFR line 19218
(`Queue-health reads are served from the same metrics store that
powers FEAT-167 — no direct broker polling from the UI path`).

*Problem*: FEAT-167 (EPIC-012) and FEAT-170 (EPIC-013) both render
worker/queue health. FEAT-167 explicitly says it lives on
`/operator/workers` (STORY-399) and calls out that `/settings` is the
wrong home. FEAT-170 does exactly what STORY-399 rejects: lives at
`/settings/fleet/*`. The two features also share the same metrics
store. From a reader's perspective there are two places to look for
the same data with opposite routing decisions. EPIC-012 C5 flagged the
FEAT-167 side; it was deferred (`Structural; needs product decision`);
the EPIC-013 side has never been flagged until now.

*Fix*: Pick one. Either collapse FEAT-167 into FEAT-170 and update the
baseline citation at line 18 (`Worker fleet health is surfaced by
FEAT-167`), or explicitly scope the two features (FEAT-167 =
tenant-facing, per-tenant-owned jobs; FEAT-170 = platform-operator
aggregated view across tenants), align the URL namespaces, and
cross-link both features in the descriptions. The routing fix in C1
(move FEAT-170 to `/operator/fleet*`) addresses the URL half.

### M8. STORY-430 UI state `forced_on_saas` has no model or API correspondence

*Citing lines*: STORY-430 `ui.states` line 20549 (`forced_on_saas`);

### M10. STORY-425 SSO-only bootstrap has no intent/claim persistence model

*Citing lines*: STORY-425 AC-002 line 20367-20375 (SSO-only mode —

acceptance criterion.

### M11. `License` has no per-tier grace-window field despite FEAT-180 saying it is configurable per tier

feature (U8).
