# EPIC-012 Re-audit — Platform foundations and customer experience (2026-04-20)

Fresh, read-only pass over `docs/requirements.yaml` for EPIC-012 (line 291) and its features FEAT-162..FEAT-169 with stories STORY-387..STORY-402. Cross-referenced against `project.non_functional_baseline` (line 13), `project.notifications_model` (line 37), `project.ui_baseline` (line 6), the `models:` section (line 22950+) and sibling epics EPIC-001, EPIC-008, EPIC-010, EPIC-013.

This audit was produced without assuming any previous finding still applies. Every finding is re-derived from the current file and cites `docs/requirements.yaml:LINE`.

## Summary

The prior EPIC-012 pass (`docs/audit/epic-012-review.md`, actioned in `docs/audit/fix-log.md`) mended the largest data-model gaps — `SupportTicket.destination`, `ResidencyConfig.mode`/`source`, `Notification.kind` enum, `WorkerJob.owner_org_id`, `LegalDocument`, `HelpArticlePin`, the `StatusPageIncident → PlatformIncident` FK. Several follow-on issues remain, and the post-fix state introduces new drift.

Four themes dominate this pass:

1. **Stories did not follow the schema changes made for them.** The model gained `HelpArticlePin` and `HelpArticle.view_count`, but STORY-398 still writes `HelpArticle.pinned` and still asks for per-time-range "views per article" that a scalar counter cannot serve. The `Notification.kind` enum was expanded to eleven values, but STORY-402 still names only the original four. `WorkerJob.owner_org_id` was added, but STORY-400's cancel AC never enforces the owner-match constraint that the model's description now documents.
2. **FEAT-167 / non_functional_baseline.workers contract is still unresolved.** Baseline line 18 still cites `FEAT-167` for worker-fleet health, yet the queue-depth / DLQ / consumer-count / propagation surfaces live entirely in FEAT-170 STORY-405..406. FEAT-167 is two stories that together contradict each other on scope (operator-route dashboard vs tenant-admin cancel). The fix-log marks this "deferred — structural".
3. **Mode-conditional acceptance criteria are still missing on tenant-facing features.** FEAT-162/163/164/165/166/168/169 now carry `mode: both`, but STORY-391/392/395/396/397/398/401/402 still contain zero mode-gated ACs. The most testable rule in FEAT-163 — "the in-app widget always points at the mode-correct page" — has no AC anywhere.
4. **FEAT-165 legal-publication path contradicts itself.** FEAT-165 description says the ToS is published by Antirion legal in saas and uploaded by the platform admin in self_hosted; STORY-396 narrative still says a `workspace admin` publishes it. `LegalDocument` exists, but there is no `TosAcceptance.required_version` field to satisfy the AC that writes it.

Counts: **6 critical**, **14 major**, **9 minor**, **3 cross-epic**, **4 missing / dangling**, **12 uncited-but-required** (3 critical, 6 major, 3 minor).

Top three highest-impact items:

1. **FEAT-167 vs baseline line 18** — baseline still promises worker-fleet health through FEAT-167 but the substantive surfaces live in FEAT-170; the two EPIC-012 stories on FEAT-167 route to `/operator/workers` (EPIC-013 territory), split audiences between "SRE" and "workspace admin", and leave the EPIC-012-vs-EPIC-013 ownership unresolved (C5, M10, M11).
2. **Post-fix model/story drift on FEAT-166 and FEAT-169** — the schema changes applied in the previous pass (HelpArticlePin, view_count, Notification.kind enum extension) were not reflected in the stories that motivated them, and the capability stores behind them (view-analytics event pipeline, notification retention) have no owning feature (C1, C2, C4, M12, U1, U2).
3. **FEAT-165 publication workflow is unowned** — STORY-396 still narrates a tenant `workspace admin` publishing the ToS, the AC writes a field (`TosAcceptance.required_version`) that does not exist on the model, `LegalDocument` is present but unused by any story, and there is no feature owning the "currently-required version" lookup that drives the FEAT-165 re-prompt (C3, M2, U3, U9).
