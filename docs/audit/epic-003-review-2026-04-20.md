## Critical

### C1. STORY-057 AC-002 writes `AlertEvent.type`, which is not a field on AlertEvent (docs/requirements.yaml:4929, 22733-22784)

STORY-057 AC-002:

> `docs/requirements.yaml:4929`
> "An AlertEvent of severity 'warn' with type 'byok.degraded' is generated"

`AlertEvent` model (`docs/requirements.yaml:22733-22784`) fields: `id, rule_id, source, severity, status, title, summary, provider_id, model_id, team_id, fired_at, acknowledged_at, resolved_at, duration_sec, fingerprint, affected_requests, error_count, oncall_user_id`. There is no `type` field.

**Problem.** Implementers will either (a) silently coerce "byok.degraded" into `title` / `summary` / `source` and lose the discriminator, or (b) fail the migration/test because no column named `type` exists. The same "AlertEvent of severity X with type Y" phrase appears in several other stories (`:7808, :10830, :16621`) — this audit only addresses the EPIC-003 touchpoint, but fixing it without aligning the sister stories will leave the system using `type` as a convention with nowhere to store it.

**Suggested fix.** At `docs/requirements.yaml:4929` write "A BYOK-degraded AlertEvent is generated with severity 'warn', source 'system' and fingerprint 'byok.degraded:<credential_id>'". (Alternatively add `kind` / `category` to the AlertEvent schema — but that is an EPIC-009 pass.)

## Major

### M1. STORY-057 AC-002 sets BYOKCredential.status="degraded" when the provider returns 401 (docs/requirements.yaml:4927-4928, 21481-21483)

STORY-057 AC-002:

> `docs/requirements.yaml:4927-4928`
> "The card shows 'Failed · 401 Unauthorized'
>  BYOKCredential.status becomes 'degraded'"

`BYOKCredential.status` enum and precedence:

> `docs/requirements.yaml:21481-21483`
> "active|degraded|overdue|invalid|revoked (precedence overdue < degraded < invalid < revoked; active when none apply)"

**Problem.** A 401 Unauthorized from the upstream provider is the canonical "credential is invalid" signal — not "probe is slow" or "provider is flapping". The enum explicitly carves out `invalid` for this case and ranks it higher than `degraded`. Writing `degraded` instead means: (a) STORY-059's list UI shows the credential as yellow-warning rather than red-critical; (b) the next scheduled probe, if it succeeds (provider recovered after a transient 401), would flip the credential back to `active` and the admin would never know the secret was rejected; (c) STORY-061's `overdue` transition could silently override `degraded`, hiding the bad-credential state entirely.

**Suggested fix.** At `docs/requirements.yaml:4928` change to "BYOKCredential.status becomes 'invalid'". Keep `degraded` for transient (e.g., 5xx, timeouts, intermittent) failures and reserve `invalid` for auth failures. Add an AC distinguishing transient vs. auth failures if product wants that split recorded.

### M2. Approval.kind "key-elevated-scope" is shared between FEAT-027 STORY-065 and FEAT-160 STORY-377 (docs/requirements.yaml:5212, 17809, 23198)

`Approval.kind` enum:

> `docs/requirements.yaml:23198`
> "budget-overage|budget-change|policy-change|key-elevated-scope|routing-rule-change|data-reveal"

STORY-065 (FEAT-027, EPIC-003 — elevate an existing ApiKey's scope):

> `docs/requirements.yaml:5212`
> "An Approval of kind 'key-elevated-scope' is created with target_type='ApiKey' and target_id=key"

STORY-377 (FEAT-160, EPIC-011 — create a new break-glass ApiKey):

> `docs/requirements.yaml:17809`
> "An Approval of kind 'key-elevated-scope' is created for the break-glass key"

**Problem.** One discriminator, two semantics. Approvers on the review surface see a mixed queue — "this is a request to widen an existing key" vs. "this is a request to mint a new elevated key". Distinguishing them requires the approver to inspect `target_id` and cross-reference whether the ApiKey row already exists. This violates the principle that `kind` is the surface-routing discriminator for Approvals.

The fix-log flags this as "deferred (EPIC-011 pass will handle)", but the collision is still live in the current file.

**Suggested fix.** Add a new enum value, e.g. `break-glass-key-create`, at `docs/requirements.yaml:23198` and update STORY-377 at `:17809` to use it. STORY-065/STORY-066 can keep `key-elevated-scope`.

### M3. FEAT-025 promises "enforce rotation policy" but no story lets an operator set `rotate_policy_days` (docs/requirements.yaml:574, 21484, 5015, 5059)

FEAT-025 description:

> `docs/requirements.yaml:574`
> "Register, rotate, test, list, label, delete and enforce rotation policy on upstream provider credentials used for outbound calls."

`BYOKCredential.rotate_policy_days` (nullable int) is at `:21484`. It is **read** by STORY-059 (list, `:5015`) and **read** by STORY-061 (enforcement, `:5059`), but **never written** by any story in FEAT-025 or elsewhere:

- STORY-060 only edits `label` (`:5038`).

**Problem.** The rotation policy is observable and enforced but not editable. An org admin cannot, through any UI or API in this epic, set or change a credential's `rotate_policy_days`. **Suggested fix.** Add a story under FEAT-025 — "Set BYOK rotation policy" — with a PATCH endpoint accepting `rotate_policy_days`. Alternatively add `rotate_policy_days` to STORY-060's label-edit body and rename the story.

### M4. `ApiKey.scopes` materialization races with `elevated_until` expiry (docs/requirements.yaml:21380-21393, 5240-5252)

`ApiKey.scopes` description:

> `docs/requirements.yaml:21382`
> "Effective scopes at authorization time (base_scopes unioned with elevated_scopes while elevated_until is in the future)"

STORY-066 AC-001 (approve) and AC-002 (expire):

> `docs/requirements.yaml:5240-5241, 5250-5252`
> "ApiKey.elevated_scopes is set to the requested scopes and ApiKey.elevated_until is set to now + duration
>  ApiKey.scopes (effective) becomes base_scopes unioned with elevated_scopes"
> "The scheduled job runs
>  ApiKey.elevated_scopes is cleared and elevated_until is set to null
>  ApiKey.scopes (effective) reverts to ApiKey.base_scopes"

**Problem.** The model description reads as if `scopes` is computed at authorization time; the story ACs read as if `scopes` is materialized by approval/expire jobs. This ambiguity has real consequences:

- If `scopes` is materialized, then from `elevated_until` to "the scheduled job runs", the key still authorizes the elevated scope. Even a 1-minute cadence leaves a security tail past the declared expiry.
- If `scopes` is computed at auth time, then STORY-066 AC-001's "ApiKey.scopes becomes base ∪ elevated" is a no-op assertion and AC-002's "reverts to base_scopes" is trivially true.
- FEAT-138 STORY-323 (edge auth) does not resolve the ambiguity either — it reads ApiKey.scopes as a simple column (`:16095, :16104`).

**Suggested fix.** Pick one representation explicitly. Either (a) remove `ApiKey.scopes` as a stored field, make it a derived accessor in STORY-323, and update STORY-066 AC language to stop asserting writes on it; or (b) keep `scopes` materialized and add an AC "edge auth re-evaluates elevated_until on every request and filters elevated_scopes out when expired, regardless of the scheduled job" to STORY-323 so the tail is closed at the edge.

## Uncited-but-required features

These are capabilities the current EPIC-003 text assumes exist, but for which no FEAT or STORY block is defined. They are not "dangling cited references" — nothing cites them — yet the epic is incoherent without them.

### U1. BYOK rotation-grace expiry job (Critical)

1. **Missing capability.** A background job that, after `BYOKCredential.rotation_grace_ends` elapses, flips the predecessor credential's `status` to `revoked` and stops it from being selected for outbound calls. The ApiKey analogue is STORY-063 (auto-revoke on TTL, `:5116-5161`); BYOK has no counterpart.
2. **What in the yaml implies it.** STORY-056 AC-002 (`:4856-4858`): "A successor BYOKCredential is created and linked via the predecessor's successor_id / rotation_grace_ends is set on the predecessor to now + grace window / Both credentials are valid for outbound calls until rotation_grace_ends elapses, after which the predecessor status becomes 'revoked'". The "after which" clause is declarative but unowned. `BYOKCredential.rotation_grace_ends` at `:21494-21497` exists solely to gate this job.
3. **Closest neighbor.** FEAT-025 (BYOK provider credentials). The job belongs as a new story under FEAT-025 — "Auto-revoke predecessor BYOK credential at grace-window end" — parallel in shape to STORY-063.
4. **Severity.** **Critical.** Without this job, rotated BYOK credentials accumulate forever in `active`-with-`rotation_grace_ends-in-past` state; outbound traffic keeps selecting stale credentials; compromise windows never close.

### U2. BYOK rotation-policy configuration surface (Major)

1. **Missing capability.** A UI/API to set or edit `BYOKCredential.rotate_policy_days`. (See M3 for the read-side evidence.)
2. **What in the yaml implies it.** FEAT-025 description at `:574` promises "enforce rotation policy". STORY-059 displays `rotate_policy_days` (`:5015`). STORY-061 enforces it (`:5059-5064`). STORY-056's register body is `{provider, secret}` only (`:4871-4874`) — no policy days. STORY-060 only edits `label` (`:5038`). Thus the field is read but never written.
3. **Closest neighbor.** FEAT-025. A new story — "Configure BYOK rotation policy per credential" — with a PATCH body containing `rotate_policy_days` would close the gap. Alternatively fold it into STORY-060 and rename "Edit a BYOK credential label" to "Edit BYOK credential metadata".
4. **Severity.** **Major.** STORY-061's enforcement loop becomes dead code for every credential whose policy is never set (i.e., every credential today).

### U3. BYOK upstream-secret inbound-leak-scan path (Major)

1. **Missing capability.** FEAT-028 (API key leak emergency response) handles **gateway-issued** API keys via an inbound secret-scan partner notification. The same risk exists for **upstream BYOK provider secrets** — if Anthropic's or OpenAI's secret-scan partner detects a leaked BYOK secret, no story routes it to `BYOKCredential.status = 'invalid' + auto-rotate`.
2. **What in the yaml implies it.** Two pieces. (a) FEAT-028's description at `:608` scopes leak response to "gateway API keys" — no explicit exclusion of BYOK, but the stories only touch `ApiKey` (`:5278, 5297-5298`). (b) `BYOKCredential.status` carves out `invalid` as a canonical state (`:21481-21483`) with no non-probe path to reach it: STORY-057's probe is manual/admin-initiated and only detects auth failures at test time, not in real time. Without an inbound-notification path, a leaked BYOK secret only surfaces when an admin clicks "Test".
3. **Closest neighbor.** FEAT-028 (leak emergency response). Either extend FEAT-028 with BYOK-secret leak-scan handling, or create a sibling feature "BYOK credential leak emergency response" in EPIC-003.
4. **Severity.** **Major.** The cost asymmetry is high — BYOK secrets are the customer's provider-account secrets; a leaked BYOK secret drives the customer's provider bill, not Antirion's. Relying on probe-on-click means discovery lag is whatever the admin's dashboard-check cadence is.

### U4. API-key usage-anomaly detector configuration surface specific to KeyAnomaly (Minor)

1. **Missing capability.** A surface that configures the detectors feeding `KeyAnomaly` (geo/spike/ua). EPIC-004 FEAT-041 is "Anomaly detector configuration" but its description cites FEAT-033 (generic `Anomaly` entity) not `KeyAnomaly`.
2. **What in the yaml implies it.** STORY-055 AC-001 (`:4810-4812`) says "The anomaly detector runs on its recent traffic / A KeyAnomaly of kind 'geo' and medium severity is created". FEAT-041 at `:747-754` is generic: "Configure the anomaly detectors that drive FEAT-033 anomaly signals — list, set thresholds, enable/disable by category". It never mentions `KeyAnomaly` or `ApiKey`. The threshold and cadence from STORY-055's non_functional ("at least a 5-minute cadence", `:4828`) thus live nowhere an operator can read or edit them.
3. **Closest neighbor.** FEAT-041. Either (a) extend FEAT-041's scope to cover `KeyAnomaly` detectors explicitly, or (b) add a FEAT under EPIC-003 that owns KeyAnomaly detector config and cross-links FEAT-041.
4. **Severity.** **Minor.** The runtime loop in STORY-055 works without config UI; this is an ergonomics gap, not a correctness one.
